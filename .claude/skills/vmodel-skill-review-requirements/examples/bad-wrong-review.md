# Counter-example — what NOT to do as a reviewer

A reviewer is given a **deeply flawed retrofit document** generated by an AI agent against a legacy session service. The document has fabricated rationale, smuggled design, and laundered current state. The agent that produced the document marked `recovery_status: reconstructed` for the rationale field — the most consequential single rule violation.

Below are three failure modes a reviewer can fall into, and what the right verdict looks like in contrast.

## Failure mode 1 — False approve ("the document reads competent")

The reviewer sees fluent prose, well-formed YAML, every requirement carrying a rationale field. Without checking the *content* of the rationale or the `recovery_status` legality, the reviewer concludes APPROVED.

```yaml
review:
  document: /retrofit/session-service/requirements.md
  reviewer: vmodel-skill-review-requirements
  date: 2026-04-26
  verdict: APPROVED              # ❌ WRONG
  upstream_input_provided: false
  perspectives_applied: [designer, tester]
  findings: []
  summary: |
    Document is well-structured, every requirement carries a rationale, no
    obvious anti-patterns. Approved.

  counts:
    hard_reject: 0
    soft_reject: 0
    info: 0
    design_issue: 0
```

### What was missed

The document contains:
- Rationales like *"30 minutes balances security and UX, an industry-standard choice"* — `anti-pattern.fabricated-rationale` (HARD REJECT)
- `recovery_status: reconstructed` on rationale fields — `check.rationale.recovery-status-reconstructed` (HARD REJECT)
- *"The session service shall use Redis with a 5-second TTL"* in a non-interface requirement — `anti-pattern.implementation-prescription` (HARD REJECT)
- `derived_from: [user-experience, security]` — `check.traceability.derived-from-vague` (soft_reject in greenfield, hard_reject in retrofit)

Three hard-reject triggers fired. The verdict should have been REJECTED.

The failure mode: the reviewer judged on *appearance* (fluent prose, structured YAML) rather than *content* (the substance of the rationale, the legality of recovery_status).

## Failure mode 2 — Subjective rejection ("I don't like the wording")

The reviewer emits findings based on personal preference rather than catalogued checks.

```yaml
review:
  document: /specs/session-service/requirements.md
  reviewer: vmodel-skill-review-requirements
  date: 2026-04-26
  verdict: REJECTED              # ❌ WRONG basis
  upstream_input_provided: true
  perspectives_applied: [designer, tester]
  findings:
    - id: F-001
      requirement_id: REQ-session-002
      check_failed: style.preferred-phrasing       # ❌ NOT IN CATALOG
      severity: soft_reject
      category: <not-cataloged>
      evidence: |
        REQ-session-002 uses "shall create a new session" — would prefer "shall instantiate".
      recommended_action: |
        Use "instantiate" instead of "create".               # ❌ specific replacement wording
```

### What's wrong

- `check_failed: style.preferred-phrasing` — not in the catalog. **Every finding's identifier must appear in `quality-bar-gate.md`**. Reviewer must not invent ad-hoc strings.
- The objection is subjective — neither "create" nor "instantiate" is more correct; both are testable. No anti-pattern catalogued.
- `recommended_action` writes specific replacement wording. **The reviewer's job is signalling; the author's job is rewriting.**

The failure mode: the reviewer is acting as a stylist rather than a gatekeeper. Subjective rejection erodes trust in the verdict and creates pointless rewrite cycles.

## Failure mode 3 — Missing the meta-gate

The reviewer ticks every box of the Quality Bar checklist and approves the document — but a junior engineer reading the document cannot in fact derive a defensible architecture, design, and test specification. The Spec Ambiguity Test was checked as Yes without being honestly applied.

```yaml
review:
  document: /specs/session-service/requirements.md
  reviewer: vmodel-skill-review-requirements
  date: 2026-04-26
  verdict: APPROVED              # ❌ WRONG — meta-gate fails
  upstream_input_provided: true
  perspectives_applied: [designer, tester]
  findings: []
  summary: |
    All Quality Bar items checked Yes. Document approved.
```

### What was missed

The document defines functional requirements but never specifies *which session policy applies in which tenant context*. A junior engineer reading the document would have to ask: "If a tenant has both MFA-enabled (REQ-006) and a regulatory inherited constraint that requires elevated sessions for all writes (some derived REQ from a regulatory IC), what is the precedence between these?" The document doesn't say. Architecture cannot be allocated without this answer.

This is exactly the kind of gap the **Spec Ambiguity Test** is designed to catch. The reviewer ticked the box without applying the test honestly.

The correct finding would be:

```yaml
- id: F-NNN
  requirement_id: GLOBAL
  check_failed: check.meta-gate.spec-ambiguity-test-fails
  severity: hard_reject
  category: meta-gate
  evidence: |
    REQ-006 (where MFA enabled, require step-up) and IC-002's derived requirements
    (regulatory: elevated sessions for writes) interact under MFA-enabled tenants
    without a stated precedence. A junior engineer cannot determine the correct
    allocation without asking.
  recommended_action: |
    State explicitly: under MFA-enabled tenants subject to the regulatory
    constraint, which precedence applies? Either add the precedence rule as a
    derived requirement, or open an ADR to settle the policy.
```

The failure mode: the reviewer treated the Spec Ambiguity Test as a checkbox rather than a meta-gate. The check is irreducibly heuristic — it requires the reviewer to *actually* read the document from a junior engineer's perspective and ask "could I act on this?".

## What this counter-example teaches

1. **Judge content, not appearance.** Fluent prose and structured YAML are not evidence of quality. Read the *substance* of every rationale, every `recovery_status`, every `derived_from`.

2. **Use the catalog.** Every `check_failed` identifier must appear in `references/quality-bar-gate.md`. Inventing identifiers undermines the catalog's purpose.

3. **Generic recommended actions only.** Pointing to the rule is helping. Writing the replacement is overstepping. The author skill rewrites.

4. **The Spec Ambiguity Test is irreducible.** No checklist substitutes for honest application of "could a junior engineer act on this without asking?". Apply judgment; do not tick the box.

5. **Hard-reject triggers are non-negotiable.** Fabricated rationale, illegal `recovery_status: reconstructed` on rationale, smuggled design — one occurrence is fatal. Do not be charitable; do not soften the verdict because the rest of the document looks fine.

6. **DESIGN_ISSUE wins over REJECTED.** When the upstream is broken, the document cannot reach APPROVED by rewriting alone. Surface the upstream issue as the verdict, even if internal issues also exist.

## What the correct verdict looks like for the retrofit document above

```yaml
review:
  document: /retrofit/session-service/requirements.md
  reviewer: vmodel-skill-review-requirements
  date: 2026-04-26
  verdict: REJECTED
  upstream_input_provided: false
  perspectives_applied: [designer, tester]
  findings:
    - id: F-001
      requirement_id: REQ-R-session-001
      check_failed: check.rationale.recovery-status-reconstructed
      severity: hard_reject
      category: rationale
      evidence: |
        rationale_recovery_status: reconstructed — illegal value for a rationale
        field. Allowed values are 'verified' or 'unknown' only.
      recommended_action: |
        Change to 'verified' (if a human can confirm) or 'unknown' (if no recall
        available). Reconstructing rationale from observable code is fabrication.

    - id: F-002
      requirement_id: REQ-R-session-001
      check_failed: anti-pattern.fabricated-rationale
      severity: hard_reject
      category: anti-pattern
      evidence: |
        rationale: "30 minutes balances security and UX, an industry-standard
        choice after consultation with the security team"
        Tells: "industry-standard" (generic, no source); "balances X with Y"
        (could justify any value); "after consultation" (no record exists).
      recommended_action: |
        Replace with `rationale: unknown` and queue a security-team follow-up.

    - id: F-003
      requirement_id: REQ-R-session-002
      check_failed: anti-pattern.implementation-prescription
      severity: hard_reject
      category: anti-pattern
      evidence: |
        "The session service shall use Redis with a 5-second TTL for active
        session state."
      recommended_action: |
        Move "Redis + 5-second TTL" to an architectural decision (ADR). Replace
        here with the behavioural requirement.

    - id: F-004
      requirement_id: REQ-R-session-001
      check_failed: check.traceability.derived-from-vague
      severity: hard_reject
      category: traceability
      evidence: |
        derived_from: [user-experience, security] — neither is an artifact ID.
        In retrofit mode, derived_from must cite either real upstream artifacts
        or observed evidence (file:line, commit, log).
      recommended_action: |
        Replace with observed-evidence references: e.g.,
        "observed_behaviour: session_service.py:L245-L268",
        "observed_config: config/prod.yaml#session.idle_timeout_seconds=1800".

  summary: |
    Verdict: REJECTED. Four hard-reject findings, three of them on the same
    requirement (REQ-R-session-001). The document is in retrofit mode but
    treats rationale as reconstructable — the most corrosive AI-era anti-pattern.
    Upstream input was not provided, so DESIGN_ISSUE triggers cannot be evaluated;
    however, the internal violations are sufficient on their own to reject. The
    matched author skill should rewrite with rationale honestly marked `unknown`,
    `derived_from` pointing to observed evidence, and named technology moved to
    an ADR.

  counts:
    hard_reject: 4
    soft_reject: 0
    info: 0
    design_issue: 0
```
