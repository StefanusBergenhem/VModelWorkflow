Non-Leaf Scope Artifact

1. V-Model Context

A Requirements artifact states what the system (or subsystem) at this scope must do, as a set of individually verifiable, traceable, solution-free statements. One Requirements document per non-leaf scope. It is the formalisation layer — the place where stakeholder intent, product outcomes, user stories, and upstream constraints become sentences that Architecture can allocate to children, tests can verify, and a junior engineer or a mid-tier AI can implement without guessing.

Position on the V

Requirements live on the left-hand descent of the V, one at every non-leaf scope. The root-scope Requirements document turns Product Brief outcomes, operational scenarios, NFRs, and constraints into testable behaviour specifications. Each branch-scope Requirements document refines the parent's allocated slice for a subsystem. Leaves have no Requirements artifact of their own — they hold a Detailed Design instead — but every leaf DD traces upward to requirements at the nearest non-leaf ancestor.

The right-hand ascent verifies the left. Integration tests at a branch verify that branch's requirements. System / end-to-end tests at the root verify root-scope requirements. Unit tests at a leaf verify the DD contracts that implement a subset of the branch requirements. Without requirements as the organising axis, there is nothing specific enough to derive tests from and nothing coarse enough to evaluate an architecture against.

User stories and requirements are not the same thing

Most teams elicit with user stories ("As a returning customer, I want a fast checkout so that I don't abandon the cart."). User stories are excellent for stakeholder conversation and backlog prioritisation; they are a conversation token, deliberately under-specified so the conversation fills in the detail.

Requirements are different: they are the formalisation of what those conversations settled on. A single user story typically produces several requirements — a functional one about the happy path, one or more unwanted-behaviour requirements for error paths, an NFR about latency or throughput, an interface requirement if the story touches an external system. The user story is retained as input (INVEST, backlog item, conversation record); the requirement is what Architecture, Detailed Design, and tests hang off.

Example: "As a returning customer, I want one-click checkout so that I don't retype my details" becomes functional ("When a returning customer with a saved payment method initiates checkout, the system shall pre-populate billing and shipping from the most recent completed order"), NFR ("The checkout initiation shall complete in ≤ 400 ms at p95 for sessions with cached payment tokens, measured at the API boundary"), and unwanted-behaviour ("If the saved payment method is expired, then the system shall prompt for a new payment method and shall not silently fall back to manual entry"). Three requirements, one user story. All three trace back to the user story as derived_from; the user story itself traces back to the Product Brief outcome it serves.

What a Requirements artifact is not

  • Not a backlog. A backlog is work to be done, ordered by value / effort / dependency. Requirements are behavioural specifications, independent of when anyone will build them.
  • Not a design. Requirements say what, not how. Naming a database, a framework, a specific algorithm, or a data structure inside a requirement statement is a category error — that is ADR, Architecture, or DD territory.
  • Not test cases. Requirements say what the system must do; test cases say how you would prove it. The two are cleanly separated: requirements live here; test cases live in TestSpec. A requirement does not list its test cases inline.

Retrofit posture

Requirements retrofitted from existing code are mixed evidence: functional behaviour is largely recoverable from code and tests; rationale is not. The framework's rule is explicit — rationale is human-only. The shape of a retrofit Requirements document, the recovery_status discipline, and the fabrication rule are taught in detail in §2.8 and demonstrated in §4.7.

Why the framework treats this layer as non-negotiable

The Spec Ambiguity Test applies universally: every specification artifact at every layer must let a junior engineer or mid-tier AI implement without guessing. Requirements are the layer where this test bites hardest. Above Requirements (Product Brief), the artifact is narrative and stakeholder-facing; ambiguity can be negotiated in conversation. Below Requirements (Detailed Design), the artifact is already specific to code-producing granularity. Requirements are the translation layer between them. If they are vague, everything below compensates by guessing, and the guesses drift out of alignment with stakeholder intent. No amount of careful downstream design or testing recovers from ambiguous requirements — it only hardens whichever interpretation got picked.

2. Best Practices

Eight disciplines. Vocabulary first, because every subsequent discipline depends on terms that mean the same thing to every reader. Type taxonomy next, because the remaining craft differs by type. Then four type-specific disciplines (structured language, statement-level quality, NFRs, interfaces) and one for constraints. Finally the lifecycle discipline — rationale, derivation, traceability, inspection — that keeps the artifact honest over time.

Frontmatter and shape

A Requirements artifact is a single Markdown file per non-leaf scope at /specs/{scope}/requirements.md (root-scope lives at /specs/requirements.md). YAML front-matter identifies the artifact; the body holds prose sections and embedded YAML blocks per requirement. Canonical front-matter:

FieldPurpose
idREQS-{scope}; at root, REQS.
scopeScope path this artifact covers.
parent_scopeParent scope path; null at root.
derived_fromUpstream artifacts (Product Brief at root; parent Requirements at branches).
governing_adrsCross-cutting ADRs that constrain this scope.
statusdraft → accepted. Re-enters draft on material edit.
dateDate of last acceptance.
recovery_statusPer-section map, retrofit only. verified | reconstructed (behaviour fields) | unknown (rationale and other human-only fields).

Per requirement, the embedded YAML block carries id, statement, rationale (required), acceptance (optional; required when the statement alone leaves test conditions ambiguous), and derived_from (the upstream user story, stakeholder need, or parent requirement).

2.1   Vocabulary discipline — glossary and ubiquitous language

Before any requirement statement has meaning, the terms it uses must mean the same thing to every reader. This is Evans' ubiquitous language adapted to the requirements layer: a single vocabulary, binding across conversation, requirements text, architecture diagrams, code, and tests, scoped to one bounded context. Within that scope there is one word per concept and one concept per word. Different scopes are free to use the same word with different meanings; that is the whole point of a bounded context.

The artifact discipline is short and non-negotiable: every non-trivial Requirements document opens with a Glossary section that names every domain term with a single definition. A term used in a requirement that is not in the glossary is either slang, a synonym of a glossary term (pick one and stick to it), or an undefined concept that needs to be added. Terms should be model-anchored — they name things in the domain the system reasons about, not things in the implementation.

glossary:
  - term: "Session"
    definition: "A server-side record of an authenticated user's continuous interaction
                 with the system, scoped to a single device and identified by an opaque
                 session token. A session is created on successful authentication and
                 destroyed on logout, idle timeout, or explicit invalidation."
    model_refs: [Session entity in the Identity bounded context]

  - term: "Idle timeout"
    definition: "A session invariant: the maximum duration between the end of one
                 authenticated request on the session and the start of the next
                 before the session is destroyed. Distinct from absolute timeout."

  - term: "Absolute timeout"
    definition: "A session invariant: the maximum duration from session creation
                 to forced destruction, independent of activity."

  - term: "Elevated session"
    definition: "A session that has completed step-up authentication (e.g., second factor
                 or re-entry of password) within the last N minutes, making it eligible
                 to perform security-sensitive operations. Distinct from Session."

The payoff is directly requirements-level. "While the user's session is idle, the system shall…" is only meaningful if session and idle mean one thing. Two readers who each imagine a different definition produce two different tests, two different designs, and an incompatible integration.

Evans' observation that vocabulary fragments quickly — "as few as three or four people" working together can produce divergent dialects — applies doubly in an AI-augmented workflow. LLM-generated requirement text defaults to generic-programmer vocabulary (Manager, Service, Handler, Record) unless the project vocabulary is established and enforced. The glossary is the project's statement that this drift is not acceptable, and the reference that review tools (human or AI) can check against. Vocabulary is checked before any other quality discipline applies: statement-level craft cannot salvage a requirement built on undefined terms.

EARS versus glossary — sentence vs. vocabulary

Structured requirements languages like EARS discipline the sentence: one verb, named actor, specified trigger, specified response. The glossary disciplines the vocabulary the sentence is built from. A grammatically perfect EARS statement built on a sloppy vocabulary is still ambiguous; rigorous vocabulary embedded in prose without structure is still untestable. The two are complementary and both load-bearing.

2.2   Requirement type taxonomy

The framework's Requirements artifact has five fixed sections, one per requirement type. Authors and reviewers must know the distinction — a requirement filed in the wrong section silently weakens the artifact because each section has different rigor rules and different downstream consumers.

TypeStatesVerified atExample (session scope)
Functional What the system does in response to stimuli or in a given state. Unit + integration + system tests, depending on scope. "When the user submits valid credentials, the system shall create a session and return its token."
Quality attribute / NFR A measurable non-functional characteristic under specified load and environment. Performance / load / chaos tests; static analysis for some. "The session-validation endpoint shall respond in ≤ 50 ms at p95 under 5,000 concurrent sessions, measured at the API gateway."
Interface A contract with an external caller, dependency, or protocol peer. Contract tests; integration tests at the interface. "The authentication callback shall accept OIDC authorization-code responses per RFC 6749 §4.1.2, and shall reject responses whose state parameter does not match the one issued."
Data Format, retention, privacy class, integrity invariants of data the system holds or exchanges. Schema tests; data-layer tests; compliance audits. "Session records shall retain the user's IP address for no more than 90 days from last activity, after which the IP field shall be deleted while the session history record is preserved."
Inherited constraint Bounds imposed from above or outside — Product Brief, governing ADR, regulatory regime — that this scope must honour. Reviewed; some are architecture-testable, some are process-verified. "Per ADR-012, all session token generation shall use a CSPRNG; direct use of java.util.Random or equivalent is prohibited."

Two level-confusion errors recur often enough to name. First, NFRs written as functional requirements: "The system shall be fast" reads as functional but is an untestable NFR. Second, design smuggled as a requirement: "The system shall use a distributed cache to store session tokens" reads as functional but is an architectural decision. Move the first to § QA with a measurable target; move the second to ADR / Architecture and, if it implies a testable behaviour at this layer, add the requirement that tests that behaviour.

2.3   Structured language — EARS as default, GWT where useful

Free-form natural language makes roughly half the requirements it produces ambiguous. EARS (Easy Approach to Requirements Syntax) is five sentence templates that cut most of the ambiguity by forcing the author to classify the requirement before writing it. The templates are learnable in under an hour and are widely adopted across aerospace, automotive, medical, and general software.

PatternTemplateWhen to use
Ubiquitous The <system> shall <response>. Always-on invariants. Behaviour that is true without a trigger or a state condition.
Event-driven (When) When <trigger>, the <system> shall <response>. Stimulus-response. The trigger is a discrete event; the response fires once per trigger.
State-driven (While) While <state>, the <system> shall <response>. Behaviour that persists for the duration of a named mode or condition.
Optional-feature (Where) Where <feature>, the <system> shall <response>. Behaviour conditional on a configuration or deployment variant.
Unwanted-behaviour (If/then) If <condition>, then the <system> shall <response>. Error paths, fault responses, security violations. Every "shall not" candidate should be rephrased as a positive "shall" under an unwanted condition.

Compound templates

Keywords combine in a canonical order: Where → While → When → If/then → shall. Use at most two keywords per statement; three or more signals that the statement is carrying multiple requirements that need to be split. "Where the tenant plan is Enterprise, while the user's session is elevated, when the user requests a bulk export, the system shall queue the export and return a job token" is a two-keyword compound (Where + When; while the user's session is elevated is a precondition, not a third template keyword — the author should still verify that two is the ceiling).

When EARS is not enough

EARS handles functional behaviour and unwanted behaviour well. It is less natural for:

  • Measurable NFRs with multi-level targets. Use the five-element performance rule (§ 2.5) or Planguage's GOAL / STRETCH / WISH / FAIL scale to express tiered quality attributes.
  • Multi-step scenarios with pre- and post-conditions. Given-When-Then (GWT) is the natural shape. GWT maps cleanly to EARS when needed: Given ↔ While (state precondition), When ↔ When (trigger), Then ↔ shall (response). Teams familiar with BDD can write acceptance criteria in GWT and derive the EARS requirement from them, or vice versa.
  • Temporal / duration expressions beyond simple triggers. "Within 5 seconds of…", "for no more than 10 minutes after…" — EARS does not forbid them but their detection and enforcement is author discipline, not template.

The default is EARS for the requirement statement; GWT is available as a form for the optional acceptance block when the statement alone leaves conditions ambiguous; Planguage appears inside NFR target specifications. Do not mix two structured forms inside one statement.

2.4   Statement-level quality — atomic, testable, solution-free

Every individual requirement statement must clear three gates.

  1. Atomic. One shall per statement, verifying one behaviour. Compound statements ("…shall X and shall Y") break atomicity and are the single most common anti-pattern. If a test would need two independent check groups to pass the requirement, split it.
  2. Testable. A tester reading the statement alone can write a test that distinguishes conforming from non-conforming behaviour. The box test is the author's own check: can I, reading only this statement and the glossary, sketch the test? If the answer is "I would need to ask someone", the statement is not yet testable.
  3. Solution-free. The statement constrains what the system does, not how. Naming a database, framework, library, data structure, or algorithm inside the statement is a category error; move it to ADR or Architecture. The one exception is interface requirements, which legitimately specify externally visible protocols and message formats — an externally-imposed interface is what, not how.

Conditional completeness — state-driven pairs

Every state-driven requirement (While <state>…) implicitly raises the question: what does the system do outside that state? A Requirements document that says "While the session is elevated, the system shall allow changes to payment methods" but never states what happens outside the elevated state has a latent gap; a reviewer has no way to tell whether the behaviour is "allow it with a prompt", "deny it", or "deny silently". Write the complementary pair whenever the state axis is load-bearing: "While the session is elevated, the system shall allow…" paired with "While the session is not elevated, the system shall prompt for step-up authentication before allowing…". State-driven requirements without complements are among the most common findings in requirements review.

2.5   Quality attributes at requirement granularity

The Product Brief page covers quality-attribute scoping: the NFR-10 floor list (availability, latency, throughput, security, privacy / compliance, data, compatibility, extensibility, operability, cost) and the six-component quality-attribute scenario form (stimulus / source / environment / artifact / response / response measure). This page covers what changes when those scenarios harden into individual, testable requirement statements.

The five-element performance rule

An NFR statement that is load-bearing — one a reviewer could approve or reject, not a wish — contains five elements:

  1. The system or subsystem the requirement applies to, named specifically.
  2. The response or behaviour being measured.
  3. The metric with unit the measurement produces (milliseconds, requests/second, bytes, nines of availability).
  4. The target value, expressed at the correct statistical level (mean is almost never right for latency; prefer a stated percentile).
  5. The condition under which the measurement is valid — load, environment, operating mode, measurement point.
# All five elements, in order:
"The session-validation endpoint [system]
  shall respond [response]
  in ≤ 50 ms [metric + unit]
  at p95 [statistical level on target]
  under 5,000 concurrent sessions in the production deployment,
  measured at the API gateway [condition]."

The condition is the element most often omitted. Without it, the metric is not reproducible: "the system shall respond in ≤ 50 ms" can be trivially satisfied under 1% load and trivially failed under 100% load, and no test configuration is defensible.

Multi-level targets — Planguage

Binary pass/fail NFRs force an all-or-nothing trade-off at implementation time. For attributes where the right target is a range — nearly all performance and availability targets — Planguage's multi-level form carries the trade-off into the requirement itself:

availability (session-validation endpoint):
  scale:     percentage of authenticated-user-facing requests completed within SLO
  meter:     computed over 30-day rolling window, excluding planned maintenance windows
  fail:      < 99.5%     # below this the system is considered in violation
  goal:      ≥ 99.9%    # target the team commits to
  stretch:   ≥ 99.95%   # desirable but not committed
  wish:      99.99%       # aspirational; not a commitment

The key discipline Planguage imposes is the separate scale and meter fields. A target without a scale (what is being measured) and a meter (how and over what window) is a number, not a measurement.

Compliance constraints translate to derived requirements

Regulatory regimes (GDPR, HIPAA, PCI-DSS, WCAG, SOC 2, CCPA) rarely speak in testable statements. They state obligations at a legal or policy level that must be translated into requirements the system and its tests can verify. The translation is a derivation step, captured explicitly.

Example: GDPR Article 17 ("Right to erasure") — a data subject may request that their personal data be erased without undue delay. At the requirements layer:

inherited_constraints:
  - source: "GDPR Article 17 (Right to erasure)"
    summary: "Data subjects have the right to have their personal data erased
              without undue delay on request, subject to limited exceptions."

derived_requirements:
  - id: REQ-SESSION-042
    statement: "When an authenticated data subject submits an erasure request for
                their session data, the system shall destroy all active sessions for
                that subject and shall remove personal identifiers from session history
                records within 7 calendar days of receiving the request."
    derived_from: [Inherited constraint – GDPR Article 17]
    rationale: "GDPR Article 17(1) requires erasure without undue delay; our DPA
                commits to 7 days for session data specifically. Full audit-log records
                are retained under the legal-obligation exception (Article 17(3)(b))
                but personal identifiers are severed per the pseudonymisation practice."

  - id: REQ-SESSION-043
    statement: "If the system retains any session record after an erasure request,
                then the retained record shall contain no directly-identifying fields
                (user ID, email, IP address, device fingerprint)."
    derived_from: [Inherited constraint – GDPR Article 17; REQ-SESSION-042]
    rationale: "Supports the pseudonymisation commitment in REQ-042's rationale;
                legal-obligation exception permits retention of pseudonymised records
                for audit, not of directly-identifying ones."

The pattern: the regulation is captured once as an inherited constraint with its source; each testable behaviour it implies is a derived requirement with a derived_from link back to the constraint. The rationale on each derived requirement explains the choice that translates the general obligation into a specific commitment (here: "7 days" is not in the regulation; it is our DPA commitment derived from "without undue delay").

2.6   Interface contracts

Interface requirements specify the boundary between this system (or scope) and anything outside it: external APIs, event streams, databases the scope does not own, hardware, user agents. Interface requirements are the single most common omission in requirements audits, and the most common failure mode is syntax without semantics — a function signature or a JSON schema with no statement of what actually happens.

Five dimensions every interface requirement covers

  1. Protocol. The communication standard and version (HTTP/1.1, gRPC, AMQP 0.9.1, OIDC 1.0, TLS 1.3, specific OAuth grant type). Protocols are what, not how: externally imposed and therefore legitimately named here.
  2. Message structure. Fields, types, units, ranges, resolutions, defaults, optionality. For APIs: request / response schemas. For queues: message payloads. For databases the scope does not own: read and write contracts.
  3. Timing. Latency budget for the response, rate limits, retry expectations, heartbeat intervals. Timing is part of the contract, not an NFR about the contract.
  4. Error handling. What the interface does when the call fails, times out, returns unexpected data, or hits a rate limit. "The system shall retry up to 3 times with exponential back-off starting at 500 ms, then fail the upstream operation with status DOWNSTREAM_UNAVAILABLE."
  5. Startup / initial state. What the interface does before it has been established — cold start, connection lost, first request after a restart. Interfaces that are silent here have latent bugs that surface in production incidents.

Design by Contract at the requirements layer

For each externally-callable operation, the interface requirement states preconditions (what the caller must guarantee), postconditions (what the system guarantees if the preconditions hold), and invariants (conditions true before and after, always). This is Meyer's Design by Contract applied one layer above code — at the requirements layer, the contract governs testability; at the code layer, DD lifts the same contract into executable assertions.

interface: POST /sessions/{id}/invalidate

precondition:
  - Authenticated caller holds role 'session-admin' OR owns the target session.
  - Session {id} exists and is in state ACTIVE or IDLE.

postcondition (on 200):
  - Session {id} is in state INVALIDATED.
  - Subsequent calls to validate this session-id return 401 with reason 'invalidated'.
  - An invalidation audit event has been appended to the session history record.

postcondition (on 404):
  - Session {id} does not exist OR is already INVALIDATED.
  - No state changes are performed.

invariants:
  - The operation is idempotent; calling it on an already-INVALIDATED session is a 404
    with no state mutation, never a 5xx.
  - No response discloses whether the caller would have been authorised had the session
    existed (prevents session enumeration).

The pre/post form makes the requirement directly test-derivable: each precondition branch is a test-case class; each postcondition is an assertion; each invariant is a property to fuzz or cross-check.

Versioning is a first-class requirement

An interface that is not versioned has silently committed to never-changing, which is a commitment the system cannot keep. The interface requirement names the version (v1, 2026-04), the compatibility regime (semver, date-based, additive-only), and the deprecation policy (how long old versions remain available, how callers are notified). Teams that omit versioning from the requirement discover, at the first breaking change, that there was no written rule for how to change the interface, and every caller's migration story is improvised.

2.7   Constraints craft

Constraints are distinct from requirements. A requirement specifies behaviour the system must exhibit (testable by observing the system); a constraint bounds the space of acceptable solutions (verifiable often by inspection, process, or architecture review rather than by runtime test). The framework keeps them in the Inherited Constraints section, and the craft rules are different from the rules for requirements.

  • Every constraint cites its source. "Must use Java 17" is a constraint; who imposed it and why? ADR-034 (company-wide JVM baseline)? A customer contract? The upstream platform team? A constraint without a source is a preference pretending to be a rule.
  • Every constraint has a rationale that names the cost of relaxing it. If the constraint could be relaxed without cost, it is not really a constraint. Name the cost ("Relaxing this requires a customer-wide re-certification", "Relaxing this creates a dependency on a service the org does not operate"). This is what lets a future review reopen the constraint when circumstances change.
  • Constraints with downstream behavioural consequences propagate as derived requirements. "Must use Java 17" alone changes nothing at the behavioural layer; "Must use Postgres 15 as the primary datastore" combined with the system's latency target may imply derived requirements about query patterns or connection-pool shape. The framework's hybrid propagation rule (ADRs with testable consequences become requirements; ADRs that only bound choices propagate via governing_adrs) applies identically to inherited constraints.
  • Constraint categories. Technical (platforms, frameworks, protocols mandated), regulatory / legal (GDPR, HIPAA, contractual obligations), organisational (company-wide policies, coding standards, procurement), financial (infrastructure budget ceilings, vendor cost caps), temporal (hard launch dates, freeze windows). Categorising helps reviewers spot under-represented categories — most teams over-list technical and under-list regulatory.

2.8   Rationale, derivation, traceability, inspection

The lifecycle discipline: how requirements start honest and stay honest.

Rationale is mandatory and captured at decision time

Every requirement has a rationale field. The rationale is not a restatement of the requirement; it is the why — the context, the options that were considered, the criterion that drove the choice. Good rationale distinguishes essential constraints from accidental ones, which is the distinction that makes future maintenance safe. Without rationale, every requirement in the system looks equally non-negotiable, and the team that inherits the artifact has to either over-conserve everything or guess at what was load-bearing.

Rationale is captured at decision time, not retroactively. Rationale produced years after the fact is lower-fidelity by an order of magnitude; the options the original author considered have been forgotten, and the human brain readily invents plausible justifications for whatever decision got made. Three failure modes are specific and worth naming:

  • LLM confident invention. An agent asked to generate rationale for an existing requirement produces a plausible-sounding justification that is not grounded in any decision ever made. The rationale reads right and is entirely fabricated.
  • Test-as-requirement inversion. A characterisation test written against existing behaviour gets treated as the "requirement", and the rationale says "the system shall do X because that is what test T verifies". The test is verifying the current code, not the original intent; the rationale is circular.
  • Laundering the current state. Rationale that explains why the current design is right rather than why the original choice was made. The current state is a fact; a rationale that only defends it is a ratification, not a reason.

The honest retrofit posture: rationale is a human-only field, and its retrofit recovery_status is verified (a human confirmed it from preserved documents or conversation) or unknown. Never reconstructed. A retrofit Requirements document with many rationale: unknown entries is correct — it honestly reports what is lost — not partial.

Derived requirements

A derived requirement is one that does not come from an upstream stakeholder need or parent requirement directly, but from a design or architectural decision at this scope that introduces a new constraint. "Per ADR-012 (token generation), the system shall use a CSPRNG" is derived — no stakeholder requested it, but the ADR's consequences require it be testable here. Derived requirements are flagged explicitly (derivation: derived) and their rationale cites the decision that introduced them. The framework refuses the pattern where derived requirements accumulate silently — every one of them is surfaced for review, because the easiest way to drift out of stakeholder alignment is to quietly add requirements that no upstream party ever agreed to.

Bidirectional traceability is the working artifact

Traceability is not paperwork; it is the mechanism that makes every other discipline provable. Forward: every requirement has a derived_from (user story, parent requirement, Product Brief outcome, inherited constraint). Backward (computed by tooling, not authored): every stakeholder need resolves to at least one requirement, every requirement to at least one allocated Architecture child and to at least one verifying test case. Missing links in either direction are findings — an orphan requirement is unjustified; an unverified requirement is unenforced.

A completeness check worth running on every Requirements artifact before review:

  • Every requirement has derived_from non-empty.
  • Every inherited constraint has a source.
  • Every governing ADR is referenced by at least one requirement or at least one derived requirement.
  • Every requirement has rationale non-empty (or recovery_status: unknown in a retrofit, never both empty).
  • Every NFR has the five elements present (system, response, metric + unit, target, condition).
  • Every interface requirement has the five dimensions present (protocol, message, timing, error, startup).

Inspection — Perspective-Based Reading

Ad-hoc review misses the defects that the author is blind to. Perspective-Based Reading (PBR) is a scenario-based inspection technique with three reviewer roles, each reading the same artifact from a different stance. Each reviewer looks for defects that the stance makes salient and the other stances miss.

  • Designer perspective. "If I were allocating this to children, could I?" Surfaces requirements that are impossible to decompose, that conflict, or that hide design decisions.
  • Tester perspective. "If I were deriving test cases from this, could I?" Surfaces ambiguity, untestable statements, missing conditions, and implicit acceptance criteria.
  • User / stakeholder perspective. "Does this capture what the stakeholder actually wants?" Surfaces outcomes the requirements fail to express, or over-specification that constrains without justification.

Empirical evidence (Shull et al., NASA GSFC, 1995–2000) shows PBR detects defect classes ad-hoc review consistently misses, with effort comparable to or modestly higher than informal walkthroughs. The discipline is cheap to adopt and the yield is specifically the kind of latent ambiguity that cascades into expensive downstream defects. PBR is the default review practice for Requirements artifacts in this framework.

3. Anti-Patterns

Nine universal failure modes in individual statements; seven specific to AI-augmented authoring and retrofit. The universal nine are well-catalogued across requirements research and audit findings. The AI-era seven are newer and the shape of future audit findings; they deserve explicit naming because their remedies differ from the classical ones.

Universal (nine)

  1. Compound requirement. Two shalls in one statement, joined by and, or, or an enumerated list. "The system shall authenticate the user and shall log the attempt." Split.
  2. Vague adjective. Fast, sufficient, appropriate, adequate, user-friendly, robust. The statement reads approvable but no test distinguishes pass from fail. Replace with a measured quantity or remove.
  3. Passive voice without actor. "Session data shall be validated." By whom, at what point, against what? Rewrite with a named subject.
  4. Unbounded list / escape clause. "Such as X, Y, Z, etc.", "including but not limited to", "as appropriate", "where applicable". Closes no option space, commits to nothing.
  5. Unbounded negative requirement. "The system shall not crash." The negative cannot be exhaustively verified, the scope is infinite. This is the anti-pattern — not negative requirements in general. Bounded negatives are legitimate and often necessary: "The system shall not log session tokens in plaintext to any destination" is bounded (tokens, plaintext, all destinations), verifiable (log-pattern review + audit), and load-bearing for security. The discipline: rewrite unbounded negatives as positive unwanted-behaviour EARS statements (If X, then the system shall Y); keep bounded negatives and verify them where they make sense.
  6. Implementation prescription. "The system shall use a distributed in-memory store for session tokens." This is ADR content, not a requirement. Move it, and replace here with the behaviour the prescription was trying to ensure.
  7. Wishful thinking. "The system shall process requests with zero latency", "shall handle infinite concurrent users". Impossible targets are not requirements; they are aspirations. Honest targets are bounded.
  8. Ambiguous pronoun or reference. "When the request arrives, the system shall validate it and return an appropriate response." It could be the request payload, the headers, the authentication token; appropriate is vague. Rewrite with explicit referents.
  9. TBD / TBR / TODO in a baselined requirement. A requirement with an open placeholder cannot be tested, cannot be allocated, cannot be reviewed. Either resolve the TBD or remove the requirement from the baseline until it is ready.

AI-era and retrofit (seven)

  1. LLM-fabricated rationale. An agent asked to retrofit or generate requirements produces a plausible-sounding rationale for every statement, regardless of whether any decision record or human source supports it. The rationale reads competent and is entirely synthesised from training priors. Tell: perfect-grammar rationales on retrofit requirements for which no preserved conversation or document exists; rationales that never say "because the original team chose X over Y" — only "because X is better than Y", which is reasoning, not recall.
  2. EARS cargo-culting. Statements that obey EARS grammar with no underlying discipline — vague trigger, vague response, glossary-free terms. "When the system experiences load, the system shall respond appropriately." Structurally event-driven; semantically vacuous. The remedy is not "write better EARS" — it is to reject EARS as a shield against thinking and require the author to pass the box test.
  3. Requirements smuggling design. A generated requirement contains an implementation choice disguised as behaviour: "The system shall store session state in Redis with 5-second TTL". The Redis and 5-second TTL are design, not requirement — but in LLM output they are sticky because training data blends the two. Review specifically for named technologies, named data structures, and named algorithms inside requirement text.
  4. Code-driven requirement fabrication. An agent asked "derive requirements from this code" reads existing behaviour and produces requirements that describe whatever the code happens to do, including bugs and accidental features. The requirements then justify the bugs as intentional. The honest retrofit posture is the opposite: behaviour observable from code is captured with recovery_status: reconstructed; intent behind the behaviour is marked unknown until a human supplies it.
  5. Level confusion. The agent mixes stakeholder need ("the finance team wants fewer errors"), system requirement ("the system shall…"), and subsystem / component requirement ("the reconciliation job shall…") in the same document without distinction. The remedy is to enforce the scope axis explicitly: every requirement's scope matches this artifact's scope; parent-scope concerns are upstream derived_from references, not requirements here.
  6. Test-as-requirement inversion. A characterisation test written against existing behaviour is promoted to "the requirement", closing the loop with no stakeholder in it. The rationale becomes circular: the requirement exists because the test exists. Symptoms: rationale fields that cite tests but never cite a decision; requirements that exactly mirror current behaviour including its quirks.
  7. Laundering the current state. Requirements that exactly describe the deployed system, with rationale that defends the current design as if it were the only possible choice. Useful for auditing what currently exists; deceptive when presented as intended behaviour. Tell: no requirement ever deviates from what the code does; no rationale ever cites a rejected alternative.

4. Examples

All examples use a single worked domain: a session-management service within a larger e-commerce platform. The service authenticates users, issues and validates session tokens, enforces session policies (idle timeout, absolute timeout, step-up for elevated operations), and integrates with an OIDC identity provider and an audit-log consumer.

4.1   EARS templates — one per pattern, plus a compound

# Ubiquitous (invariant)
REQ-001: The session service shall use a cryptographically secure pseudo-random
         number generator for all session token generation.

# Event-driven (When)
REQ-002: When a user submits a valid credential pair, the session service shall
         create a new session, persist it in the ACTIVE state, and return the
         session token and its absolute-timeout expiry timestamp.

# State-driven (While)
REQ-003: While a session is in the IDLE state, the session service shall reject
         validation requests for that session with status 401 and reason
         'idle-timeout'.

# Optional-feature (Where)
REQ-004: Where the tenant has enabled multi-factor authentication, the session
         service shall require step-up authentication before promoting a session
         to the ELEVATED state.

# Unwanted-behaviour (If/then)
REQ-005: If three consecutive credential-validation attempts for the same user
         account fail within 10 minutes, then the session service shall lock
         the account for 15 minutes and shall emit a 'credential-lockout' audit
         event.

# Compound (Where + When)
REQ-006: Where the tenant plan is Enterprise, when an elevated session requests
         a bulk export larger than 10 MB, the session service shall enqueue the
         export to the long-running-job queue and shall return a job token with
         status 202.

4.2   User story → requirements translation

# Input — user story (elicitation)
"As a returning customer, I want my session to stay active while I'm shopping so
 that I don't have to log in again mid-checkout."

# Output — three requirements (formalisation)

REQ-010 (functional — state-driven):
  statement: "While a session has recorded authenticated activity within the last
              {idle_timeout} minutes, the session service shall keep the session
              in the ACTIVE state."
  rationale: "Supports the returning-customer outcome from PB-Outcome-3 (keep
              authenticated users from being logged out mid-task). Idle timeout
              is parameterised per ADR-018 (session-policy configuration)."
  derived_from: [user-story-US-42, PB-Outcome-3, ADR-018]

REQ-011 (NFR — latency):
  statement: "The session-validation endpoint shall respond in ≤ 50 ms at p95
              under 5,000 concurrent sessions in the production deployment,
              measured at the API gateway."
  rationale: "Validation runs on every authenticated request. PB-NFR-latency
              commits to ≤ 400 ms end-to-end per request at p95; the validation
              budget of 50 ms leaves headroom for downstream operations."
  derived_from: [user-story-US-42, PB-NFR-latency]

REQ-012 (unwanted-behaviour — error path):
  statement: "If a validation request arrives for a session that has exceeded
              its absolute-timeout, then the session service shall return 401
              with reason 'absolute-timeout' and shall not extend the session."
  rationale: "Absolute timeout is a hard security boundary (ADR-018); even an
              actively-used session must be re-authenticated once a day."
  derived_from: [user-story-US-42, ADR-018]

One user story, three requirements, each testable on its own, each traceable back to the story and forward to the architecture allocation and tests. The user story itself is preserved as elicitation input — it lives in the product backlog, not in this Requirements document.

4.3   Five-element performance rule — before and after

# Before — three elements missing
"Session validation shall be fast."
                 ← No metric, no target, no condition. "Fast" is an adjective.

# After — all five elements present
REQ-015: "The session-validation endpoint          ← 1. system named
            shall respond                          ← 2. response named
            in ≤ 50 ms                          ← 3. metric + unit
            at p95                                 ← 4. target at a stated percentile
            under 5,000 concurrent sessions in     ← 5. condition
            the production deployment,                 (load + environment +
            measured at the API gateway."              measurement point)

  rationale: "Validation is on the hot path of every authenticated request.
              The 50 ms p95 budget derives from the PB-NFR-latency commitment
              of 400 ms end-to-end at p95 and the eight downstream operations
              sharing the budget (roughly 45 ms each plus overhead)."

4.4   Interface requirement — five dimensions

REQ-020: Session-validation interface

PROTOCOL:
  The session service shall expose session validation via HTTP/1.1 POST to
  '/sessions/validate' with TLS 1.3, API version v1.

MESSAGE STRUCTURE:
  request body:
    token        required, string, opaque session token, 64-128 chars
    purpose      optional, enum { 'read', 'write', 'elevated' }, default 'read'
  response body (200):
    valid        required, boolean
    subject      required on valid=true, string, stable user identifier
    session_state required on valid=true, enum { ACTIVE, IDLE, ELEVATED }
    reason       required on valid=false, enum { 'expired', 'invalidated',
                                                 'not-found', 'idle-timeout',
                                                 'absolute-timeout' }

TIMING:
  Response within 50 ms at p95 under normal load (see REQ-015).
  Requests lasting > 500 ms shall be timed out server-side with status 504.

ERROR HANDLING:
  On downstream store unavailability, the service shall retry the internal
  read up to 2 times with 50 ms back-off, then fail with 503 and header
  'Retry-After: 1'. The service shall never silently return valid=false on
  infrastructure failure — that condition is 5xx, not 401.

STARTUP / INITIAL STATE:
  On cold start before the backing store is reachable, the service shall
  respond 503 with body { 'ready': false, 'reason': 'warming' } and shall
  fail the '/health/ready' probe until the store has been reachable for
  10 consecutive seconds.

PRE/POST (DbC):
  precondition:    caller holds valid API credentials (handled by gateway).
  postcondition:   on 200 with valid=true, the session's last-activity
                   timestamp has been updated iff purpose != 'read'.
  invariant:       the response never discloses the subject on valid=false
                   (prevents session-existence enumeration).

VERSIONING:
  Additive-only changes within v1 (new enum values require client handling).
  Breaking changes increment the major version; v1 remains available for
  minimum 12 months after v2 is announced.

4.5   Compliance constraint → derived requirements (GDPR Article 17)

inherited_constraints:
  - id: IC-GDPR-17
    source: "GDPR Article 17 (Right to erasure)"
    summary: "Data subjects may request erasure of their personal data without
              undue delay, subject to Article 17(3) exceptions."
    category: regulatory
    rationale_for_binding: "The service processes personal data (email, IP, device
                            fingerprint) within session records; GDPR applies to
                            all EU data subjects using the platform. Cost of
                            relaxing: DPA breach, supervisory-authority
                            intervention, material fines (up to 4% global turnover)."

# Derived requirements

REQ-050 (data — retention + erasure action):
  statement: "When an authenticated data subject submits an erasure request via
              the account-settings interface, the session service shall destroy
              all ACTIVE and IDLE sessions belonging to that subject and shall
              remove directly-identifying fields (user_id, email, ip_address,
              device_fingerprint) from all session history records for that
              subject within 7 calendar days of receiving the request."
  rationale: "Article 17(1) requires erasure without undue delay. Our DPA
              (section 6.3) commits specifically to 7 days for session data.
              History records are pseudonymised rather than deleted to preserve
              the audit-log legal-obligation exception (Article 17(3)(b)).
              Alternatives considered: full deletion of history (rejected on
              audit-log grounds); 30-day window (rejected as outside DPA
              commitment)."
  derived_from: [IC-GDPR-17]
  derivation: derived

REQ-051 (unwanted-behaviour — no partial retention):
  statement: "If the session service retains any session history record for a
              subject after an erasure request has been processed, then the
              record shall contain no directly-identifying fields."
  rationale: "Supports REQ-050's pseudonymisation commitment; protects against
              partial-erasure defects where some identifying fields are cleared
              and others are missed. Every history record schema change must
              re-verify this invariant."
  derived_from: [IC-GDPR-17, REQ-050]
  derivation: derived

REQ-052 (interface — request acknowledgement):
  statement: "When the account-settings interface receives an erasure request,
              the session service shall return status 202 with a request
              correlation-id within 2 seconds, and shall emit a
              'erasure-requested' audit event with that correlation-id."
  rationale: "Article 15(3) requires timely acknowledgement. The audit event
              supports proof-of-compliance for supervisory-authority
              inspections."
  derived_from: [IC-GDPR-17]
  derivation: derived

The translation pattern: one regulatory constraint, three derived requirements across three types (data, unwanted-behaviour, interface). Each derived requirement names the specific commitment the system makes (7 days, pseudonymisation, 202 acknowledgement) and cites the place that commitment was made (our DPA, our architecture, our audit practice). A reviewer can trace GDPR Article 17 through to the test cases in TestSpec that will verify the system honours it.

4.6   Anti-pattern before / after

# 1. Compound
BEFORE: "The system shall authenticate the user and shall log the attempt."
AFTER:
  REQ-A: "When a user submits a credential pair, the session service shall
          validate it against the identity provider."
  REQ-B: "When the session service validates a credential pair, it shall emit
          a 'credential-validation' audit event with outcome (success|failure)
          and user identifier."

# 2. Vague adjective
BEFORE: "The session service shall be highly available."
AFTER:  "The session-validation endpoint shall achieve ≥ 99.95% availability
         over any rolling 30-day window, excluding planned maintenance
         windows declared at least 48 hours in advance via the status page."

# 3. Passive voice without actor
BEFORE: "Session tokens shall be invalidated on logout."
AFTER:  "When a user logs out, the session service shall transition the
         session to INVALIDATED and shall reject all subsequent validation
         requests for that session with status 401 and reason 'invalidated'."

# 4. Unbounded negative
BEFORE: "The session service shall not fail."          ← infinite scope,
                                                          untestable
AFTER (bounded, rewritten positive):
        "If the session service is unable to reach the backing store, then it
         shall return 503 to validation requests with 'Retry-After: 1',
         shall fail the readiness probe, and shall not return 200 with
         valid=false (which would incorrectly signal 'session is invalid'
         rather than 'service is unhealthy')."

# 5. Implementation prescription
BEFORE: "The session service shall use Redis with a 5-second TTL for active
         session state."
AFTER:  Move the storage choice to ADR-018 (session-store choice) and
        ADR-024 (freshness policy). Replace here with the behaviour:
        "While a session is in the ACTIVE state, the session service shall
         reflect session-state changes to all nodes within 5 seconds of
         commit, measured at p99 under normal load."

4.7   Retrofit / AI-era side-by-side — fabricated vs. honest

A team runs a retrofit agent against a four-year-old internal session service. The code is intact; the original designers have left; no decision records are preserved.

# BAD — LLM-fabricated retrofit requirement

REQ-R-001:
  statement: "The session service shall use a 30-minute idle timeout for all
              sessions."
  rationale: "30 minutes balances security (limiting session hijack windows)    ← Plausible, generic,
              with user experience (avoiding frequent re-authentication).          and entirely invented.
              This is an industry-standard timeout value chosen after              No decision record
              consultation with the security team."                                exists citing a 30-min
                                                                                   choice. No security
                                                                                   team was consulted for
                                                                                   this retrofit. The
                                                                                   agent generated

                                                                                   "sounds right".
  derived_from: [user-experience, security]     ← Vague, non-artifact references. Not real links.

                                                  outcome.
  rationale_recovery_status: unknown          ← Only 'verified' or 'unknown' allowed here.

               timeout is the intended policy;     to fill it.

  derived_from: [observed_behaviour: session_service.py:L245-L268,

The honest retrofit tells a future reader exactly what is known and what is not: the behaviour is confirmed from code, the reasoning behind the behaviour is not preserved, and a specific human owner has been queued to close the gap. A team reading the honest version can tell at a glance that REQ-R-001 is operationally correct but its "why" needs a human. A team reading the fabricated version has no idea that the rationale was generated by an agent with no source — and a year later will cite the fabricated rationale in a review meeting as "what we decided".

5. Quality Bar

Yes/No checklist, grouped by concern. A Requirements artifact passes when every applicable item is answered Yes. The meta-gate at the end — the Spec Ambiguity Test — overrides all of them: an artifact that ticks every box but cannot let a junior engineer or mid-tier AI derive defensible Architecture, DD, and TestSpec from it fails.

Vocabulary discipline

  • Is a Glossary section present and non-empty for every non-trivial artifact?
  • Is every domain term used in a requirement statement defined in the glossary?
  • Is there one word per concept and one concept per word within this scope?
  • Are generic-programmer placeholder terms (Manager, Service, Handler, Record) audited out of statements except where they name actual domain concepts?

Requirement type clarity

  • Is every requirement filed under the correct type section (functional / QA / interface / data / inherited constraint)?
  • Are NFRs written in NFR form (not as functional statements with an adjective) and functional requirements in functional form (not as NFRs with a behaviour smuggled in)?
  • Has level confusion been audited — does every requirement's scope match this artifact's scope?

Statement-level quality

  • Is every statement atomic (one shall, one behaviour)?
  • Does every statement pass the box test (a tester could write the test from the statement and the glossary alone)?
  • Is every statement solution-free (no named technologies, frameworks, libraries, data structures, or algorithms, except in interface requirements where the protocol is externally imposed)?
  • Is every state-driven requirement paired with its complementary out-of-state behaviour, or is the out-of-state case explicitly out of scope?
  • Is structured language (EARS primary; GWT where scenario-heavy) used consistently rather than mixed within a statement?

NFR measurability — the five-element rule

  • Does every NFR name the system or subsystem specifically?
  • Does every NFR name the response or behaviour being measured?
  • Does every NFR give a metric with unit?
  • Does every NFR give a target value at the correct statistical level (percentile where applicable, not raw mean)?
  • Does every NFR state the condition under which the measurement is valid (load, environment, operating mode, measurement point)?
  • For tiered NFRs, is the Planguage form (scale, meter, fail, goal, stretch, wish) used rather than a single flat target?

Interface contract completeness

  • Does every interface requirement specify protocol, message structure, timing, error handling, and startup / initial state?
  • Are pre- and post-conditions and invariants stated for each externally-callable operation?
  • Is a versioning and deprecation policy stated for every versioned interface?
  • Are externally-imposed protocols cited by specification (RFC, OIDC version, protocol RFC / draft) rather than by informal name?

Constraints discipline

  • Does every inherited constraint cite its source (ADR, regulation, contract, policy)?
  • Does every inherited constraint name the cost of relaxing it?
  • Is every constraint categorised (technical / regulatory / organisational / financial / temporal)?
  • For constraints with behavioural consequences at this scope, are derived requirements authored and cross-linked?

Rationale — no fabrication

  • Does every requirement carry a rationale field with non-trivial content?
  • Is rationale captured at decision time (or, for retrofits, explicitly marked recovery_status: unknown where the original reasoning is lost)?
  • Is rationale in retrofit mode marked with verified or unknown only, never reconstructed?
  • Are rationales audited for circularity (not "because test T passes") and for laundering ("because the current design is right")?

Traceability completeness

  • Does every requirement have non-empty derived_from?
  • Does every derived requirement cite the decision (ADR or architectural choice) that introduced it?
  • Are governing ADRs referenced in the front-matter and reflected in at least one requirement or derived requirement in the body?
  • Has the artifact been run through the traceability validator and returned zero orphan requirements and zero unresolved references?

Inspection discipline

  • Has the artifact been read from at least the designer and tester perspectives before review?
  • Where a stakeholder-facing outcome is at stake, has the user / stakeholder perspective also been read?
  • Is each perspective's review findings logged, so future revisions can check whether a defect class was missed rather than detected-and-resolved?

Retrofit honesty

  • For a retrofit: are behaviour fields marked verified (human-confirmed) or reconstructed (derived from code / tests / schemas)?
  • Are rationale, open alternatives, and intent fields marked verified or unknown only?
  • Does every unknown field have a follow-up owner and action queued?
  • Are derived-from links in a retrofit to observed evidence (file + line; git commit; operational log), not fabricated artifact names?

Spec Ambiguity Test — meta-gate

  • Could a junior engineer or mid-tier AI, reading only this Requirements artifact (plus its glossary and governing ADRs), derive a defensible Architecture allocation to children, a Detailed Design for each leaf, and a TestSpec whose cases verify every requirement — without asking clarifying questions?

If the answer is No, the artifact under-specifies its translation role. Revise before accepting. This test overrides every box above: a Requirements artifact that passes all the other checks but fails this one has not done the job Requirements exists to do.
