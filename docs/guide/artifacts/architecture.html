Non-Leaf Scope Artifact

1. V-Model Context

An Architecture artifact states how a non-leaf scope is decomposed into children, how those children interact, and how they compose into a running system. One Architecture document per non-leaf scope. It is the allocation layer — the place where Requirements become a set of subordinate boxes with responsibilities, interfaces, and a runtime shape; where governing ADRs become concrete structural and behavioural commitments; and where integration and system tests derive their targets.

Position on the V

Architecture lives on the left-hand descent of the V, one at every non-leaf scope. The root-scope Architecture describes how the whole system is decomposed and composed. Each branch-scope Architecture refines its parent's allocated slice for a subsystem. Leaves have no Architecture artifact of their own — they hold a Detailed Design instead — but every leaf DD traces upward to the Architecture of its nearest non-leaf ancestor, receives a share of that ancestor's requirements via allocation, and honours its interface contracts.

The right-hand ascent verifies the left. Integration tests at a branch verify that branch's Architecture: the components interact as specified; the interfaces behave as contracted; the chosen composition pattern holds under load, failure, and concurrency. System / end-to-end tests at the root verify root-scope Architecture against Product Brief outcomes and root Requirements. Unit tests at a leaf are a different axis — they verify Detailed Design, not Architecture. Architecture is the first layer where tests can be integration in the meaningful sense: exercising cross-component behaviour rather than single-unit correctness.

The scope tree and where Architecture sits

A scope tree of variable depth. Root is the whole product. Non-leaf scopes are branches — each with its own Architecture. Leaf scopes are where subdivision stops earning its keep — each with its own Detailed Design. Architecture appears wherever a scope has children to coordinate.

Requirements, Architecture, and TestSpec move together down the non-leaf scopes. Leaves exchange Architecture for Detailed Design. ADRs are flat and cross-cutting — they do not sit inside the tree; they constrain whichever scopes reference them.

Inputs and outputs

An Architecture document consumes four things and produces five. Inputs: the parent's allocation (requirements inherited from the ancestor), the scope's own Requirements artifact, governing ADRs (cross-cutting decisions that bound this scope's choices), and observable / externally-imposed constraints (platform, integration partner, regulatory). Outputs: a decomposition (the set of child scopes with responsibilities); interface contracts between those children and with external parties; a composition specification (runtime pattern, wiring approach, and at root, deployment intent); a traceability footprint (every requirement allocated to at least one child; every governing ADR honoured); and often a set of derived requirements — obligations that arise from the decomposition itself and did not exist at the Requirements layer.

C4 as a bridge

Most working software engineers model architecture with some variant of Simon Brown's C4 model (System Context → Container → Component → Code). The VModelWorkflow Architecture artifact is not a replacement for C4; it is a more opinionated shape that fits neatly on top of it.

Our scope tree is C4's hierarchical zoom: root scope ≈ Context / Container level; branch scopes ≈ Component level; leaves ≈ Code level. Our Structure Diagram (mandatory in every Architecture artifact) is a C4 Container or Component diagram at the scope being documented. Readers familiar with C4 do not have to re-learn anything — they produce the same diagrams they already produce, and our framework adds a per-scope artifact shape around them (Decomposition entries, Interface contracts, Composition section) so that an allocation discipline and a traceability discipline ride alongside the drawings.

If your team already uses arc42: the Structure Diagram fits arc42 §5 (Building Block View); Composition fits arc42 §6 (Runtime View) and §7 (Deployment View); Decomposition entries and Interfaces fit arc42 §5 sub-sections. The rigor of an Architecture artifact is a strict subset of the arc42 template, with the additions that arc42 leaves to taste (interface contracts, traceability links, Quality Bar) made mandatory.

This page teaches craft that holds across domains

Examples throughout this page use a web-service domain (an order checkout service, implementable in Java / Spring Boot but written so the pattern is language-agnostic). This is a choice of shared vocabulary, not a scope limitation. Decomposition, interface design, composition patterns, and deployment intent apply identically to desktop applications (the "orchestration target" becomes installer / packaging; "environments" become OS versions), mobile apps (orchestration target = Play Store / App Store release channel; environments = API levels / device classes), batch and streaming pipelines (orchestration target = scheduler and cluster; runtime-unit boundaries = job stages), and embedded firmware (orchestration target = firmware image and bring-up sequence; environments = board revisions). The artifact shape, the Quality Bar, and the Spec Ambiguity Test are invariant. What changes is the vocabulary the Composition section reaches for when the deployment story is not a container in a cloud.

What an Architecture artifact is not

  • Not a Requirements document. Requirements say what the system must do. Architecture says how it is organised to do it. The two are cleanly separated; behaviour statements belong in Requirements, and structural statements belong here.
  • Not a Detailed Design. Architecture stops at the interface between children. What happens inside a child — algorithms, data structures, error handling within the box — is Detailed Design territory. Crossing that line converts Architecture from a coordination document into a per-child blueprint that will be wrong before anyone reads it.
  • Not a collection of ADRs. ADRs justify individual load-bearing decisions. Architecture applies those decisions to a specific scope: the ADR says "we use an event bus for cross-service notifications"; the Architecture says "in this scope, the pricing component publishes PriceChanged events consumed by the cart component over the bus described in ADR-017." ADRs are referenced via governing_adrs, not inlined.
  • Not a deployment runbook. Deployment intent — environments, orchestration target, runtime-unit boundaries — lives in the root-scope Composition section. Deployment procedure (step-by-step, rollback playbook, on-call response) lives in operational documentation outside this framework.
  • Not a C4 diagram on its own. A box-and-line drawing without per-child responsibilities, interface contracts, a stated composition pattern, and traceability links back to requirements is not an Architecture artifact — it is a picture. Most teams stop at the picture. This framework does not.

Retrofit posture

Architecture retrofitted from existing code is a mixed-evidence document. Decomposition (what components exist, where the boundaries actually run) is largely recoverable from the code: dependency analysis, package structure, deployment manifests, and runtime call graphs all leave fingerprints. Interfaces (what each component exposes, what it consumes) are similarly observable. The rationale — why the boundaries were drawn where they were, which alternatives were rejected, which constraints were binding — is not. The framework's rule is explicit: rationale is human-only. A retrofit Architecture records observed structure with recovery_status: reconstructed; it records interpretation of that structure (what the responsibility of a component should be, not just what its code does) with verified where a human has confirmed it; and it marks anything intent-bearing that lacks a human source with recovery_status: unknown. Drawing a clean structure diagram that papers over the actual runtime mess — laundering the current architecture — is the single sharpest retrofit anti-pattern and the skill layer refuses to emit it.

Why the framework treats this layer as non-negotiable

Architecture is where the majority of long-lived cost is committed. A bad decomposition or a vague interface will outlast the team that wrote it; a wrong composition pattern will resurface as scaling pain, incident patterns, and rewrite pressure for years. Testing catches nothing at this layer: integration tests verify that the composition you chose works, not that it was the right choice. The Quality Bar at this layer is the last opportunity for a human to push back before the decomposition hardens into twenty leaf Detailed Designs and hundreds of unit tests that all assume the shape. The Spec Ambiguity Test applies with extra force: a diagram plus four paragraphs of prose is not enough — the artifact must specify, at interface granularity, what each child does and how the children coordinate, to the point that a junior engineer or mid-tier AI could produce defensible Detailed Designs for each leaf without asking clarifying questions.

2. Best Practices

Nine disciplines. Decomposition discipline first, because every subsequent discipline rests on correct boundaries. Interface contracts next, because that is where the boundaries become committable. Then the composition stack: protocol and sync/async choice, the composition patterns catalog, data and persistence architecture, resilience, and cross-cutting concerns (observability and security). Deployment intent at root scope. Finally evolution — fitness functions, strangler patterns, testability — because Architecture that cannot change is a liability long before it is a legacy.

Frontmatter and shape

An Architecture artifact is a single Markdown file per non-leaf scope at /specs/{scope}/architecture.md (root-scope lives at /specs/architecture.md). YAML front-matter identifies the artifact; the body holds prose sections, Mermaid diagrams, and embedded YAML blocks for per-child and per-interface entries. Canonical front-matter:

---
id: ARCH-checkout
scope: checkout
parent_scope: null          # root
derived_from: [REQ-001, REQ-002, REQ-017, ...]
governing_adrs: [ADR-012, ADR-019]
recovery_status: reconstructed      # omitted on greenfield
---

The mandatory section structure of the body is specified in TARGET_ARCHITECTURE §5.3: Metadata, Overview (2–3 paragraphs), Structure Diagram (Mermaid), Decomposition (one entry per child), Interfaces (one entry per cross-child or external interface), Composition (runtime pattern, wiring, key sequence diagrams, deployment intent at root). All six sections are mandatory; a Composition section that is empty or reduced to a sentence is a Quality Bar failure regardless of the quality of the other sections.

2.1   Decomposition discipline

Decomposition is the primary architectural decision at any non-leaf scope. A wrong decomposition cannot be rescued by clever interfaces, resilient composition, or strong tests — it leaks entropy into every downstream artifact. Five principles, in order of precedence.

Information hiding (Parnas). Each module's boundary hides a design decision likely to change. The published test is not "what does this module do?" but "if requirement X changes, how many modules have to change with it?" A good decomposition answers one. Canonical example: Parnas's KWIC system contrasted two decompositions that looked similar on paper — one by processing step, one by hidden decision (line storage format, character encoding, sort strategy). Only the second survived a change in any of those hidden decisions without cascading. The mental test: list the next three changes the business is likely to demand; check that each change lands in one module, not a diagonal across the whole tree.

Cohesion and coupling (Constantine & Myers, Stevens). Inside a module, elements should belong together because they contribute to one well-defined responsibility (functional cohesion, the strongest form). Between modules, the interaction should carry the minimum information needed (data coupling over structure-passing over shared state over control). The useful form for an architect is: write one sentence stating what the module is responsible for. If the sentence needs and, or an enumerated list, the module is doing two jobs and will change for two reasons. Split it, or accept that one box in the diagram hides two that need separate interfaces.

Bounded contexts (Evans, DDD). A bounded context is a region of the system within which a single model of the domain applies and every term carries one meaning. The boundary is linguistic: when the same word (order, customer, shipment) starts to mean different things in two regions of the codebase, a context boundary is forming whether you named it or not. Strategic DDD gives you the vocabulary to name it on purpose. At Architecture time, look for where the ubiquitous language fractures: those are the lines of the diagram. Drawing a child at a boundary that cuts through the language will cost you interface breakage for the lifetime of the system.

Context-mapping patterns. When children are themselves bounded contexts that have to talk, DDD names the relationships. Three are worth knowing by name at an architect's level:

  • Anticorruption Layer (ACL). One context's model would contaminate the other; a translation layer inside one of the two quarantines the other's vocabulary. Used heavily when integrating with a legacy system or a third-party API whose model you cannot fix and do not want to propagate.
  • Partnership. Two contexts evolve in lock-step with mutual dependency acknowledged; interfaces change together; release coordination is explicit. Reserve for genuinely coupled teams; defaulting to Partnership because it is comfortable is how a system acquires six teams that cannot ship independently.
  • Published Language. A stable, versioned vocabulary (a schema, an event format, a protocol) that third parties can depend on; changes follow a deprecation policy. Used at external-facing boundaries (public APIs, event buses consumed by independent teams, inter-service contracts in a large organisation).

Evans catalogues nine; these three cover the majority of real decisions. If the right pattern is not one of these, name the pattern explicitly and cite its source.

Conway's Law, inverted. Conway observed that systems replicate the communication structure of the organisations that build them. Skelton and Pais's Team Topologies (2019) made the operational flip: choose the contexts you want teams to own — stream-aligned teams on value streams, platform teams on internal capabilities, complicated-subsystem teams on narrow specialisation, enabling teams on short-term uplift — then staff the teams to match. At Architecture time, the question is symmetric. If your decomposition forces six teams to coordinate on every change, the decomposition is wrong. If it maps one child to one team with a clear cognitive load, the decomposition has a chance of staying honest under pressure. Architecture that ignores team topology designs its own drift.

Depth heuristics. Three rules of thumb for when to stop subdividing:

  • Depth test. If children interact in ways the parent's Architecture cannot explain, go deeper — the parent is under-specifying; the actual design lives a level down.
  • Cognitive-load test. A scope should fit in one Requirements / Architecture pair that a reviewer can hold in their head. If a reviewer has to page in and out to reason about the box, the box is too large or too small (usually too large; a diagram with 14 children is a decomposition that has not decomposed).
  • Change-blast test. If subdividing further would not let parts change independently, you have gone too deep — you have split something that has no seam. Undo.

The three tests pull against each other and that is the point. A good decomposition sits where none of them fires.

2.2   Interface design and contracts

Interfaces are the committable surface of an Architecture. Components come and go; interfaces, once published, are load-bearing for everything downstream. The Best Practice rule: specify interfaces to the level that a second team could build against them without talking to the first.

Syntax vs semantics. The syntactic interface — signature, types, error enum — is what a compiler or schema validator can check. The semantic interface — preconditions, postconditions, invariants, failure modes, quality-of-service — is what a caller actually needs to code against. Most interface documentation ships only the syntactic half; most integration failures come from the missing semantic half. An Architecture interface entry specifies both, or the interface is not specified.

Design by Contract (Meyer, Hoare). Every externally callable operation has three contract clauses: preconditions the caller must satisfy (what is true on the way in), postconditions the supplier guarantees (what is true on the way out, including error returns), and invariants that hold across the operation (state properties the operation preserves). Hoare's foundational axiom — {P} S {Q}, "if P holds and S executes, then Q holds" — is the formal shape. At Architecture level the clauses are prose or structured YAML; at Detailed Design level they become assertable predicates; at test level they become test cases. The chain breaks if any layer skips them. The value is fault attribution: when a call fails a postcondition, the supplier is at fault; when it fails a precondition, the caller is at fault; anyone can tell which from the interface doc.

SEI nine-part interface template. The Carnegie Mellon SEI's Views and Beyond gives a complete interface document structure, of which we use the essentials per Architecture interface: identity (name, version), resources provided (operations with syntax and semantics), data types (structures passed across), exceptions / errors (raised when, propagated how), quality attributes (performance, throughput, availability budget carried by this interface), element requirements (constraints on the caller or supplier beyond the operations themselves), rationale (why this interface, why not a different shape), usage guide (how to call correctly). Specify nine when you can; at minimum, an Architecture interface entry carries the first five. The last four are where drift usually hides.

Interface Segregation (ISP, Martin). No component should depend on an interface method it does not use. Fat interfaces couple consumers to changes they do not care about. When a Decomposition entry allocates three distinct responsibilities, each responsibility gets its own narrow interface — not a single god-interface whose every change forces a re-review of every consumer. Segregated interfaces also survive teardown: retiring one responsibility does not force a breaking change on the other two.

Versioning and deprecation. Any externally visible interface needs a versioning scheme (semantic versioning, MAJOR.MINOR.PATCH, is the defensible default) and a deprecation policy. The Architecture artifact records the scheme and the policy; it does not wait for a breaking-change emergency to invent them. Discipline: a new interface is born with version 1.0.0, an additive change bumps MINOR, a breaking change bumps MAJOR and is preceded by a deprecation window with both versions live. A Published Language interface (see §2.1) needs an explicit deprecation window (weeks, months, quarters) documented in the interface entry, not assumed.

What goes in the Architecture entry vs what goes downstream. Architecture specifies the contract — shape, semantics, errors, quality-of-service, rationale. It does not specify the implementation: the serialization format chosen, the specific framework's routing layer, the internal caching strategy. Those belong in the relevant leaf Detailed Design. If the choice has downstream consequences that affect how callers must behave (e.g., "all requests are idempotent" is a contract; "implemented using a Redis deduplication store" is not), it is a contract concern and belongs in the interface entry.

2.3   Protocol and sync/async choice

Before you pick a composition pattern you pick a shape of communication. Four dominant families, each with distinct trade-offs along coupling, evolution, discoverability, and streaming.

  • REST / HTTP+JSON. Request-response, stateless, schema-last. Strengths: universal tooling, trivial to curl, cache-friendly, evolves by adding fields. Weaknesses: weak typing at the boundary, no native streaming contract, polling or long-polling for event-like behaviour. Default choice for external APIs consumed by heterogeneous clients.
  • gRPC / protobuf. Request-response and streaming, strongly typed via IDL, code-generated clients. Strengths: contract-first, low-latency binary, bidirectional streaming. Weaknesses: binary wire format (harder to debug), browser support requires a proxy, breaking changes are easier to introduce because generated code masks them. Default for internal high-throughput service-to-service calls.
  • GraphQL. Client-driven query shape over a strongly typed schema. Strengths: one endpoint for many read-shape needs, avoids chatty REST graphs, strong introspection. Weaknesses: non-trivial caching, authorisation granularity, and rate-limiting (per-field is the unit of concern); N+1 resolver problem is a near-universal misstep. Default for front-ends with many varying read patterns over a rich domain model.
  • Events / messaging. Asynchronous publish-subscribe or queue-based. Strengths: decoupled temporal contract, natural fan-out, back-pressure-friendly. Weaknesses: eventual consistency, schema evolution across asynchronous consumers, debugging distributed event flows. Default for integrations where the producer should not know or care who consumes.

The choice is the first composition decision. A REST contract between two components is a different architecture than the same two components connected by an event bus: the REST version couples the producer's availability to the consumer's; the event version buffers, but trades immediate consistency for eventual. The Architecture entry names the protocol and the rationale.

Sync vs async as a meta-decision. Cross-cutting every family above is the synchronous-vs-asynchronous question. Three axes decide it:

  • Coupling. Synchronous calls couple availability (if the callee is down, the caller fails or waits); asynchronous calls do not. If you need the producer to keep working when the consumer is down, you need async.
  • Failure mode. Synchronous surfaces failure at the call site — easy to handle, easy to see. Asynchronous pushes failure into retry logic, dead-letter queues, and compensation — harder to debug, harder to reason about, and a poor fit for "user is waiting on the outcome" workflows.
  • Back-pressure. Synchronous naturally applies back-pressure (a slow callee slows every caller). Asynchronous needs an explicit back-pressure mechanism (bounded queues, consumer acknowledgements, reactive streams). Systems that fail to design back-pressure into async flows fail by cascade: the queue fills, the producer OOMs, the consumer is unchanged — and nobody saw it coming.

The Best Practice is not "prefer async". It is: name the axis that dominates this interface, make the choice, record the rationale in the interface entry or a governing ADR. Defaulting to sync because it is easier is a legitimate answer when the coupling trade-off does not bite; defaulting to async because it is trendy is how you acquire a distributed system with synchronous semantics and asynchronous blast radius.

2.4   Composition patterns catalog

The Composition section of an Architecture artifact states, explicitly, what runtime pattern the children are wired together in. "We will figure it out at build time" is not an answer. A short catalog follows, with enough detail to name the pattern, state when it fits, and what it demands from the wiring.

Request-response (client-server, RPC, HTTP). One component calls another and waits for a result. Fits: user-facing workflows where the user blocks on the outcome; read paths; transactional writes where the outcome must be confirmed before the caller proceeds. Implications for wiring: timeout discipline at every call; idempotency where the network can retry; back-pressure via connection limits and queue caps; circuit breakers to avoid caller exhaustion. Anti-pattern: request-response chains longer than three hops turn every downstream latency spike into a caller outage.

Event-driven (publish-subscribe). A producer emits events on a topic; any number of consumers subscribe and act asynchronously. Fits: fan-out notifications, audit logging, downstream integrations that must not block the producer, workflows where the consumer set changes independently. Implications: event schema is a published contract (versioned, deprecation-windowed); message bus topology is an architectural artifact (topic per aggregate, partition key choice, retention policy); consumer idempotency is non-negotiable because at-least-once is the baseline. The fan-out is the strength; debugging the fan-out is the weakness — distributed tracing across async edges is not optional.

Event sourcing. State is stored as a stream of domain events; current state is a fold over the stream. Fits: domains where the why of every change is as important as the current value (finance, audit-heavy regulation, systems with complex undo); domains where multiple read models need to be derived from the same writes. Implications: event schema evolution becomes the hard problem (immutable history plus evolving code); projections are first-class architectural elements; snapshotting is a performance concern the Architecture must address. Expensive choice; wrong for most systems; extraordinary when right.

Saga. A long-running, multi-step workflow coordinated across services, with compensating actions for partial failure. Two variants: orchestration (a coordinator service owns the workflow) and choreography (each service reacts to events and emits the next). Fits: cross-service business transactions where distributed atomic commit is not available; human-approval-gated workflows; retry-and-compensate flows. Orchestration is easier to reason about and harder to scale; choreography is the reverse. Implications: compensation is a design axis (not every step has a clean inverse); timeout and retry budgets are explicit; visibility (where is this saga stuck?) is a first-class Architecture concern.

Pipeline / data flow. A sequence of stages, each consuming the previous stage's output. Fits: batch processing, streaming analytics, ETL, build systems, media processing. Implications: stage interfaces are the pressure points (what exactly flows between stage 3 and stage 4?); parallelism is per-stage with bounded buffers between; back-pressure is explicit. Under-served by most patterns catalogues because it does not look like request-response; deserves first-class treatment whenever data transformation is a load-bearing part of the system.

Layered (n-tier). Stacked layers where each layer depends only on the layer beneath (presentation → business logic → data access). Fits: simple CRUD applications where the same entity flows through every layer. Anti-pattern warning: in non-trivial systems, layers become horizontal slices through every feature rather than vertical boundaries around concepts, producing a distributed monolith (see §3). Teams change feature-by-feature, not layer-by-layer; if your decomposition does not match how change arrives, it will punish every change.

Hexagonal (ports and adapters, Cockburn). An inner domain hexagon with driving ports (primary, invoked by users / jobs / APIs) and driven ports (secondary, depended on by the domain for persistence / messaging / external services). Adapters translate between ports and concrete technologies. Fits: any domain complex enough to have real business logic that should not know the database. Implications: the domain depends on abstractions, not concretes; adapters are per-technology and swap without touching the domain; testability is a structural property, not a shoehorn. Hexagonal is arguably the single most cost-effective pattern for long-lived domain-heavy services. The Architecture artifact names the ports (one entry per port, primary and secondary).

Clean architecture (Martin). A more prescriptive elaboration of hexagonal, organised as concentric rings (entities → use cases → interface adapters → frameworks and drivers) with a strict dependency rule: source code dependencies point only inward. Fits: teams that benefit from opinionated structure, especially when the team is large or the domain is regulated. Implications identical to hexagonal plus the inward-dependency discipline.

Microservices vs modular monolith. Two deployment topologies with similar decomposition. A modular monolith is a single deployable with strict internal module boundaries, in-process calls, and a shared database (or per-module schemas in a shared database). Microservices are the same decomposition spread across independently deployable services with network calls and independent persistence. The choice is not "which is better"; it is when does the independent-deployment benefit outweigh the distributed-system cost? A useful heuristic: if two modules do not need to scale independently, fail independently, or be owned by independent teams, they do not need to be independent services. Defaulting to microservices for a small team with one deployment cadence is how projects acquire distributed monoliths.

Serverless / FaaS. Composition as short-lived functions triggered by events, HTTP requests, or schedules. Fits: spiky loads, glue code, event processors, infrastructure automation. Implications: cold-start latency is a first-class NFR; stateless-between-invocations is the contract; the wiring lives largely in the cloud provider's configuration (event sources, permissions, triggers); observability requires provider-specific instrumentation. Does not fit: long-running stateful workflows, high steady-state traffic (cost and cold-start both bite), deep end-to-end latency budgets.

Wiring: DI, middleware, and message-bus topology. Whatever pattern the Composition section names, three wiring concerns are always explicit in the Architecture artifact:

  • Dependency injection strategy. How components acquire their dependencies: constructor injection, a DI container, a composition root. Name the strategy so a leaf DD does not reinvent it per component.
  • Middleware stack. The ordered chain that every request (or event) passes through — authentication, authorisation, logging, tracing, rate limiting, request validation. Order matters, and the Architecture states it: tracing outermost, auth before authz, rate limiting before handler, logging on exit is not something you discover by running production for six months.
  • Message-bus topology. Event-driven and saga patterns rely on a bus; topic naming, partition key choice, retention policy, dead-letter routing, and consumer group layout are architectural decisions, not infrastructure details. The Architecture artifact names them.

2.5   Data and persistence architecture

Data topology is where most post-launch regret accumulates. An Architecture that decomposes beautifully at the component level and punts persistence to "we'll use a Postgres" is a ticking liability; by the time the topology becomes a problem, it is expensive to change.

Database-per-service vs shared database. The orthodox microservices answer is database-per-service (each service owns its schema; no other service reads its tables; cross-service joins become cross-service calls). The benefit is independent schema evolution and blast-radius isolation. The cost is data consistency across services is now an application concern. The shared-database approach (one database, schemas per module) is cheaper to operate and trivial for joins, but couples deployment: a schema migration affects every service. The Architecture entry names the choice and the rationale; a modular monolith can legitimately share a database; a microservices system that shares a database has imported microservices' operational cost without its independence benefit.

Consistency model. Strong consistency (every read sees every prior write, globally) costs availability or latency under partitions — the CAP trilemma (Brewer) and its PACELC refinement (Abadi) state the trade-off formally; Kleppmann's Designing Data-Intensive Applications is the standard reference. Most systems accept some eventual consistency somewhere; the Architecture decision is where. The useful question is not "strong or eventual?" but "what is the maximum staleness window the user can tolerate for this read path, and what is the reconciliation story when two writes collide?" Eventual consistency without a reconciliation story is a bug factory.

Read replicas and CQRS. Read traffic commonly exceeds write by 10× or more. A read replica (same schema, asynchronous copy) absorbs read load with a bounded staleness budget; CQRS (Command-Query Responsibility Segregation) takes the split further with a separate read model optimised for query shape (denormalised, indexed for the read patterns, fed by events). CQRS is powerful when read and write shapes diverge sharply; overkill when they do not.

Event sourcing as persistence. See §2.4. When event sourcing is the composition pattern, persistence is the event log plus projections; the Architecture entry for the log (retention, partitioning, schema evolution) is the central data decision.

Multi-tenancy. Four isolation tiers, increasing in cost and isolation: row-level (tenant_id column on every table), schema-level (schema per tenant in a shared database), database-level (database per tenant on shared infrastructure), deployment-level (separate deployment per tenant). Architecture entry chooses a tier per data category; a high-isolation tier for audit data can coexist with row-level for most operational tables. Defaulting to row-level for everything is cheap until the first compliance audit; defaulting to deployment-level for everything is expensive forever.

2.6   Resilience and fault tolerance

Resilience is a property of composition, not of components. A component that behaves correctly in isolation can bring down a system when composed with others under load or partial failure. Four patterns are baseline architectural vocabulary; the Architecture artifact names which apply where and why.

Bulkhead. Partition a shared resource so that failure in one partition cannot starve others. Classic form: per-dependency thread and connection pools. If the pricing service is slow, only the pool allocated to pricing fills; the inventory service's pool is untouched, and inventory calls continue. Without bulkheads, a single slow dependency drains the global pool and cascades. The Architecture entry names the partitions.

Circuit breaker. Three-state machine: closed (normal, count errors), open (reject requests fast when error rate exceeds a threshold, letting the downstream recover), half-open (probe with a small number of requests; if they succeed, close). The value is not "retry less" but "stop the feedback loop where a failing dependency exhausts its caller". Circuit breakers are modal behaviour (different responses in different states), which complicates testing — the AWS Builders' Library explicitly recommends caution. Use at cross-service boundaries; do not use inside a tight internal loop where the operation cost is low enough that failing fast saves nothing.

Retry with backoff and jitter. Exponential backoff (wait 2^n × base) spreads retries away from the failure moment; jitter (a random component of the wait) prevents synchronised retry storms. Without jitter, every caller retries at the same second and the downstream sees a second wave identical in size to the first. The classic cautionary number: a five-layer call chain where each layer retries three times on failure sees 3^5 = 243× normal load on the innermost dependency. Retry is only safe with idempotent operations; for side-effecting operations, idempotency is a deliberate design (idempotency keys, conditional writes, deduplication stores).

Graceful degradation vs fallback. Degradation shrinks scope on the same path: the e-commerce landing page loads without the recommendation carousel; the checkout proceeds without the loyalty-points lookup. Fallback is a parallel substitute path (try primary, then secondary). Degradation is preferable — fewer code paths, smaller blast radius, fewer failure modes to test. Fallback looks attractive until the fallback path has its own failure mode that differs from the primary's, and your incident report has three distinct failure categories. The Architecture decides, per non-essential dependency, whether degradation is designed in.

Failure domain design. A failure domain is a set of components that fail together. Two components sharing a process, a host, a database, a region, or a cloud provider are in the same failure domain at that level. The Architecture artifact is explicit: which domains matter (process / host / availability zone / region / provider), which components share which domains, and where the boundaries are drawn on purpose.

Common-cause failure. Two supposedly independent redundant components can fail together if they share a cause: the same library with the same bug, the same specification interpreted the same wrong way, the same operator running the same bad command on both. Redundancy without independence is theatre. The Architecture entry for a redundancy mechanism names the independence property: "primary and secondary regions are in different cloud providers" is a real independence claim; "we run two instances of the same binary behind a load balancer" is not (shared binary, shared library versions, shared deploy pipeline — every common-cause path).

2.7   Observability and security as architectural concerns

Cross-cutting concerns that teams try to bolt on post-hoc are the ones most punished for the effort. The Architecture artifact specifies where they enter.

Observability — telemetry emergence points. Three axes: structured logs, metrics, distributed traces. The architectural decisions are where each enters the system and who consumes it. For logs: what fields are carried on every record (request ID, tenant, user, correlation ID, span context), which components log at which level, what is the retention and the search tool. For metrics: which metrics are SLO-bearing (user-visible promises), which are diagnostic (for on-call), what dimensions they carry. For traces: every cross-process boundary propagates a trace context; sampling policy is explicit (100% of errors and 1% of successes is a common baseline). OpenTelemetry is the current de facto standard for instrumentation; the Architecture entry names it or a reasoned alternative. Observability bolted on at incident time is always slower and always misses the context that would have made the incident short.

Security — trust zones. A trust zone is a region of the system that shares a threat model. The internet is one zone; the authenticated user session is another; the internal network between services is another; the service's own process is another. Architecture draws the zone boundaries explicitly and specifies what crosses each boundary and with what guarantees. The STRIDE mnemonic (Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege) is the standard checklist for each crossing; the Architecture artifact does not teach STRIDE but names the threats considered and the controls.

Secrets flow. Credentials, tokens, keys, and signed URLs cross boundaries and get logged, cached, and inherited. The Architecture entry for a secret names: where it originates (secret manager, KMS, external identity provider), which components hold it in memory (and for how long), which components see it only as a bearer token, and where it must never appear (logs, metrics, traces, error messages). The laws of leakage are not subtle; the shape of the leakage is almost always in the Architecture's omissions.

Authentication and authorisation as interface concerns. Every externally callable interface entry states authentication requirements (what identity is asserted, how it is verified, what claims are carried) and authorisation requirements (what permission is required, evaluated at which layer — the gateway, a middleware, the handler). Mixing these up is how you get systems that authenticate correctly and authorise catastrophically. The Architecture is the right layer to state them because it is where cross-component guarantees become visible; putting them in DD alone duplicates the decision per component.

2.8   Deployment intent at root scope

The root-scope Architecture's Composition section carries one concern no branch does: deployment intent. The framework's stance, post-pivot: Architecture Composition is the authoritative spec for deployment; Infrastructure-as-Code (terraform, Kubernetes manifests, compose files) is the implementation of that intent; there is no separate prose Detailed Design layer above the IaC.

Environments. Every environment the system runs in (dev, staging, production, production-EU, customer-tenant-X) is named; the differences across environments are enumerated (scale, data set, external integrations, observability stack); where environment differences matter behaviourally, they are explicitly tied to requirements or ADRs. Environments that exist but are not enumerated are environments that drift.

Orchestration target. The target platform for runtime composition: Kubernetes (which distribution, which ingress, which service mesh if any), a serverless platform (which provider, which FaaS and managed services), container orchestration (ECS, Nomad, Swarm), a VM-based deployment (systemd units, cloud-init), a bare-metal target (for embedded: firmware image + hardware bring-up sequence). The choice determines what "wiring" concretely means and bounds the operational primitives the rest of the architecture can reach for.

Runtime-unit boundaries. A runtime unit is a unit of deployment, scaling, and failure. Each runtime unit has its own lifecycle (start, stop, rolling restart), its own observability surface (metrics, logs, traces emerge from this unit), its own resource budget (CPU, memory, concurrency), and its own failure story (what happens when this unit dies). The Architecture artifact at root maps which components from the Decomposition live in which runtime units; co-location is a choice (shared fate is the cost) and so is separation (network boundary and operational overhead are the cost).

IaC as implementation. Terraform, Kubernetes manifests, Helm charts, and compose files are declarative: they describe the runtime and the platform applies them. Architecture Composition specifies the intent (what environments exist, what the orchestration target is, where the unit boundaries fall); IaC artifacts implement that intent concretely. The Architecture does not re-specify what the IaC already states; it specifies what the IaC cannot capture (rationale, tying choices to requirements and ADRs, naming the unit-boundary logic behind the manifests).

12-factor as cloud-native reference. Heroku's Twelve-Factor App (Wiggins, 2011) catalogues operational norms — config in environment, processes disposable, logs as streams, dev/prod parity — that any cloud-native composition either follows or explicitly departs from. Root-scope Architecture states the stance factor by factor where the factor is load-bearing for the system. 12-factor is not craft doctrine; it is a useful reference that makes departures visible.

Cost as a constraint. Serverless vs container vs VM, managed service vs self-hosted, hot replica vs cold backup, and multi-region active-active vs active-passive are all cost decisions as much as architectural ones. Cost sits under Constraints in Requirements; at Architecture it resurfaces as a binding input to orchestration and runtime-unit choices. Naming the cost model in the Architecture (monthly infra cost envelope, cost-per-request target, cost-of-a-9 on availability) prevents the common failure where a well-engineered architecture is killed by the bill.

Architecture-as-code tooling. Mermaid (used in this framework), Structurizr (Brown's C4 workbench), PlantUML, D2, and draw.io are mechanisms for keeping diagrams close to the code they describe. The Architecture artifact does not mandate a tool; it does mandate that the Structure Diagram and Composition sequence diagrams live in the artifact, not in an image pasted from whiteboarding. Mermaid is the framework's default because it is text, version-controllable, and AI-readable; Structurizr is a legitimate alternative when the team is already committed to it.

2.9   Architecture evolution and fitness functions

Architecture that cannot change becomes a liability before it becomes a legacy. The Best Practice is to design for evolution deliberately: specify the properties the Architecture must preserve, automate their verification, and plan the migration path for changes that do not preserve them.

Architecture as hypothesis. Every non-trivial decomposition is a bet on how the system will change over the next one to three years. The bet is wrong some of the time; the Architecture is a hypothesis, not a statute. Naming the hypothesis ("we expect the checkout service to remain a single deployable while inventory and pricing scale independently") converts an implicit assumption into a testable one, and gives future teams a first-class place to write down when the hypothesis broke.

Evolutionary architecture (Ford, Parsons, Kua, 2017). Accept that Architecture evolves; build verification into the development loop to catch deviation early. The load-bearing device is the fitness function: an automated check that an architectural property still holds. Ford et al. classify them along two axes — atomic (one property) vs holistic (multiple together), triggered (runs on events, e.g., CI, deploy) vs continuous (runs in production, observes live behaviour).

Fitness functions in CI. Four concrete categories worth most non-trivial Architectures:

  • Dependency direction. A test that fails if a forbidden package imports another (e.g., domain imports infrastructure). Tools: ArchUnit (Java), dependency-cruiser (JS/TS), custom AST scripts.
  • Latency budget. A test (load or synthetic) that fails if a critical interface exceeds its budgeted p95. Run on a recurring schedule, not just release; regressions creep in from underlying platform changes.
  • Module size and coupling. A test that fails when a module exceeds a line or complexity threshold, or when its coupling to peers exceeds a number. Crude but catches the slow slide toward god components.
  • Security posture. A test that fails when a known-bad dependency pattern appears (SQL string concatenation, plaintext secret in config) or when a boundary loses a required control (unauthenticated endpoint on a service that should authenticate).

Fitness functions are not a substitute for integration tests; they cover a different shape of drift — the slow erosion of architectural properties that no functional test will notice. The Architecture artifact names which properties are load-bearing enough to warrant a fitness function, and references the check in the governing Architecture entry.

Strangler fig (Fowler, 2004). A migration pattern for replacing a legacy system without a big-bang rewrite. Stand a new system alongside the old; route traffic to the new for capabilities it covers; expand the new system's coverage; retire the old when the vine has covered the tree. Relevant here because the Architecture artifact for a system being retrofitted often describes the target state, with strangler-fig migration between current and target as the change plan. The Architecture entry for a component being strangled names the routing mechanism (reverse proxy, feature flag, conditional dispatch), the acceptance criteria for retiring the legacy path, and the rollback point.

Testability as an architectural quality. A system's testability — how easily integration and unit tests can be written against its components — is a property of its Architecture, not of its test suite. Hexagonal's port separation, dependency injection, a pure-core / impure-shell split, and explicit test seams all earn their keep here. An Architecture that makes every component depend on a singleton database client is an Architecture that mandates heavyweight integration tests for every unit test; one that specifies driven-port abstractions lets the same tests run in seconds against fakes. The Quality Bar in §5 has a card for this; the Best Practice rule is to name, in each Decomposition entry, the test-seam shape the component provides.

3. Anti-Patterns

Six universal Architecture failure modes, plus four specific to AI-augmented authoring and retrofit. The universal six are named in the literature across decades; the AI-era four are newer and their remedies differ because the failure is in the generation process, not the author's discipline.

Universal (six)

  1. Big ball of mud (Foote & Yoder, 1997). No discernible decomposition; every component depends on every other; changes ripple unpredictably. Symptoms: a diagram nobody draws because it would fit on a wall; onboarding times measured in months; "where does X happen?" is answered by a guided tour. Tell: the Architecture artifact exists but the Structure Diagram is either missing, absent of arrows, or a spider's web of arrows with no discernible flow.
  2. Distributed monolith. Microservices decomposition without microservices independence — services cannot be deployed, scaled, or failed independently because they share a database, a schema, a deploy cadence, or a synchronous request chain. The operational cost of a distributed system without the benefits. Tell: every release requires all services to ship together; any one service going down takes down the feature; integration tests must spin up the whole constellation.
  3. God component. One component in the Decomposition owns everything central to the domain; peripheral components are thin wrappers around its API. The single-responsibility test fails loudly (the god component's one-sentence responsibility needs three ands). Tell: the god component's leaf DDs total more lines than the rest of the scope combined; most changes require touching it.
  4. Premature decomposition. Boundaries drawn where the domain has not yet shown seams. Ten services for a system three people can fit in a room for. Cost: all the distributed-system tax (network, schema, deployment coordination) for none of the independent-evolution benefit (there is only one team). Tell: services that always deploy together; services that cannot be tested independently; "we need to call across three services for a login".
  5. Stale architecture documentation. The diagram is two years old; the running system has evolved; newcomers trust the diagram and discover the divergence the hard way. Universal, mostly lived as operational pain rather than named as an architectural failure. Tell: the Architecture artifact was last edited in the initial release; the codebase has had two major refactors since; integration tests pass but describe a different topology than the diagram.
  6. Cyclic dependencies. Components depend on each other in a cycle (A imports B imports C imports A). Consequences: no independent build order, no independent release, no module-level replacement. Tell: the dependency-direction fitness function fails; changes in one component break compilation in a "downstream" component that is actually upstream in the cycle.

AI-era and retrofit (four)

  1. Laundering the current architecture. A retrofit-generated Architecture that describes the system's current structure as if every boundary were intentional and every interface contracted. The diagram looks clean; the rationale fields cite no rejected alternatives; every decision appears inevitable. The document is an apologetic, not an Architecture. Tell: zero recovery_status: unknown values; no acknowledgement that any of the observed boundaries were accidental; no gap report accompanying the artifact.
  2. Fabricated decomposition rationale. An AI agent asked to retrofit produces plausible-sounding rationale for every child's existence, phrased as if recalled from the original design. The rationale reads competent and is entirely synthesised from training priors. Tell: rationales that say "chosen because it matches the single-responsibility principle" rather than "the original team chose this boundary because [specific constraint / date / incident]". Fabricated rationale is confident precisely where a real one would be uncertain; the remedy is enforced recovery_status: unknown on rationale fields during retrofit, not better prompting.
  3. Ad-hoc composition. The Composition section is a sentence or a single diagram with no stated pattern, no wiring strategy, no runtime-unit boundaries, and no deployment intent at root. The system is "a bunch of services that talk to each other". Symptom: every incident post-mortem names a different emergent behaviour; nobody can predict cascade paths; capacity planning is guesswork. Tell: the Composition section is shorter than a single Decomposition entry; no named runtime pattern; no sequence diagrams.
  4. Missing composition spec. The Decomposition and Interfaces are thorough; Composition is empty or reduced to "see deployment config". Teams build the system, discover the composition at integration time, and hardcode whichever quirk of the orchestration they first got to work. The document exists but abdicates the most load-bearing section. This is the most common failure mode in Architectures produced by engineers who are fluent in component design but treat runtime as "someone else's problem"; mandate a non-trivial Composition section at the Quality Bar and reject the artifact when it is absent.

4.3   Interface contract — Design-by-Contract form

interfaces:
  - name: ICartFinaliseCheckout
    from: api-gateway
    to: cart
    protocol: HTTP/1.1 + JSON (REST)
    contract:
      operation: "POST /checkout/{cart_id}/finalise"
      preconditions:
        - "Cart exists and is in state ACTIVE or PRICED."
        - "Authenticated session's subject owns the cart."
        - "Idempotency-Key header present and globally unique per 24h window."
      postconditions:
        on_success:
          - "Order is committed; an OrderPlaced event is published within 5s."
          - "Cart transitions to state COMPLETED; no further operations accepted."
          - "Response carries the order id and the captured payment amount."
        on_precondition_failure:
          - "HTTP 409 with typed error (cart-state, auth, idempotency-collision)."
          - "No state mutation."
        on_downstream_failure:
          - "HTTP 502 or 504 with typed error (inventory-unavailable, pricing-stale, payment-declined, payment-timeout)."
          - "Cart state unchanged; any in-flight reservations released within 30s."
      invariants:
        - "Double submission with the same idempotency key returns the first response body and does not re-commit."
        - "If HTTP 200 is returned, the OrderPlaced event is guaranteed to be published."
      errors:
        - { code: cart-state, http: 409, meaning: "Cart is not in a finalisable state." }
        - { code: auth, http: 403, meaning: "Session subject does not own the cart." }
        - { code: idempotency-collision, http: 409, meaning: "Idempotency key already seen with different payload." }
        - { code: inventory-unavailable, http: 502, meaning: "Reservation could not be completed." }
        - { code: pricing-stale, http: 502, meaning: "Signed quote has expired; client must re-price." }
        - { code: payment-declined, http: 502, meaning: "PSP refused authorisation." }
        - { code: payment-timeout, http: 504, meaning: "PSP did not respond within 8s." }
      quality_attributes:
        availability: "99.9% monthly, per platform SLO (REQ-051)."
        latency: "p95 ≤ 1200 ms under nominal load (REQ-052)."
        idempotency_window: "24 hours from first observation of the key."
      rationale: "REST over JSON chosen for the gateway boundary (see ADR-008). Idempotency key mandated by REQ-028 (at-least-once order-placement semantics over unreliable clients)."

The contract is a full Design-by-Contract specification of one interface: preconditions the caller must satisfy, typed postconditions on success / precondition-failure / downstream-failure, invariants across repeat calls, a typed error enum with HTTP mapping, and quality attributes that carry budget obligations into this boundary. A reader could write the integration test suite from this entry without looking at any other artifact.

Five observations about this sequence as a Composition element. First, the gateway's middleware stack (AuthN → AuthZ → RateLimit → Trace) is stated explicitly on the edge, not left as "the gateway does its thing". Second, the pricing quote is signed and has a validity window, making the pricing → cart interaction a published contract rather than an implicit assumption. Third, the inventory reservation is a first-class artifact with a TTL — an explicit release path on abandonment is a design requirement, not an afterthought. Four runtime units: a read-heavy compute deployment for the gateway / cart / pricing; a writer deployment for inventory / payment / order-committer; managed Postgres; managed Kafka. # What NOT to do — same requirements, five tightly-coupled services

decomposition:
  - id: cart-service
    purpose: "Cart and pricing and checkout workflow."
    responsibilities:
      - "Hold cart items."
      - "Call pricing-service for totals on every update."  ← chatty sync call on every edit
      - "Call inventory-service on every view."              ← sync read on every cart display
      - "Orchestrate checkout."
      - "Persist cart to its own database."                  ← per-service DB, same transactional scope as inventory
    allocates: [REQ-001, REQ-002, REQ-005, REQ-010, REQ-014, REQ-017]  ← six requirements in one child

  - id: pricing-service
    # ...

  - id: inventory-service
    purpose: "Stock reservation."
    responsibilities:
      - "Reserve stock on cart update."                      ← coupled to cart's lifecycle; one of the services can't ship independently
      - "Return current stock on every read."
      - "Persist to its own database."                       ← distributed transaction with cart on checkout

The failure modes of this decomposition are visible before anything runs. cart-service hosts six requirements with three distinct change drivers (cart UX, pricing rules, inventory behaviour) — one god component masquerading as a coordinating service. Every cart update triggers synchronous calls to pricing and inventory; the three services cannot ship independent releases because their call contracts change together; the per-service databases require a distributed transaction on checkout (or, more commonly, an incident catalogue of inconsistent states). The diagram would look like independent microservices; the operational reality is a monolith with HTTP instead of function calls. The Quality Bar in §5 catches this via the one-sentence responsibility test (the cart-service sentence needs two ands) and the cross-service transaction smell.

The laundered version reads smoother; it is generic to the point of interchangeability and its rationale cites principles rather than history. The honest version carries the same observed structure with every responsibility tied to a file-and-line, and the rationale field says we do not know rather than inventing a plausible answer. The framework's retrofit skill enforces the honest shape at the tool layer: the laundered output cannot pass review because the rationale field under recovery_status: reconstructed requires either a preserved decision record or a marker of unknown. "Because it follows DDD principles" is what a language model produces from training priors; it is not a retrofit finding.

5. Quality Bar

Yes/No checklist, grouped by concern. An Architecture artifact passes when every applicable item is answered Yes. Two cards are meta-or-load-bearing and carry a visual accent: Composition completeness (where silent omission has the largest blast radius) and the Spec Ambiguity Test (the meta-gate that overrides everything above it).

Decomposition and boundaries

  • Is every child's purpose stated in exactly one sentence without conjunctions?
  • Does every child's responsibility list contain at most three items, each scoped at architectural (not implementation) level?
  • Is every parent-allocated requirement named in at least one child's allocates? (No requirements orphaned.)
  • Does every child allocate at least one requirement? (No components orphaned.)
  • Has the one-sentence-responsibility test been walked per child (no hidden ands)?
  • Has the depth/cognitive-load/change-blast trio been checked, with a record of which one fired if decomposition was revised?
  • Where a bounded-context language fracture exists, is the child boundary drawn at the fracture?

Interfaces and contracts

  • Does every interface entry carry preconditions, postconditions (on success, on precondition failure, on downstream failure), invariants, a typed error enum, and quality attributes?
  • Does every externally callable interface state authentication and authorisation requirements?
  • Is every interface segregated by responsibility (no fat god-interfaces)?
  • Does every versioned interface name its versioning scheme and deprecation policy?
  • Is rationale carried on every interface (at least one line tying the choice to a requirement, ADR, or constraint)?
  • Are externally-imposed protocols cited by specification (RFC, API spec, standard identifier), not informal name?

Composition completeness — load-bearing

  • Is the runtime pattern named explicitly (request-response / event-driven / pipeline / hexagonal / …) rather than implied?
  • Does every runtime pattern choice have a rationale (inline or governing_adrs)?
  • Is the middleware stack ordered and stated (authN / authZ / rate limit / trace / …)?
  • Is the DI / composition-root strategy named?
  • Is the message-bus topology specified where applicable (topics, partition keys, retention, dead-letter, consumer groups)?
  • Does every critical flow (happy path plus one or two failure paths) have a sequence diagram?
  • At root scope: are environments, orchestration target, and runtime-unit boundaries named?
  • At root scope: does deployment intent resolve to concrete IaC artifacts (by reference) or explicitly planned ones?
  • At root scope: does every runtime-unit boundary have an integration-test target in the branch / root TestSpec?

Quality-attribute coverage

  • Does every NFR from the parent Requirements appear as either an Architecture child's allocated requirement or a cross-cutting composition commitment?
  • Are latency / throughput / availability budgets allocated to specific interfaces or components, not left at system-wide level?
  • Are consistency-model choices (strong / eventual / read-your-writes / bounded staleness) named per data path?
  • Is the cost model (envelope, cost-per-request target, cost-of-a-9 if availability is a binding concern) stated at root scope?

Resilience coverage

  • Are bulkhead partitions named for shared resources that matter (connection pools, thread pools, external-dependency pools)?
  • Are circuit breakers placed at cross-service boundaries where caller exhaustion is a real failure mode?
  • Is retry policy (backoff, jitter, max attempts) specified where retries are safe, with idempotency design where they are not?
  • Is graceful degradation designed-in (not fallback-in) for non-essential dependencies, with the essential-vs-non-essential split named?
  • Are failure domains (process / host / AZ / region / provider) named, and is every redundancy claim backed by a named independence property (not assumed)?

Security and observability at boundaries

  • Are trust zones drawn explicitly with named boundary crossings?
  • Is authn / authz stated per externally callable interface, with the evaluation layer (gateway, middleware, handler) named?
  • Is secrets flow specified (origin, in-memory-holders, where they must never appear)?
  • Are telemetry emergence points specified (what is logged, what is metric'd, what is traced; sampling policy; common context fields)?

Rationale and traceability

  • Does every Decomposition entry, interface entry, and load-bearing composition choice carry a rationale field with non-trivial content?
  • Are governing ADRs listed in front-matter and referenced at the point in the body where each decision lands?
  • Has the traceability validator been run and returned zero orphan children, zero unallocated requirements, zero unresolved references?
  • Are fitness functions named for load-bearing architectural properties (dependency direction, latency budget, module size, security posture)?

Retrofit discipline

  • Is observed structure marked recovery_status: reconstructed with evidence (file/line, commit, schema, operational log)?
  • Is human-only content (rationale, rejected alternatives, original intent) marked verified or unknown only — never reconstructed?
  • Is every unknown field paired with a follow-up owner and an action?
  • Is laundering audited — does any rationale read as generic principle invocation rather than historical recall?
  • Is the gap report populated (lost rationale, structural drift, missing ADRs, coverage gaps, ambiguous spec)?

Spec Ambiguity Test — meta-gate

  • Could a junior engineer or mid-tier AI, reading only this Architecture artifact (plus its governing ADRs and parent Requirements), produce defensible Detailed Designs for each leaf, and a TestSpec whose cases verify every interface and every composition-level invariant — without asking clarifying questions?

If the answer is No, the artifact under-specifies its allocation role. Revise before accepting. This test overrides every box above: an Architecture artifact that passes all the other checks but fails this one has not done the job Architecture exists to do.
