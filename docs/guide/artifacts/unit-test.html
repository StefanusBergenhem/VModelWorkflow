Level 3 — Verification

What is Unit Testing?

Unit testing is the practice of verifying individual software units — functions, classes, or modules — in isolation. It is the most granular level of testing, targeting the smallest pieces of behavior that can be independently exercised.

A unit test answers one question: “Does this unit do what it was designed to do?”

Effective unit testing provides:

  • Fast feedback — unit tests run in milliseconds, catching bugs immediately
  • Design pressure — code that is hard to unit test is poorly designed (tight coupling, hidden dependencies, mixed concerns)
  • Regression protection — when code changes, failing tests pinpoint exactly what broke
  • Living documentation — tests describe what the unit does, with executable examples
  • Refactoring safety net — confident refactoring because tests catch behavioral regressions

What is a “Unit”?

A unit is the smallest piece of code that can be meaningfully tested in isolation. In object-oriented code, it is typically a single class or method. In functional code, it is a single function. The key criterion: the unit has defined inputs, defined outputs, and defined behavior rules — all specified in the Detailed Design.

Unit Testing in the V-Model

Unit tests sit on the right side of the V at Level 3, directly opposite the Detailed Design on the left side. They verify that each unit behaves exactly as its design specifies.

Position in the V

The right arm of the V verifies each specification level:

Unit Test → Integration Test → SW Verification → System Test

Unit tests are the first verification activity. They take the Detailed Design as input and verify that the source code correctly implements each design element.

The Critical Principle: Tests Verify Design, Not Code

This is the most important concept in V-model testing. Tests are derived from the design specification, not from the code. You read the design, derive test cases from what the design says should happen, and then check whether the code does that.

If you derive tests by reading the code, you are testing that the code does what the code does — a tautology. You will never find a bug this way, because the tests will mirror whatever the code actually does, including its mistakes.

What “Verifies” Means

Verification at the unit level means proving that:

  • Every behavior rule in the design is correctly implemented
  • Every interface contract (inputs, outputs, types, constraints) is honored
  • Every error condition produces the specified error response
  • State transitions (if any) follow the design specification
  • Boundary conditions are handled correctly

Producing Quality Unit Tests

This section is the comprehensive guide to writing effective, industry-standard unit tests. It covers test design philosophy, derivation strategies, test structure, test doubles, common smells, coverage analysis, and AI-assisted testing. Each principle includes a concrete example.

3.1 Test Design Principles

Before writing any test, you need to understand what makes a test good. These principles are the foundation of the testing craft — they apply regardless of language, framework, or safety standard.

Test Behavior, Not Implementation

The most consistently cited principle across testing literature: verify what a unit does (its observable outputs given specific inputs), not how it does it (which internal methods it calls, what data structures it uses, or how its algorithm works).

David Bernstein articulates this as: “Writing tests against the behaviors you want to create rather than how you implement those behaviors is one of the keys to having the tests support you in refactoring code rather than breaking when you refactor code.” — Test Behaviors, Not Implementations, tobeagile.com

Bad — Tests the internal mechanism
Java@Test void calculatesDiscount() { DiscountService service = new DiscountService(mockPriceRepo); service.applyDiscount(order, 10); // Verifying internal call sequence, not outcome verify(mockPriceRepo).fetchBasePrice(order.getId()); verify(mockPriceRepo).updatePrice(order.getId(), any()); verifyNoMoreInteractions(mockPriceRepo); }
Good — Tests the observable outcome
Java@Test void ten_percent_discount_reduces_price_from_100_to_90() { Order order = anOrder().withTotal(Money.of(100)).build(); DiscountService service = new DiscountService(priceRepository); Order discounted = service.applyDiscount(order, 10); assertEquals(Money.of(90), discounted.getTotal()); }

Why this matters:

  • Behavior tests survive refactoring. Change from a list to a priority queue internally? Behavior tests still pass — the external result hasn’t changed.
  • Behavior tests only fail when behavior changes. A test that breaks when you rename a private method provides noise, not signal.
  • Implementation tests double the cost of change. You change the code and the tests, because the tests are coupled to the mechanism.

Kent Beck separates these into two distinct properties of good tests: “Respond to behavior changes” and “Ignore structure changes.” When tests satisfy both, they enable confident refactoring. — Kent Beck, “Programmer Test Principles”, Medium, 2019

V-Model Connection

In safety-critical development, this maps directly to the V-model principle: tests verify requirements, not code. DO-178C’s “low-level requirements-based testing” is behavior testing by another name. The requirement specifies what the unit should do — its contract. The test verifies that contract.

Tests as Executable Specifications

A well-written test suite is the most accurate documentation of what a system actually does. Unlike written documentation, tests are verified by the compiler and runtime on every build — they cannot silently go stale.

Kent Beck positioned TDD as “primarily a specification technique with a side effect of ensuring that your source code is thoroughly tested.” — Scott Ambler, “Introduction to Test Driven Development”, summarizing Beck’s Test-Driven Development: By Example, Addison-Wesley, 2003

Robert C. Martin goes further: “What makes a clean test? Three things: readability, readability, and readability. Readability is perhaps even more important in unit tests than it is in production code.” — Robert C. Martin, Clean Code: A Handbook of Agile Software Craftsmanship, Chapter 9: Unit Tests, Prentice Hall, 2008

The elements that make tests communicate effectively:

  1. Test name describes the scenario and expected outcome. A test named should_reject_withdrawal_when_balance_insufficient communicates the requirement being verified.
  2. Arrange-Act-Assert structure is visible. The reader immediately sees setup, action, and verification.
  3. Noise is eliminated. Irrelevant setup, magic numbers, and infrastructure concerns are extracted to helpers and builders.
  4. Domain language is used. Tests use the same vocabulary as the requirements they verify.

What Makes a Good Assertion

Assertions are the core of a test — they define what “correct” means. A test without a meaningful assertion is not a test.

Specificity: Assert on exact expected values, not vague properties.

Weak — passes even if the result is completely wrong
JavaassertNotNull(result); assertTrue(result.getItems().size() > 0);
Strong — verifies exact expected behavior
JavaassertEquals(Money.of(150, Currency.USD), calculator.calculateTotal(order)); assertThat(result.getItems()).containsExactly("Widget", "Gadget");

Determinism: Assertions must produce the same result every time. Assertions that depend on wall-clock time, random values, or network state are non-deterministic and produce flaky tests. All external sources of non-determinism must be controlled (inject a Clock, seed a Random, stub external calls).

One logical concept per test: The “one assertion per test” rule is widely cited but misunderstood. The actual principle is one logical concept. Multiple assertions verifying the same outcome are fine — if cancelling a reservation should return HTTP 200 and remove it from the database, those two assertions belong together. Multiple assertions verifying different behaviors are problematic. Gerard Meszaros calls the anti-pattern “Assertion Roulette” — unrelated assertions that make it impossible to determine which behavior failed. — Gerard Meszaros, xUnit Test Patterns: Refactoring Test Code, Addison-Wesley, 2007; see also “Stop requiring only one assertion per unit test”, Stack Overflow Blog, 2022

Meaningful failure messages: When a test fails, the combination of the test name and the assertion message should diagnose the problem without reading the test body. Fluent assertion libraries (AssertJ for Java) produce clear messages by default:

AssertJ failure outputExpecting: <45.00 USD> to be equal to: <50.00 USD> but was not.

Testability Is a Design Signal

Multiple authorities converge on this insight: if a unit is hard to test in isolation, the design is coupled. This is not a testing problem — it is a design problem. — See Michael Feathers, Working Effectively with Legacy Code, Prentice Hall, 2004; Steve Freeman & Nat Pryce, Growing Object-Oriented Software, Guided by Tests, Addison-Wesley, 2009

Testable (Good Design)Untestable (Bad Design)
Dependencies injected via constructorDependencies created internally with new
Pure functions (same input → same output)Functions that read global state or system clock
Small, focused classesGod classes with 20+ methods
Clear interfaces at boundariesDirect calls to infrastructure

TDD advocates argue that writing tests first improves design precisely because it forces you to confront coupling, hidden dependencies, and mixed concerns before they are baked into the implementation.

The F.I.R.S.T. Properties

Robert C. Martin defines five non-negotiable properties for unit tests: — Robert C. Martin, Clean Code, Chapter 9: Unit Tests, Prentice Hall, 2008

PropertyMeaningViolation
FastTests run in milliseconds. A slow test suite is a test suite that developers stop running.Tests that hit databases, networks, or file systems without justification
IndependentNo test depends on another test. Run any test alone, in any order, in parallel.Shared mutable state between tests, tests that must run in sequence
RepeatableSame result on any machine, any time. No environmental dependencies.Tests that depend on timezone, locale, file paths, or network availability
Self-validatingBoolean output: pass or fail. No human interpretation needed.Tests that print output for manual inspection
TimelyWritten close in time to the code — ideally before (TDD).Tests written weeks later from memory, missing edge cases the developer has forgotten

The Economics of Testing

Defects found later cost more to fix. The mechanism is straightforward: a requirements bug requires changing a document; a production bug requires diagnosis, code changes, rebuilds, re-testing, patch deployment, customer support, and potential safety/regulatory reporting.

Key data points:

  • The NIST 2002 study estimated the annual cost of software defects in the US at $59.5 billion, with improved testing infrastructure able to reduce this by $22.2 billion — NIST Planning Report 02-3, “The Economic Impacts of Inadequate Infrastructure for Software Testing”, 2002
  • Google found that 84% of test failure pass-to-fail transitions are flaky, not real bugs — meaning bad tests waste enormous engineering time — “Flaky Tests at Google and How We Mitigate Them”, Google Testing Blog, 2016
  • Atlassian estimated 150,000 developer hours per year lost to flaky test investigation — “Measuring the Cost of Flaky Tests”, Atlassian Engineering Blog

In safety-critical domains, post-release defect costs are amplified further by re-certification impact: a defect found after certification may require re-verification, re-validation, and regulatory notification — costs that can dwarf the original development effort.

Bad Tests Are Worse Than No Tests

A test suite that is brittle, slow, flaky, or coupled to implementation becomes a liability. Developers stop trusting the suite, stop running it, and stop writing tests. The “test tax” — the cost of maintaining bad tests — compounds with every refactoring. The solution is not fewer tests but better tests: behavior-focused, specific assertions, fast execution, zero flakiness.

3.2 Test Derivation Strategies

Test derivation is systematic, not creative. Four strategies exist, each targeting a different dimension of coverage. A well-tested unit uses all four. No single strategy is sufficient alone.

ISO 26262 is the only safety standard that explicitly names these methods (Part 6, Table 11). DO-178C requires the same practices in substance (“normal range testing” per Section 6.4.2.1 + “robustness testing” per Section 6.4.2.2) but uses different terminology. ASPICE and IEC 62304 leave method selection to the project. — See ISO 26262-6:2018 §9–10; DO-178C §6.4; ASPICE SWE.4; IEC 62304 §5.5

Strategy 1: Requirement-Based Testing

Source: The behavior rules in the Detailed Design.

Each behavior entry describes a rule, condition, or processing step. Every one is a testable claim about what the unit does. Read the rule, identify the trigger and expected outcome, write one test.

Compound conditions require decision table testing — when a behavior has multiple conditions (e.g., “when mode is startup AND rate exceeds limit”), test each meaningful combination:

Example — Decision table for compound condition
Java// Design says: "When mode is startup AND rate exceeds limit, clamp to minimum" @Test void startup_mode_with_excess_rate_clamps_to_minimum() { calculator.setMode(Mode.STARTUP); FuelResult result = calculator.calculate(rateAboveLimit); assertEquals(MINIMUM_RATE, result.adjustedRate(), 0.001); } @Test void startup_mode_within_limit_does_not_clamp() { calculator.setMode(Mode.STARTUP); FuelResult result = calculator.calculate(rateWithinLimit); assertNotEquals(MINIMUM_RATE, result.adjustedRate(), 0.001); } @Test void normal_mode_with_excess_rate_does_not_clamp() { calculator.setMode(Mode.NORMAL); FuelResult result = calculator.calculate(rateAboveLimit); assertNotEquals(MINIMUM_RATE, result.adjustedRate(), 0.001); }

Sequential steps: If the behavior describes a sequence, test the full happy path plus each step’s failure path.

Strategy 2: Equivalence Class Partitioning

Source: The input and output specifications in the design.

Divides the input domain into groups where the unit should behave identically for all values within a group. Test one representative from each group instead of exhaustively testing all values.

Input TypeClasses to Derive
enum with valuesOne class per enum value
integer/float with constraintsValid range, below range, above range
StringEmpty, typical, at max length (if constrained)
booleantrue, false
ListEmpty, single element, multiple elements, at max (if constrained)
ObjectValid complete, missing optional fields, missing required fields
Example — Equivalence classes for a temperature input (range: −40 to 150)
Java@Test void nominal_temperature_classified_as_normal() { assertEquals("NORMAL", sensor.classify(55.0)); // mid-range representative } @Test void below_range_temperature_rejected() { assertThrows(OutOfRangeException.class, () -> sensor.classify(-41.0)); // below minimum } @Test void above_range_temperature_rejected() { assertThrows(OutOfRangeException.class, () -> sensor.classify(151.0)); // above maximum }

Combining inputs: Don’t test all combinations (combinatorial explosion). Test each input’s classes independently while holding others at valid defaults. Add combination tests only for inputs the design says interact.

Strategy 3: Boundary Value Analysis

Source: The input constraints and configuration defaults in the design.

Targets the edges of valid ranges, where off-by-one errors and incorrect comparisons hide. For a numeric input with constraint >= min and <= max:

Test PointValueExpected
Below minimummin − 1 (or min − epsilon for floats)Error / rejection
At minimumminValid behavior
Nominal(min + max) / 2Valid behavior
At maximummaxValid behavior
Above maximummax + 1 (or max + epsilon)Error / rejection
Example — Boundary tests for fuel rate (valid range: 0.0 to 100.0)
Java@Test void rate_below_minimum_rejected() { assertThrows(InvalidRateException.class, () -> calculator.process(-0.001)); // just below 0.0 } @Test void rate_at_minimum_accepted() { FuelResult result = calculator.process(0.0); assertTrue(result.isValid()); } @Test void rate_at_maximum_accepted() { FuelResult result = calculator.process(100.0); assertTrue(result.isValid()); } @Test void rate_above_maximum_rejected() { assertThrows(InvalidRateException.class, () -> calculator.process(100.001)); // just above 100.0 }

String and collection boundaries:

Input TypeBoundary Tests
String with length constraintEmpty, length 1, at max length, max + 1
List with min/max itemsEmpty, min items, max items, max + 1
Map/objectEmpty, one entry, at capacity

Strategy 4: Error Handling and Fault Injection

Source: The error handling specification, plus gaps not covered by explicit error handling.

Explicit error handling: Each entry in the design’s error handling section has a condition and a specified behavior. For each: set up the condition, call the unit, assert the specified response.

Example — Testing explicit error handling
Java// Design says: "When sensor returns NaN, unit shall return last known good value" @Test void nan_sensor_reading_returns_last_known_good() { unit.process(validReading(42.0)); // establish last known good FuelResult result = unit.process(nanReading()); // inject NaN assertEquals(42.0, result.value(), 0.001); }

Implicit error handling (fault injection): The design may not cover every failure mode. Probe for these:

Fault CategoryWhat to Test
Null inputsPass null for each input — does it fail gracefully?
Type violationsPass wrong types if the language allows (dynamic languages)
Resource exhaustionEmpty collections, zero-length strings, maximum-size inputs
Dependency failuresIf the design implies external calls, what if they fail/timeout?
Concurrent accessIf constraints mention thread-safety, test concurrent calls
State corruptionIf internal state exists, call methods in unexpected order

Combining Strategies: The Coverage Matrix

A coverage matrix maps every design element to the test cases that verify it. It is the primary tool for ensuring completeness — every row must have at least one test.

Example — Coverage matrix
| Design Element | Test Case(s) | Strategy | |-----------------------------|---------------------------------------|-------------| | behavior: startup limit | startup_mode_limits_rate_to_min | req-based | | behavior: normal calc | normal_mode_applies_correction | req-based | | error: negative rate | negative_rate_throws | error | | input: rate range [0,100] | rate_at_zero, rate_at_max | boundary | | input: rate range [0,100] | rate_at_fifty (nominal) | equivalence | | input: rate above max | rate_above_max_rejected | boundary |

Walk the checklist after writing all tests: every behavior rule tested? Every error condition? Every input covered by at least equivalence class + boundary tests? Stateful transitions? Configuration variants?

3.3 Test Structure and Readability

Consistent test structure makes tests readable, maintainable, and self-documenting. Tests should be “boring and obvious” — no cleverness, no surprises.

Arrange / Act / Assert

Every test has three clear sections, visually separated:

  1. Arrange — set up inputs, state, and dependencies
  2. Act — call the unit under test exactly once
  3. Assert — verify the output matches the expected result from the design
Good — Clear Arrange/Act/Assert with visual separation
Java@Test void startup_mode_limits_rate_to_minimum() { // Arrange var config = new FuelConfig(Mode.STARTUP, 0.0, 100.0, 0.75); var calculator = new FuelRateCalculator(config); double excessRate = 150.0; // Act FuelResult result = calculator.calculate(excessRate); // Assert assertEquals(0.0, result.adjustedRate(), 0.001); assertFalse(result.withinLimits()); }

The one Act rule: each test calls the unit under test exactly once. Multiple Act sections in one test usually means the test is verifying multiple behaviors. The BDD equivalent — Given/When/Then — maps directly: Given (Arrange), When (Act), Then (Assert).

Test Naming

A test name is the first thing a developer reads when a test fails. It should communicate the scenario and expected outcome without requiring the developer to read the test body.

ConventionExampleTrade-off
method_condition_expectedcalculateTotal_withDiscount_returnsReducedPriceSystematic; breaks if method is renamed
should_expected_when_conditionshould_return_zero_when_list_is_emptyReads like a requirement; verbose
given_when_thengiven_empty_cart_when_checkout_then_throwMaps to BDD and AAA; very verbose
Sentence (with @DisplayName)@DisplayName("Rejects negative fuel rate with error")Most readable in reports; extra annotation

Pick one convention per project and stick with it. All conventions share the same goal: describe the scenario, not the method being tested.

Bad — Method names, not scenarios
test_calculate test_error test1 testProcess
Good — Scenarios that read like requirements
rejects_negative_fuel_rate_with_error startup_mode_limits_rate_to_minimum normal_mode_applies_correction_factor nan_sensor_reading_returns_last_known_good

Parameterized Tests

When multiple test cases follow the same structure but differ only in input values and expected outputs, parameterized tests eliminate duplication. In JUnit 5:

Example — Parameterized boundary value tests
Java@ParameterizedTest(name = "rate {0} is valid") @ValueSource(doubles = {0.0, 50.0, 100.0}) void valid_rates_are_accepted(double rate) { FuelResult result = calculator.process(rate); assertTrue(result.isValid()); } @ParameterizedTest(name = "rate {0} is rejected") @ValueSource(doubles = {-0.001, 100.001, -100.0, 999.9}) void invalid_rates_are_rejected(double rate) { assertThrows(InvalidRateException.class, () -> calculator.process(rate)); }

Parameterized tests map naturally to test derivation strategies: @ValueSource for boundary values, @CsvSource for equivalence classes with expected outputs, @EnumSource for enum-based partitioning.

When not to parameterize: If different inputs require different assertion logic, separate tests are clearer. Parameterized tests should share both structure and assertion pattern. When the test name with parameters doesn’t clearly communicate the scenario, use separate named tests instead.

Test File Organization

The universally accepted convention is mirror structure: the test directory mirrors the source directory. In Maven/Gradle projects:

src/main/java/com/example/billing/InvoiceCalculator.java src/test/java/com/example/billing/InvoiceCalculatorTest.java

One test class per production class supports discoverability, package-private access in Java, automatic build tool discovery, and direct V-model traceability (each unit has a corresponding verification artifact).

For complex units, use @Nested classes (JUnit 5) to group related tests:

Example — @Nested for grouping by state or behavior
Javaclass FuelRateCalculatorTest { @Nested class WhenInStartupMode { @Test void limits_rate_to_minimum() { ... } @Test void ignores_correction_factor() { ... } } @Nested class WhenInNormalMode { @Test void applies_correction_factor() { ... } @Test void rejects_negative_rate() { ... } } @Nested class ErrorHandling { @Test void nan_returns_last_known_good() { ... } @Test void null_input_throws_IllegalArgument() { ... } } }

Reducing Noise, Highlighting Intent

A test should make the relevant details obvious and hide the irrelevant ones. If a test creates a 10-field object but only the “rate” field matters, only the rate should be visible in the test body. Use builders or factory methods for the rest (see Section 3.5 for Test Data Builders).

Rules for test code:

  • No conditional logic. Tests should be straight-line code. If your test has if/else or loops, it is doing too much.
  • No inheritance. Sharing setup through test class hierarchies hides context and creates Mystery Guest problems. Prefer composition (helper methods, builders).
  • Minimal shared state. Use @BeforeEach for genuinely common setup, but keep it short. If a reader must scroll to @BeforeEach to understand a test, the setup is hiding too much.

3.4 Test Doubles

A test double is any object that stands in for a real dependency during testing — analogous to a stunt double in film. Gerard Meszaros defined the taxonomy in xUnit Test Patterns: Refactoring Test Code (Addison-Wesley, 2007); Martin Fowler popularized it in “Mocks Aren’t Stubs” (martinfowler.com, 2007).

The Taxonomy

TypePurposeHas Behavior?Verifies Interactions?
DummyFills a parameter list; never actually usedNoNo
StubReturns canned answers to control indirect inputsCanned onlyNo
SpyRecords calls for later assertionCanned onlyAfter the fact
MockPre-programmed with expectations about calls it should receiveCanned onlyYes (behavior verification)
FakeWorking but simplified implementation (e.g., in-memory database)YesNo

The terminology problem: In practice, “mock” is used as a generic term for all test doubles. Developers say “I’ll mock that” when they might actually create a stub or fake. Mocking frameworks like Mockito compound this — mock(SomeClass.class) creates an object that can function as a stub, spy, or mock depending on configuration. The imprecision matters because the choice of test double type has real consequences for test quality.

State Verification vs. Behavior Verification

This is Fowler’s key distinction (“Mocks Aren’t Stubs”). State verification checks the outcome — what state the system is in after the action. Behavior verification checks the interactions — what calls were made to collaborators.

State verification — checks the outcome
Java@Test void filling_order_reduces_warehouse_inventory() { Warehouse warehouse = new Warehouse(); warehouse.add("Widget", 50); Order order = new Order("Widget", 10); order.fill(warehouse); assertTrue(order.isFilled()); assertEquals(40, warehouse.getInventory("Widget")); }
Behavior verification — checks the interactions (fragile)
Java@Test void filling_order_calls_warehouse_correctly() { Warehouse mockWarehouse = mock(Warehouse.class); when(mockWarehouse.hasInventory("Widget", 10)).thenReturn(true); Order order = new Order("Widget", 10); order.fill(mockWarehouse); // Breaks if implementation changes HOW it checks inventory verify(mockWarehouse).hasInventory("Widget", 10); verify(mockWarehouse).remove("Widget", 10); }

State verification aligns with behavior testing (Section 3.1) because it verifies what happened rather than how it happened. Only use behavior verification when the interaction itself is the observable behavior (e.g., verifying that a notification was sent).

When to Use Test Doubles

  • External dependencies — network, file system, database, hardware sensors
  • Slow dependencies — anything that would make tests too slow to run frequently
  • Non-deterministic dependencies — system clock, random number generators, external APIs
  • Hard-to-configure dependencies — require complex infrastructure or state setup
  • Error paths — when you need to simulate failures that are hard to trigger naturally

The preference hierarchy: Use the simplest double that gives the right signal.

Real implementation → prefer over → Verified fake → prefer over → Stub → prefer over → Mock

When NOT to Use Test Doubles (Over-Mocking)

Over-mocking is the most common test double mistake. It produces tests that pass against a fiction — they verify that the code does what the mocks say it should do, not what the real system does.

Bad — Over-mocking: testing interaction choreography
Java@Test void processOrder_callsEverythingInOrder() { verify(validator).validate(order); verify(pricer).calculatePrice(order); verify(taxService).applyTax(any()); verify(repository).save(any()); verify(notifier).sendConfirmation(any()); // Tests pass even if the actual business logic is completely wrong // because nothing verifies the final state or output }

Signs of over-mocking:

  • Test setup is longer than the test itself
  • Tests break when you refactor internal structure without changing behavior
  • The test uses verify() more than assertEquals()
  • You cannot describe what the mock replaces and why in one sentence
  • You are mocking value objects, data structures, or types you don’t own

Only Mock Types You Own

Freeman and Pryce: don’t mock third-party APIs directly. When you mock a library, you encode assumptions about its behavior that can be wrong or go stale. Instead: — Steve Freeman & Nat Pryce, Growing Object-Oriented Software, Guided by Tests, Addison-Wesley, 2009, Chapter 8; see also “Don’t mock what you don’t own”, Findmypast Tech

  1. Define an interface in your domain language for what you need
  2. Write an adapter that implements the interface using the third-party API
  3. Mock the interface you own in unit tests
  4. Integration-test the adapter against the real API

This connects directly to the Dependency Inversion Principle (DIP) and hexagonal architecture — depend on abstractions you control, not on implementations you don’t.

Contract Testing: Keeping Fakes Honest

Fakes can drift from reality — the real database has a quirk that the in-memory fake doesn’t replicate. Contract tests solve this: write an abstract test suite for the interface, then run it against both the fake and the real implementation.

Example — Contract test shared between fake and real implementation
Java// Abstract contract test — defines what any UserRepository must do abstract class UserRepositoryContractTest { abstract UserRepository createRepository(); @Test void saved_user_can_be_found_by_id() { UserRepository repo = createRepository(); User user = new User("u-1", "Alice"); repo.save(user); assertEquals(Optional.of(user), repo.findById("u-1")); } @Test void finding_nonexistent_user_returns_empty() { UserRepository repo = createRepository(); assertEquals(Optional.empty(), repo.findById("no-such-id")); } } // Run against the fake (in unit tests) class InMemoryUserRepositoryTest extends UserRepositoryContractTest { UserRepository createRepository() { return new InMemoryUserRepository(); } } // Run against the real implementation (in integration tests) class PostgresUserRepositoryIT extends UserRepositoryContractTest { UserRepository createRepository() { return new PostgresUserRepository(testDb); } }

Test Doubles and Documentation

Every test double in a test suite should be documentable: what it replaces, what contract it implements, and why a real implementation isn’t used. In safety-critical domains, an auditor may ask why tests use substitutes rather than real components — the answer must be defensible (speed, determinism, isolation from hardware) and the substitution must not mask integration issues.

3.5 Test Smells and Maintainability

Test smells are surface indicators of deeper test design problems — the testing equivalent of code smells. The concept was introduced by van Deursen, Moonen, van den Bergh, and Kok in their paper “Refactoring Test Code” (XP2001 conference, 2001) and expanded by Meszaros into a comprehensive catalog of 18 smells in xUnit Test Patterns: Refactoring Test Code (Addison-Wesley, 2007; online catalog).

The Most Damaging Smells

SmellSymptomHarmFix
Fragile TestTest breaks when unrelated code changesDevelopers stop trusting tests, stop refactoringTest behavior not implementation; use builders
Obscure TestCannot see cause and effect between setup, action, and assertionTests fail as documentation; maintenance cost increasesInline essential setup; extract irrelevant details to helpers
Eager TestTests too many things in one test methodWhen it fails, you don’t know which behavior brokeSplit into focused tests, one concept each
Mystery GuestTest depends on data or resources not visible in the test bodyCannot understand the test without reading external files or setupMake all relevant data visible in the test; use inline test data
General Fixture@BeforeEach creates far more state than any single test needsHard to understand what each test depends on; changes to fixture break unrelated testsMove setup into individual tests or use Test Data Builders
Conditional Test Logicif/else or loops in test codeThe test itself might have bugs; test should be straight-lineSplit into separate tests for each condition
Erratic Test (Flaky)Passes sometimes, fails sometimes, without code changesErodes trust in entire suite; wastes investigation timeControl non-determinism (inject Clock, seed Random, isolate state)
Slow TestTest takes seconds instead of millisecondsDevelopers run tests less often; bugs accumulateReplace I/O with test doubles; move slow tests to integration tier

Fragile Tests: The Deep Dive

Fragile tests are the most damaging smell because they directly undermine the purpose of testing. Meszaros identifies four types of fragility (xUnit Patterns: Fragile Test):

Interface Sensitivity: Tests break when method signatures or constructors change, even when behavior is preserved. Common when tests call constructors with positional arguments.

Fragile — Constructor with positional args breaks when signature changes
JavaOrder order = new Order("ORD-001", "customer-42", "2024-01-15", "PENDING", "USD", null, null);
Robust — Builder with sensible defaults
JavaOrder order = anOrder().withId("ORD-001").build();

Behavior Sensitivity: Tests break when internal implementation changes but observable behavior does not. Strongly associated with behavior verification using mock objects — the test describes how the software works, so it only passes if implemented one specific way.

Data Sensitivity: Tests break when test data changes. A shared fixture used by 50 tests breaks 30 of them when one test adds a new field.

Context Sensitivity: Tests break in different environments (different timezone, locale, machine, database). Caused by failing to isolate from external context.

Common Anti-Patterns

The “Doesn’t Throw” Test: The only assertion is that calling the function doesn’t raise an exception.

Bad
@Test void processes_input() { unit.process(validInput); // no assertion — passes for any implementation }

The Mirror Test: The test recomputes the expected value using the same logic as the implementation. If the formula has a bug, the test has the same bug.

Bad — Duplicating implementation logic
double expected = input.volume / input.time * CORRECTION_FACTOR; assertEquals(expected, unit.calculateRate(input));
Fix — Hardcoded expected value from design spec
// 100.0 / 10.0 * 0.75 = 7.5, per design spec Table 3 assertEquals(7.5, unit.calculateRate(input));

The Tautology Test: Asserts something always true regardless of implementation.

Bad
assertNotNull(result); // any non-trivial implementation returns something

The Test That Tests the Framework: Tests that the language or framework works, not that your code works. If removing your implementation would still let the test pass, it is worthless.

Quick Self-Check

For each test you write, ask:

  1. Delete test: If I deleted the implementation, would this test fail? If no → worthless.
  2. Wrong answer test: If the implementation returned a wrong but structurally valid answer, would this test catch it? If no → weak assertion.
  3. Mock audit: For every test double — can I state in one sentence what it replaces and why? If no → unclear dependency.
  4. Name check: Does the test name tell me what scenario it covers? If no → poor documentation value.

Maintainability Patterns

Test Data Builders: Fluent builders for creating test objects with sensible defaults. Only the field relevant to the specific test is set explicitly; everything else uses a valid default.

Example — Test Data Builder
Javapublic class OrderBuilder { private String id = "ORD-" + UUID.randomUUID().toString().substring(0, 8); private Money total = Money.of(100.00); private String currency = "USD"; private OrderStatus status = OrderStatus.PENDING; public static OrderBuilder anOrder() { return new OrderBuilder(); } public OrderBuilder withTotal(Money total) { this.total = total; return this; } public OrderBuilder withStatus(OrderStatus s) { this.status = s; return this; } public Order build() { return new Order(id, total, currency, status); } } // In tests — only the relevant field is visible Order order = anOrder().withTotal(Money.of(250)).build();

Custom Assertion Methods: Domain-specific assertions that improve readability and centralize verification logic.

Example — Custom assertion with AssertJ
Java// Instead of three low-level assertions: assertEquals("ACTIVE", account.getStatus()); assertTrue(account.getBalance().compareTo(BigDecimal.ZERO) > 0); assertNotNull(account.getActivationDate()); // One domain-specific assertion: assertThat(account).isActiveWithPositiveBalance();

Object Mother: Factory methods for common test objects. Simpler than builders but less flexible — best for objects that rarely need customization.

The Rule of Three: Tolerate duplication in tests until you see the same pattern three times. Then extract a shared helper. Premature extraction creates obscure tests where the reader must chase through helper hierarchies to understand a single test.

3.6 Coverage and Completeness

Coverage measures how thoroughly tests exercise the code and the design. There are two fundamentally different types, and confusing them is a common mistake.

Requirements Coverage vs. Structural Coverage

TypeWhat It MeasuresHow to Achieve
Requirements CoverageEvery design element has at least one testCoverage matrix (Section 3.2)
Structural CoverageEvery code path has been executed by testsCode coverage tools (JaCoCo, lcov, coverage.py)

Requirements coverage comes first. It answers: “Did we test what the design says the unit should do?” Structural coverage comes second. It answers: “Did we accidentally leave any code paths untested?”

Why 100% Structural Coverage Is Not Enough

You can achieve 100% line coverage with tests that assert nothing. You can achieve 100% branch coverage without testing any boundary values. Structural coverage is a necessary complement to requirements coverage, not a replacement for it. A test suite with 100% structural coverage but no requirements traceability is a compliance failure in every safety standard.

DO-178C Section 6.4.4 makes this explicit: structural coverage is an adequacy check, not a test derivation method. If coverage analysis reveals unexecuted code, the response is to add requirements-based tests, identify dead code, or trace the gap to a missing requirement — not to write tests “to the code” just to achieve numbers. — DO-178C §6.4.4, “Structural Coverage Analysis”

Structural Coverage Levels

LevelWhat It MeasuresWhen Required
Statement CoverageEvery executable statement has been invoked at least onceASIL A / DAL D — minimum baseline
Decision (Branch) CoverageEvery branch (true/false) of every decision has been takenASIL B / DAL C
MC/DCEvery condition in a decision independently affects the outcomeASIL D / DAL A — most rigorous

MC/DC Explained

Modified Condition/Decision Coverage requires test cases that show each condition in a compound decision independently affects the result. For if (a && b):

TestabResultShows
1truetruetrueChanging a alone flips the result
2falsetruefalse
3truefalsefalseChanging b alone flips the result (compare with test 1)

Three test cases are sufficient: tests 1, 2, 3.

Mutation Testing: The Real Quality Metric

Code coverage answers “was this line executed?” but not “would the test catch a bug here?” Mutation testing answers the second question.

How it works: A mutation testing tool (PIT/pitest for Java) systematically introduces small faults (“mutants”) into the code — negating a condition, changing a return value, removing a method call — and runs the test suite against each mutant. If the tests kill the mutant (detect the fault), the tests are effective at that location. If the mutant survives (tests still pass despite the fault), the tests have a gap.

Common mutation operators:

OperatorExample MutationWhat a Surviving Mutant Reveals
Negate conditionalif (a > b) → if (a <= b)Boundary not tested or assertion too weak
Change return valuereturn true; → return false;Return value not asserted
Remove method calllist.add(item); → deletedSide effect not verified
Math operator changea + b → a - bCalculation result not checked against specific expected value

Why it matters: A test suite can achieve 100% line and branch coverage while having a mutation score near zero — meaning the tests execute every line but verify nothing meaningful. This is especially common with AI-generated tests (see Section 3.7). Mutation testing is the quality gate that catches tautological tests, weak assertions, and missing edge cases that coverage metrics miss.

Dealing with Coverage Gaps

When structural coverage analysis reveals uncovered code:

  1. Missing test case: The design specifies a behavior you forgot to test. Add the test.
  2. Dead code: The code cannot be reached by any input. This is a defect — remove the dead code or fix the design.
  3. Defensive code: Error handling for conditions that “should never happen.” Either test it through fault injection or document why it cannot be reached.

3.7 AI-Assisted Test Development

AI-generated tests are increasingly common. This section addresses the specific quality risks of AI-assisted test development, grounded in empirical data from 2023–2025 studies.

The Empirical Reality

FindingSource
34–62% of LLM-generated unit tests are syntactically invalid (won’t compile)Yuan et al., “On the Evaluation of Large Language Models in Unit Test Generation”, 2024
Without existing test context, only 7.5% of Copilot-generated tests passTU Delft, “Using GitHub Copilot for Test Generation in Python”, AST 2024
AI tests can achieve 100% line/branch coverage with only 4% mutation score“On Mutation-Guided Unit Test Generation”, 2025
47.4% of Copilot-generated tests contain at least one test smell (most common: Assertion Roulette, Magic Number Test)Reported in multiple secondary analyses of Copilot test quality, 2024; primary source unverified
AI coding agents use mocking 95% of the time vs. 91% for humans, with less diversity in test double strategiesIslam et al., “Are Coding Agents Generating Over-Mocked Tests?”, MSR 2026

The headline number: 100% coverage with 4% mutation score. AI tests can execute every line without detecting any faults — they run the code but verify nothing meaningful. Coverage metrics are near-worthless as a quality indicator for AI-generated tests.

Common AI Testing Failures

  • Tautological tests — recomputing expected values using the same logic as the implementation (the Mirror Test from Section 3.5)
  • Assertion-free tests — tests that only check “doesn’t throw” or assertNotNull
  • Happy-path bias — AI generates mostly positive test cases, misses error paths and boundaries
  • Over-mocking — mocks everything, including things that should use real implementations
  • Hallucinated assertions — assertions on properties that don’t exist or don’t mean what the LLM thinks
  • Copy-paste patterns — many structurally identical tests with slight variations that don’t add meaningful coverage

Code-to-Test vs. Spec-to-Test

A fundamental distinction:

  • Code-to-test: “Generate tests for this code.” The AI reads the implementation and produces tests that exercise it. This is what every current AI testing tool does (Copilot, Diffblue Cover, EvoSuite, CodiumAI).
  • Spec-to-test: “Derive tests from this specification.” The AI reads the design and produces tests that verify the specified behavior, independent of the implementation.

Code-to-test is structurally at odds with the V-model principle that tests verify the design, not the code. Tests generated from code will mirror the implementation — including its bugs. Spec-to-test aligns with requirements-based testing.

Best Practices for AI-Assisted Testing

  1. Provide design specifications as context, not just code. AI test quality improves dramatically with a clear specification of expected behavior.
  2. Generate incrementally. One test class at a time. Review and validate before moving on.
  3. Explicitly prompt for edge cases. AI rarely generates boundary values, error paths, or concurrent access tests unless specifically asked.
  4. Review every test against the anti-pattern checklist (Section 3.5). AI tests frequently contain Mirror Tests, assertion-free tests, and over-mocking.
  5. Run mutation testing as the quality gate. Coverage metrics are insufficient. Mutation score is the reliable indicator of whether AI-generated tests actually catch faults.
  6. Treat AI tests as first drafts. They are starting points for a human engineer, not finished artifacts.

Safety-Critical Implications

LLMs are non-deterministic and unexplainable, making formal tool qualification (DO-330 TQL-1, ISO 26262 Part 8) currently infeasible. AI-generated tests must be treated as developer artifacts subject to full human review — the same scrutiny as human-written tests, with additional attention to the failure modes listed above.

If an AI agent writes both the code and the tests, the “tests verify design, not code” principle is at risk — the agent may unconsciously align test expectations with implementation behavior rather than design intent. Independent human review of the test derivation rationale (the coverage matrix, not just the test code) is a mitigation.

V-Model Specific Considerations

Beyond writing quality tests, the V-model imposes specific requirements on how unit testing is managed and documented.

Test Independence

Safety standards require varying degrees of independence between the person who writes the code and the person who writes or reviews the tests:

Assurance LevelIndependence Requirement
Low (DAL D / ASIL A)Same developer may write code and tests
Medium (DAL C / ASIL B)Tests should be reviewed by someone other than the author
High (DAL A–B / ASIL C–D)Tests should be derived and/or reviewed by an independent team

Test Documentation

Standards require documented evidence of testing. The level of formality scales with assurance:

  • Test plan (what will be tested, how, with what criteria for pass/fail)
  • Test cases (the actual tests with expected results)
  • Test results (evidence of execution: pass/fail, coverage metrics, environment)
  • Traceability data (mapping from design elements to tests to results)

Traceability Requirements

Unit tests must trace bidirectionally to the Detailed Design:

  • Forward trace (design → test): every design element has at least one test
  • Backward trace (test → design): every test maps to a specific design element

Orphan tests (tests with no corresponding design element) are as problematic as orphan code — they indicate either missing design or scope creep in testing.

Robustness Testing

DO-178C requires robustness test cases at DAL C and above (Section 6.4.2.2): boundary values, error paths, timing edge cases, invalid state transitions, out-of-range inputs. ISO 26262 requires fault injection testing at ASIL C and D (Table 10). These map to Strategies 3 and 4 in Section 3.2.

Regression Testing

When code changes, all previously passing tests must continue to pass. In V-model terms, any code change that causes a previously-passing test to fail requires either a fix to the code (the change introduced a defect) or a documented update to the design and corresponding test update (the requirement changed).

Framework Integration

How unit tests integrate with the VModelWorkflow templates, schemas, and traceability system.

Input: Detailed Design Artifact

Unit tests are derived from the Detailed Design artifact (defined in schemas/artifacts/detailed-design.schema.yaml). The derivation strategies in Section 3.2 map to the design artifact’s structure:

  • behavior → Strategy 1 (requirement-based testing)
  • interfaces.inputs → Strategies 2 and 3 (equivalence classes and boundaries)
  • error_handling → Strategy 4 (error handling and fault injection)
  • internal_state → State transition tests
  • configuration → Configuration variant tests

Traceability Links

Unit tests participate in the following trace link types (defined in schemas/traceability/link-types.yaml):

Detail Design → verified-by → Unit Test
Source Code → verified-by → Unit Test

Coverage Matrix as Trace Validation

The coverage matrix (Section 3.2) is not just a testing tool — it is a traceability artifact. It provides the mapping that the traceability validation engine needs to check completeness: every design element has a test, every test maps to a design element.

Test Result Artifacts

Test execution produces result artifacts that feed into the traceability system: test execution logs, pass/fail status per test case, structural coverage reports, and the coverage matrix with pass/fail annotations.
