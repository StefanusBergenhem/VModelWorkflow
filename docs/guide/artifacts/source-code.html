Level 4 — Implementation

What is Source Code?

Source code is the human-readable instructions that tell a computer what to do. But in professional software engineering, source code is more than “the thing that runs” — it is a traceable engineering artifact that implements a design specification.

The distinction matters. Writing code that works is necessary but not sufficient. Professional source code must be:

  • Correct — it implements the design specification faithfully, no more and no less
  • Verifiable — a tester can derive tests from the design and check that the code passes them
  • Traceable — every public function, class, or module maps back to a specific design element
  • Maintainable — another engineer (or a future version of yourself) can read, understand, and safely modify the code
  • Reviewable — a reviewer can assess whether the code complies with the design and coding standards

Code as a Communication Medium

Code is read far more often than it is written. It communicates intent to three audiences: the compiler (which enforces syntax), the test suite (which enforces behavior), and other humans (who enforce understanding). Writing code that satisfies only the compiler is the lowest possible bar.

Source Code in the V-Model

Source code sits at the bottom of the V — the vertex where the left side (specification) meets the right side (verification). It is the concrete realization of the Detailed Design.

Position in the V

The left arm of the V progressively decomposes abstract requirements into concrete designs:

System Req → SW Req → Architecture → Detail Design → Source Code

Source code is the final output of the specification side. It takes the Detailed Design as input and produces executable software. On the right side, unit tests verify that the code correctly implements the design.

What “Implements” Means

In V-model terms, implementation is a one-to-one mapping from design elements to code structures:

Design ElementCode StructureTraceability
Unit (in detailed design)Class or moduleOne design unit → one code file (typically)
Interface inputs/outputsMethod signaturesTypes, names, and constraints match the design
Behavior rulesMethod bodiesLogic implements each rule faithfully
Error handlingException handling, return codesEach error condition has a corresponding code path
Internal stateInstance fieldsState variables match the design specification
ConfigurationConstructor parameters, config objectsConfigurable values match design defaults and constraints

The Implementation Contract

Implementation is not freewheeling. The code must implement exactly what the design specifies — no more, no less. Adding functionality beyond the design (“gold plating”) creates untraceable code. Omitting functionality creates verification gaps. Both are compliance failures in safety-critical domains.

Standards Perspective

Every major safety standard explicitly addresses source code as a controlled artifact:

StandardSectionKey Requirement

ASPICESWE.3Software units must be produced according to the detailed design. Coding standards compliance is a base practice.

Producing Quality Source Code

This section is the comprehensive guide to writing professional, industry-standard source code. It covers coding standards, clean code principles, architecture at the code level, AI-assisted development practices, and code review. Each principle includes a concrete example.

3.1 Coding Standards

A coding standard is a documented set of rules governing how source code is written in a project. It is not a style preference — it is an engineering control required by every major safety standard.

Why You Need a Coding Standard

Every safety standard (DO-178C, ISO 26262, ASPICE, IEC 62304) requires a defined coding standard. The reasons are practical:

  • Reduces ambiguity — two developers make the same choice for the same situation
  • Enables static analysis — tools can check compliance automatically
  • Reduces cognitive load during review — reviewers focus on logic, not formatting
  • Prevents dangerous language features — language subsets exclude error-prone constructs

Components of a Coding Standard

1. Naming Conventions

Names are the primary documentation. A name should reveal intent, provide context, and be searchable.

Bad
double d = t * 7; int n = getN(); void proc(String s) { ... }
Good
double weeklyDistanceKm = dailyDistanceKm * DAYS_PER_WEEK; int activeConnectionCount = connectionPool.getActiveCount(); void validateSensorReading(String rawReading) { ... }

Rules of thumb:

  • Use domain vocabulary from the project glossary — if the design says “fuel rate”, the code says fuelRate, not rate or fr
  • Boolean variables and methods read as true/false questions: isValid, hasExpired, canRetry
  • Avoid abbreviations unless universally understood in the domain (rpm, id)
  • Class names are nouns (SensorReader), method names are verbs (readSensor)

2. Code Structure and Style

  • Consistent indentation (spaces or tabs — pick one, enforce it)
  • One concept per file — a class and its directly related types
  • Braces always, even for single-line blocks (prevents dangling-else bugs)
  • Import grouping: standard library, third-party, internal — separated by blank lines

3. Complexity Constraints

Complexity limits are measurable, enforceable, and directly linked to defect rates.

MetricLimitWhy
Cyclomatic complexity≤ 10 per functionAbove 10, defect probability increases sharply. Each independent path through a function adds 1.
Nesting depth≤ 3–4 levelsDeep nesting makes control flow hard to follow. Extract nested blocks into named functions.
Function length≤ 50 lines (target 5–15)Short functions are easier to test, review, and understand.
File length≤ 300–500 linesLong files indicate a class doing too much.
Parameters per function≤ 3–4More parameters increase calling complexity. Group related parameters into an object.

Cyclomatic Complexity Explained

Cyclomatic complexity counts the number of independent paths through a function. Start at 1, add 1 for each if, else if, for, while, case, catch, &&, or ||.

Java// Complexity: 1 (straight-line code) double calculateArea(double radius) { return Math.PI * radius * radius; } // Complexity: 4 (1 + 3 conditions) String classify(int score) { if (score >= 90) return "A"; // +1 else if (score >= 70) return "B"; // +1 else if (score >= 50) return "C"; // +1 else return "F"; }

4. Language Subset

A language subset restricts which language features may be used. This prevents error-prone constructs that are legal but dangerous.

LanguageEstablished StandardKey Restrictions
CMISRA C:2023No dynamic memory after init, no recursion, no pointer arithmetic beyond arrays
C++AUTOSAR C++14 / JSF++ AVNo exceptions in safety-critical paths, no multiple inheritance of implementation, RAII required
JavaCERT Java / project-specificNo reflection in safety paths, no raw types, no mutable static fields

Common prohibited patterns across languages:

  • Implicit type conversions that lose precision (e.g., double to int)
  • Goto or unrestricted jumps
  • Dynamic memory allocation in runtime loops (safety-critical systems)
  • Mutable global state — makes concurrency and testing unpredictable
  • Variadic functions — no compile-time type checking on arguments

5. Error Handling

Error handling is not optional. Every error path must be explicitly handled.

  • Use exceptions (or Result types) over error codes — error codes are easy to ignore
  • Use Optional / Option instead of returning null
  • Never swallow exceptions with empty catch blocks
  • Fail fast — detect errors at system boundaries, not deep in business logic
  • Include diagnostic context in error messages (what failed, with what inputs, what was expected)
Bad
Java// Swallowed exception, null return, no context Config loadConfig(String path) { try { return parser.parse(readFile(path)); } catch (Exception e) { return null; // caller gets null with no idea why } }
Good
Java// Specific exception, diagnostic context, fail-fast Config loadConfig(String path) { String content = readFile(path); // throws IOException — let it propagate Config config = parser.parse(content); if (!config.isValid()) { throw new ConfigValidationException( "Invalid config at " + path + ": " + config.getValidationErrors()); } return config; }

6. Comments and Documentation

Comments explain why, not what. If you need a comment to explain what the code does, the code is not clear enough.

  • Public API: document contracts (preconditions, postconditions, exceptions thrown)
  • Non-obvious business rules: explain the domain context
  • Do not comment obvious code — i++ // increment i adds noise, not clarity
  • Comments rot — when code changes, comments often don’t. Prefer expressive code over explanatory comments

Scaling by Assurance Level

Not every project needs the same rigor. Scale the coding standard to the assurance level:

AspectLow AssuranceMedium AssuranceHigh Assurance
Coding standardProject-defined guidelinesFormal standard (e.g., project CERT subset)Industry standard (MISRA, AUTOSAR) with tool enforcement
Complexity limitsGuidelines, not enforcedEnforced by CI, exceptions documentedEnforced by CI, no exceptions
Language subsetAvoid known pitfallsDocumented restricted featuresFormal subset with static analysis verification
Code reviewPeer reviewChecklist-based reviewIndependent review with traceability check

3.2 Clean Code Principles

Clean code principles are about writing code that humans can understand and safely modify. These principles are independent of any specific language or framework.

SOLID Principles

SOLID is the foundation of maintainable object-oriented design. Each principle solves a specific problem that, left unchecked, makes code brittle, hard to test, and expensive to change.

Single Responsibility Principle (SRP)

A class should have one, and only one, reason to change. “Reason to change” means one stakeholder or business rule.

Practical test: Can you describe the class’s purpose in one sentence without using “and”?

Bad — Three responsibilities in one class
Javaclass ReportGenerator { List<Sale> fetchSalesData(DateRange range) { ... } // data access Report applyBusinessRules(List<Sale> sales) { ... } // domain logic void formatAndSendEmail(Report report) { ... } // presentation + I/O }
Good — One responsibility per class
Javaclass SalesRepository { List<Sale> fetchByDateRange(DateRange range) { ... } } class SalesReportBuilder { Report buildFrom(List<Sale> sales) { ... } } class ReportEmailSender { void send(Report report, EmailAddress recipient) { ... } }

Now each class traces to one design element. Testing each is trivial. Changing data access doesn’t affect email formatting.

Open/Closed Principle (OCP)

Software entities should be open for extension, closed for modification. New behavior should not require editing existing, tested code.

Bad — Adding a format requires modifying existing code
Javaclass ReportExporter { void export(Report report, String format) { if (format.equals("PDF")) { ... } else if (format.equals("CSV")) { ... } else if (format.equals("XML")) { ... } // every new format = edit here } }
Good — New formats added without modifying existing code
Javainterface ReportFormatter { byte[] format(Report report); } class PdfFormatter implements ReportFormatter { ... } class CsvFormatter implements ReportFormatter { ... } // Adding XML: create XmlFormatter, no existing code changes class ReportExporter { void export(Report report, ReportFormatter formatter) { byte[] output = formatter.format(report); // ... } }

Liskov Substitution Principle (LSP)

Subtypes must be substitutable for their base types without altering program correctness. If code works with a base type, it must work with any derived type.

Practical test: Can you replace every instance of the parent class with the child class and have all existing tests still pass?

Bad — Square violates Rectangle contract
Javaclass Rectangle { void setWidth(int w) { this.width = w; } void setHeight(int h) { this.height = h; } int area() { return width * height; } } class Square extends Rectangle { void setWidth(int w) { this.width = w; this.height = w; } // surprise! void setHeight(int h) { this.width = h; this.height = h; } // surprise! } // Code expecting Rectangle: set width=5, height=3, expect area=15 // With Square: area is 9. Contract violated.
Good — Model the actual relationship instead of forcing inheritance
Javainterface Shape { int area(); } class Rectangle implements Shape { private final int width, height; Rectangle(int width, int height) { this.width = width; this.height = height; } int area() { return width * height; } } class Square implements Shape { private final int side; Square(int side) { this.side = side; } int area() { return side * side; } } // Both are Shapes. No inheritance means no broken contract. // Code that takes Shape and calls area() works correctly for both.

Interface Segregation Principle (ISP)

Clients should not be forced to depend on methods they do not use. Prefer several small, focused interfaces over one large interface.

Bad — Fat interface forces unnecessary implementations
Javainterface Vehicle { void drive(); void fly(); void sail(); } // A car must implement fly() and sail() with empty bodies or exceptions
Good — Segregated interfaces
Javainterface Drivable { void drive(); } interface Flyable { void fly(); } interface Sailable { void sail(); } class Car implements Drivable { ... } class Seaplane implements Drivable, Flyable, Sailable { ... }

Dependency Inversion Principle (DIP)

High-level modules should not depend on low-level modules. Both should depend on abstractions.

Bad — Business logic depends on infrastructure
Javaclass OrderService { private MySqlDatabase db = new MySqlDatabase(); // hardwired dependency void placeOrder(Order order) { db.insert("orders", order.toMap()); } }
Good — Both depend on abstraction
Javainterface OrderRepository { void save(Order order); } class OrderService { private final OrderRepository repository; OrderService(OrderRepository repository) { this.repository = repository; } void placeOrder(Order order) { repository.save(order); } } // In production: new OrderService(new MySqlOrderRepository()) // In tests: new OrderService(new InMemoryOrderRepository())

DRY, KISS, YAGNI

DRY (Don’t Repeat Yourself): Every piece of knowledge has a single, authoritative representation. Duplicated logic means duplicated bugs and inconsistent fixes.

Bad — DRY violation: tax logic duplicated across methods
Javaclass InvoiceService { Money calculateTotal(List<LineItem> items) { Money subtotal = sumItems(items); Money tax = subtotal.multiply(0.21); // tax logic here return subtotal.add(tax); } Money calculateRefund(LineItem item) { Money tax = item.price().multiply(0.21); // same tax logic, duplicated return item.price().add(tax); } } // If tax rate changes to 0.25, you must find and update both places. // Miss one and invoices disagree with refunds.
Good — Single source of truth for tax calculation
Javaclass TaxCalculator { private static final BigDecimal TAX_RATE = new BigDecimal("0.21"); Money calculateTax(Money amount) { return amount.multiply(TAX_RATE); } } class InvoiceService { private final TaxCalculator tax; Money calculateTotal(List<LineItem> items) { Money subtotal = sumItems(items); return subtotal.add(tax.calculateTax(subtotal)); } Money calculateRefund(LineItem item) { return item.price().add(tax.calculateTax(item.price())); } }

KISS (Keep It Simple, Stupid): The simplest solution that works correctly is the best solution. Complexity is a cost, not a feature.

Bad — KISS violation: over-engineered for a simple task
Java// Task: check if a list contains a negative number boolean hasNegative(List<Integer> numbers) { return numbers.stream() .collect(Collectors.partitioningBy(n -> n < 0)) .get(true) .stream() .findFirst() .isPresent(); }
Good — Simplest correct solution
Javaboolean hasNegative(List<Integer> numbers) { return numbers.stream().anyMatch(n -> n < 0); }

YAGNI (You Ain’t Gonna Need It): Don’t build for hypothetical future requirements. Build what is needed now, make it clean, and extend later if needed.

Bad — YAGNI violation: building for hypothetical future
Java// Requirement: "Calculate fuel rate from a sensor reading" // Developer builds: a plugin system, three strategy interfaces, a factory, // and a config-driven pipeline — because "we might need multiple algorithms someday" interface FuelStrategy { double calculate(double input); } interface FuelStrategyFactory { FuelStrategy create(String name); } class FuelStrategyRegistry { ... } class ConfigDrivenFuelPipeline { ... } // 200 lines of framework for a 5-line calculation
Good — Build what is needed now
Java// Requirement: "Calculate fuel rate from a sensor reading" double calculateFuelRate(double rawRate, double correctionFactor) { return rawRate * correctionFactor; } // If a second algorithm is needed later, refactor then. // You'll know the actual requirement, not a guess.

The Tension Between DRY and KISS

DRY can conflict with KISS. Extracting three similar-but-different code blocks into a shared abstraction satisfies DRY but may introduce a complex abstraction that violates KISS. The rule of thumb: tolerate duplication until you’ve seen the pattern three times (the “Rule of Three”). Then extract, because by the third occurrence you understand the actual commonality.

Function Design

Functions are the atomic unit of code. Well-designed functions make code readable, testable, and composable.

RuleGuidelineWhy
Length5–15 lines (max 50)Short functions do one thing and are easy to name
Do one thingDescribable in one sentence without “and”Enables meaningful names and focused testing
One abstraction levelDon’t mix high-level orchestration with low-level byte manipulationMixing levels makes code hard to follow
Arguments≤ 3 (ideally 0–2)Many arguments suggest the function is doing too much
No side effectsA function named validate should not also save to databaseHidden effects cause bugs that are hard to trace
No flag argumentsDon’t pass booleans that switch behaviorA boolean argument means the function does two things
Bad — Function does multiple things at different abstraction levels
Javavoid processOrder(Order order) { // Validate if (order.getItems().isEmpty()) throw new ValidationException("no items"); if (order.getTotal() < 0) throw new ValidationException("negative total"); // Calculate tax double tax = order.getTotal() * 0.21; order.setTax(tax); // Save String sql = "INSERT INTO orders (id, total, tax) VALUES (?, ?, ?)"; db.execute(sql, order.getId(), order.getTotal(), tax); // Send email emailService.send(order.getCustomerEmail(), "Order confirmed", buildBody(order)); }
Good — Each function does one thing at one abstraction level
Javavoid processOrder(Order order) { validateOrder(order); applyTax(order); orderRepository.save(order); notificationService.sendConfirmation(order); }

Class Design

  • Single Responsibility — same as SRP above, applied at the class level
  • High Cohesion — all methods in a class operate on the same data. If half the methods ignore half the fields, the class has low cohesion and should be split.
  • Composition over Inheritance — prefer “has-a” over “is-a”. Inheritance couples parent and child tightly and makes changes to the parent ripple unpredictably.
  • Depend on Abstractions at Boundaries — at the edges of your module (where it talks to I/O, databases, external services), depend on interfaces, not implementations.
Bad — Inheritance for code reuse
Java// SpecialStack extends ArrayList just to reuse add/get methods class SpecialStack extends ArrayList<String> { // Inherits 30+ methods, most of which break stack semantics // e.g., .add(0, item) inserts at bottom — not stack behavior }
Good — Composition: use a list, don’t be a list
Javaclass SpecialStack { private final List<String> items = new ArrayList<>(); void push(String item) { items.add(item); } String pop() { return items.remove(items.size() - 1); } String peek() { return items.get(items.size() - 1); } boolean isEmpty() { return items.isEmpty(); } // Only stack operations exposed — no way to break semantics }

Code smells are surface indicators of deeper design problems. Recognizing them is the first step to refactoring.

Javavoid applyDiscount(double price, int type) { if (type == 1) return price * 0.9; if (type == 2) return price * 0.8; // What is type 1? What is 0.9? }
Good — Named constants and meaningful types
Javaenum DiscountType { MEMBER, VIP } static final double MEMBER_DISCOUNT = 0.10; static final double VIP_DISCOUNT = 0.20; Money applyDiscount(Money price, DiscountType type) { return switch (type) { case MEMBER -> price.subtract(price.multiply(MEMBER_DISCOUNT)); case VIP -> price.subtract(price.multiply(VIP_DISCOUNT)); }; }

Immutability

Immutable objects cannot be changed after creation. This eliminates an entire category of bugs: unexpected mutation, stale references, and concurrency issues.

Good — Java Record as immutable value object
Java// Java 17 record: immutable by default, equals/hashCode/toString generated record SensorReading(String sensorId, double valueCelsius, Instant timestamp) { SensorReading { if (sensorId == null || sensorId.isBlank()) { throw new IllegalArgumentException("sensorId must not be blank"); } } }

Guidelines:

  • Make fields final by default. Only make them mutable with explicit justification.
  • Use records or value objects for data that doesn’t change after creation.
  • Return defensive copies of mutable collections from getters.
  • Exception: performance-critical inner loops where object allocation is measured as a bottleneck.

Design Patterns

The Gang of Four (GoF) catalogued 23 design patterns in 1994, organized into Creational, Structural, and Behavioral categories. Not all 23 are equally useful in modern practice. The six below are selected because they directly solve problems this guide already covers:

  • Strategy → enforces OCP (add behavior without modifying existing code)
  • Factory Method → enforces DIP (decouple creation from usage)
  • Adapter → enables hexagonal architecture (wrap external interfaces)
  • Decorator → enforces SRP + OCP (add cross-cutting concerns without modifying classes)
  • Builder → manages complexity for objects with many parameters
  • Null Object → eliminates defensive null checks, simplifies control flow

The remaining GoF patterns (Abstract Factory, Prototype, Singleton, Bridge, Composite, Facade, Flyweight, Proxy, Chain of Responsibility, Command, Interpreter, Iterator, Mediator, Memento, Observer, State, Template Method, Visitor) are valid solutions to specific problems. Learn them, but don’t apply them preemptively. If you encounter a problem that a pattern solves, use it. If not, simpler code is better code (KISS, YAGNI).

PatternProblem It SolvesPrincipleExample
StrategyMultiple algorithms for the same task, selected at runtimeOCPDifferent compression algorithms, sorting strategies
Factory MethodObject creation logic is complex or varies by typeDIPCreating validators based on configuration
AdapterIntegrating code with an incompatible interfaceDIPWrapping a legacy API to match your domain port
DecoratorAdding behavior without modifying existing classesSRP + OCPAdding logging, caching, or validation around a service
BuilderConstructing complex objects step by stepKISSBuilding test data or configuration objects
Null ObjectEliminating null checks with a do-nothing implementationLSPNoOpLogger instead of if (logger != null)
Example — Strategy pattern solving an OCP problem
Java// The problem: adding a new validation rule requires editing the validator // The pattern: each rule is a Strategy, new rules are added without touching existing code interface ValidationRule { Optional<String> validate(Order order); } class MinimumAmountRule implements ValidationRule { Optional<String> validate(Order order) { return order.total().isLessThan(Money.of(10)) ? Optional.of("Order total must be at least 10") : Optional.empty(); } } class OrderValidator { private final List<ValidationRule> rules; OrderValidator(List<ValidationRule> rules) { this.rules = rules; } List<String> validate(Order order) { return rules.stream() .map(rule -> rule.validate(order)) .flatMap(Optional::stream) .toList(); } } // Adding a new rule: create a class, add it to the list. No existing code changes.
Example — Decorator pattern adding logging without modifying the service
Java// Original service — untouched class FuelRateService implements RateService { FuelResult calculate(double rawRate) { /* core logic */ } } // Decorator — adds logging, delegates to the wrapped service class LoggingRateService implements RateService { private final RateService delegate; private final Logger logger; LoggingRateService(RateService delegate, Logger logger) { this.delegate = delegate; this.logger = logger; } FuelResult calculate(double rawRate) { logger.info("Calculating rate for input: {}", rawRate); FuelResult result = delegate.calculate(rawRate); logger.info("Result: {}", result); return result; } } // Usage: new LoggingRateService(new FuelRateService(), logger) // FuelRateService has no idea it's being logged. SRP preserved.

Anti-patterns to avoid:

  • Singleton — hidden global state, untestable, concurrency hazard. Use dependency injection instead.
  • Service Locator — hides dependencies, makes code hard to reason about. Use constructor injection instead.
  • God Object — one class that knows and does everything. Split by responsibility.
  • Anemic Domain Model — domain objects with only getters/setters, all logic in service classes. Put behavior where the data is.

3.3 Architecture at the Code Level

Architecture at the code level is about how you organize modules, enforce boundaries, and manage dependencies. Good code-level architecture makes the system testable, changeable, and understandable.

Separation of Concerns

Each unit of code should address one concern. Concerns separate at multiple levels:

  • Function level — each function handles one operation
  • Class level — each class handles one responsibility
  • Module level — each module handles one domain concept
  • Layer level — each layer handles one type of concern (domain, application, infrastructure)

The Core Rule: Business Logic Must Not Depend on Infrastructure

This is the single most important architectural rule at the code level. Business logic (domain rules, calculations, validations) must never import, reference, or depend on infrastructure (databases, HTTP frameworks, file systems, message queues).

If your FuelCalculator class imports java.sql.Connection, the architecture is broken.

Hexagonal Architecture (Ports and Adapters)

Hexagonal architecture enforces the core rule through a simple structure:

Ports are interfaces defined by the domain. They express what the domain needs (e.g., “I need to save an order”) without specifying how.

Adapters are implementations of ports that connect to infrastructure (e.g., “save an order to PostgreSQL”).

The Rule: Dependencies point inward. Domain code never imports adapter code. Adapters import domain interfaces.

Directory Structuresrc/ domain/ ← Pure business logic (no framework imports) model/ ← Entities, value objects ports/ ← Interfaces (OrderRepository, NotificationSender) services/ ← Domain services (FuelCalculator, RiskAssessor) application/ ← Use cases / orchestration PlaceOrderUseCase.java infrastructure/ ← Adapters (implementations of ports) persistence/ ← PostgresOrderRepository messaging/ ← RabbitMqNotificationSender api/ ← HTTP controllers, CLI handlers

Testing Benefit

Because domain code depends only on interfaces (ports), unit tests can use in-memory fakes:

Java// Test uses fake adapter — no database, no network var repository = new InMemoryOrderRepository(); var service = new OrderService(repository); service.placeOrder(testOrder); assertEquals(testOrder, repository.findById(testOrder.getId()));

Testability as a Design Constraint

If code is hard to test, the design is wrong. Testability is a signal, not a goal — it reveals design quality.

Good Signal (Testable)Bad Signal (Untestable)
Dependencies injected via constructorDependencies created internally with new
Pure functions (same input → same output)Functions that read global state or system clock
Small, focused classesGod classes with 20+ methods
Clear interfaces at boundariesDirect calls to frameworks or infrastructure
Value objects (immutable data)Mutable objects passed through many layers

Functional Core, Imperative Shell

Separate pure logic (calculations, decisions) from impure I/O (reading sensors, writing databases). The pure core is trivially testable. The impure shell is thin and simple.

Good — Pure core, imperative shell
Java// PURE CORE — no I/O, no side effects, easy to test record FuelResult(double adjustedRate, boolean withinLimits) {} FuelResult calculateFuelRate(double rawRate, FuelConfig config) { double adjusted = rawRate * config.correctionFactor(); boolean withinLimits = adjusted >= config.minRate() && adjusted <= config.maxRate(); return new FuelResult(adjusted, withinLimits); } // IMPERATIVE SHELL — thin, does I/O, calls the pure core void processSensorReading() { double rawRate = sensorPort.readFuelRate(); // I/O FuelConfig config = configPort.loadFuelConfig(); // I/O FuelResult result = calculateFuelRate(rawRate, config); // PURE telemetryPort.report(result); // I/O }

Cohesion and Coupling

Cohesion (internal) measures how related the elements within a module are. High cohesion is good — everything in the module serves the same purpose.

Coupling (external) measures how dependent modules are on each other. Low coupling is good — changing one module doesn’t force changes in others.

Cohesion TypeQualityExample
FunctionalBestAll methods work together to perform one task
SequentialGoodOutput of one method feeds into the next

CoincidentalWorstMethods are unrelated, just grouped together (“Utils” class)
Coupling TypeQualityExample

Interface-Driven Design

Interfaces define contracts between modules. Use them at boundaries — not everywhere.

When to use interfaces:

  • At infrastructure boundaries (database, external service, file system)
  • When multiple implementations exist or are likely
  • When you need to swap implementations for testing

When NOT to use interfaces:

  • For every class (YAGNI — don’t create FooService and FooServiceImpl when there will only ever be one implementation)
  • For internal domain logic that has no reason to vary

Architecture Checklist

  1. Can you draw the dependency graph, and do arrows point inward (toward domain)?
  2. Does any domain class import infrastructure code?
  3. Can you test business logic without a database, network, or file system?
  4. Is each module describable in one sentence?
  5. If you delete an adapter, does domain code still compile?
  6. Are interfaces defined by the consumer (domain), not the provider (infrastructure)?
  7. Is there one clear entry point for each use case?

3.4 AI-Assisted Development

AI-generated code is increasingly common. This section addresses the specific quality risks of AI-assisted development and provides practices for managing them.

The Reality of AI-Generated Code (2025–2026 Data)

Research and industry data show consistent patterns:

FindingSource
AI-generated code has ~1.7x more bugs than human-written codeStack Overflow Developer Survey 2025; GitClear analysis
Code review time increased ~91% for AI-heavy PRsIndustry reports from large enterprises
Security vulnerabilities 1.5–2x more common in AI codeSecurity research studies 2024–2025
AI excels at boilerplate but struggles with domain-specific logicConsistent across multiple studies

This is not an argument against AI — it is an argument for structured processes around AI code. The V-model’s specification-first, verification-heavy approach is exactly what AI-assisted development needs.

Common Mistakes AI Agents Make

Logic and Correctness:

  • Plausible-but-wrong formulas (correct structure, wrong operator or constant)
  • Off-by-one errors in loops and boundary conditions
  • Hallucinated API methods that don’t exist in the actual library version
  • Stale knowledge about deprecated or renamed APIs
Bad — Plausible but wrong (off-by-one)
Java// AI generates this — looks reasonable, but wrong for inclusive ranges int countInRange(List<Integer> items, int min, int max) { return (int) items.stream() .filter(i -> i > min && i < max) // should be >= and <= .count(); }

Architectural Problems:

  • No awareness of existing architecture — creates new patterns instead of following established ones
  • Copy-paste across modules instead of reusing shared components
  • Over-abstraction (“AbstractStrategyFactoryBuilder”) for simple problems
  • Inconsistent style between generated segments

Testing Failures:

  • Tautological tests that recompute expected values using the same logic
  • Missing edge cases and boundary conditions
  • Over-mocking that disconnects tests from real behavior
  • Tests that pass for any implementation (assertion-free or structural-only)

Security:

  • Hardcoded credentials or API keys in generated examples
  • SQL injection via string concatenation instead of parameterized queries
  • Missing input validation at system boundaries

Best Practices for AI-Assisted Development

1. Specification First: Always have a detailed design before generating code. AI performs dramatically better with a clear specification than with a vague instruction. The V-model’s DRTDD approach (Design → Require → Test → Implement) is ideal.

2. Review Every Line: Treat AI output the same as a junior developer’s pull request. Key review areas:

AreaWhat to Check
LogicAre operators correct? Off-by-one? Edge cases handled?
Error handlingAll paths covered? No swallowed exceptions?
API usageDo the methods actually exist in this library version?
SecurityInput validated? No injection vectors? No hardcoded secrets?
ArchitectureFollows project patterns? No new unnecessary abstractions?
NamingConsistent with project vocabulary? Meaningful?

3. Incremental Generation: Generate one function or class at a time. Verify it before moving on. Do not generate an entire module in one shot.

4. Context Management: AI quality depends on context. Provide:

  • The detailed design artifact
  • Existing code patterns from the same module
  • The project’s coding standard
  • The target test framework and conventions

Structuring Code for AI Maintainability

Code that is easy for AI to understand is also easy for humans. The principles overlap:

  • One concept per file, under 300 lines
  • 2–3 directory levels maximum depth
  • Functions 5–20 lines with explicit types
  • No hidden state — all dependencies visible in the constructor
  • Descriptive names as the primary context signal — AI uses names to understand intent

V-Model Specific Considerations

Beyond writing quality code, the V-model imposes specific requirements on how source code is managed as an artifact.

Traceability Requirements

Every public code element must trace to a Detailed Design element. This is bidirectional:

  • Forward trace (design → code): every design element has a corresponding code implementation
  • Backward trace (code → design): every public code element maps to a design element

Orphan code (code with no design element) and orphan design (design with no code) are both compliance gaps.

Dead Code and Deactivated Code

Safety standards are strict about code that doesn’t execute:

  • Dead code (unreachable code) must be removed. In DO-178C, dead code requires a justification for why it exists and cannot be removed.
  • Deactivated code (code disabled by configuration or build flags) must be documented with: what it does, why it is deactivated, under what conditions it would be activated, and how it was verified.

No Gold Plating

Adding functionality beyond the design specification creates untraceable code. In safety-critical domains, untraceable code is a finding. The discipline is: implement exactly what the design specifies, nothing more.

Source-to-Object Code Traceability

At the highest assurance levels (DO-178C DAL A), you must demonstrate that the compiled object code corresponds to the source code. This means understanding compiler optimizations, ensuring no unintended code is introduced by the toolchain, and verifying that the object code does not contain functionality absent from the source.

Code Review Independence

Review independence scales with assurance level:

Assurance LevelReview Independence
Low (DAL D / ASIL A)Peer review (same team member)
Medium (DAL C / ASIL B)Independent review (different team member, not the author)
High (DAL A–B / ASIL C–D)Independent review with documented evidence, potentially by an independent verification team

Framework Integration

How source code integrates with the VModelWorkflow templates, schemas, and traceability system.

Input: Detailed Design Artifact

Source code takes the Detailed Design as its specification. The design artifact (defined in schemas/artifacts/detailed-design.schema.yaml) provides:

  • interfaces — the public API that the code must implement
  • behavior — the rules that the code must follow
  • error_handling — the error conditions the code must handle
  • internal_state — the state the code must manage
  • configuration — the configurable parameters

Traceability Links

Source code participates in two primary trace link types (defined in schemas/traceability/link-types.yaml):

Detail Design → implemented-by → Source Code
Source Code → verified-by → Unit Test

Artifact Envelope

Like all VModelWorkflow artifacts, source code is wrapped in an artifact envelope (defined in schemas/artifacts/artifact-envelope.schema.yaml). The envelope provides metadata: artifact ID, version, status (draft/review/approved), author, and content hash for staleness detection.

Review Record

Code reviews produce a review record artifact with findings, verdicts, and traceability to the code under review. The review checklist in Section 3.5 above maps directly to the review record template fields.
