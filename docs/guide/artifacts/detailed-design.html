Leaf-Scope Artifact

1. V-Model Context

A Detailed Design (DD) is the specification of a single leaf scope — the unit of code the team will implement and unit-test. It fixes the contracts, invariants, algorithms, state, and error behaviour of that leaf at a level of detail from which code and unit tests can both be produced without guessing.

Where DD sits in the scope tree

DD lives at leaf scopes only. Non-leaf scopes carry Requirements and Architecture; the tree recurses through Architecture decompositions until subdivision stops earning its keep — at that point the scope is a leaf, and the leaf owns a DD. The DD's parent_architecture link points to the Architecture that allocated work to this leaf; the DD's sibling TestSpec owns the unit tests that verify it.

Every leaf component in the parent Architecture's decomposition gets a DD — mandatory. The presence or absence of a [DEFER-DD: …] marker in upstream artifacts does not determine whether a DD is needed. Markers describe content gaps inside future DDs; they never gate the existence of a DD. If the parent Architecture allocates a leaf, that leaf has a DD by mandate, even if no upstream marker mentions it.

The DD does not re-derive what the parent Architecture already fixed. It receives the leaf's responsibilities, the interfaces it must honour with its siblings, and the ADRs it is governed by. Interface contracts are frozen at Architecture scope; the DD implements them, never refines them. The DD's Public Interface section names the actual functions that satisfy the architectural contract and references the interface by ID; the architectural interface entry remains the single source of truth for preconditions, postconditions, invariants, and errors that callers can rely on. Its job is to say precisely what the code inside this leaf must be true of — and nothing else.

Why the framework has it

Every artifact in the framework earns its place by producing something the rest of the system cannot. At the leaf, the DD's contribution is making the gap between specification and code closable without a judgement call: a reader with the DD in hand can implement the code, and a different reader with the DD in hand can write the tests, and the two outputs will agree.

The framework holds DDs to three success criteria, all of them practical:

  • Junior-implementable. A developer unfamiliar with the codebase can produce a correct implementation from the DD alone.
  • Language-portable. An equivalent implementation could be produced in Java, Python, or Go from the same DD. The design constrains behaviour, not language idioms.
  • Test-derivable. A test engineer can write unit tests from the DD before the code exists. If the DD does not let the tests be written, it is incomplete.

These are the same test applied three ways. A DD that passes all three closes the spec-to-code gap; a DD that fails any one of them has either over-specified (paraphrased code) or under-specified (handed the judgement to the implementer).

What a DD is for

A DD carries the information that the code cannot convey on its own. The reader of code sees what the implementation does; the reader of a DD sees what must be true of any valid implementation, why those things must be true, and what breaks if they are not. Rationale, constraints, contracts, valid state spaces, invariants, and error semantics all belong in the DD because they are not recoverable from the code.

What a DD is not

  • Not a code paraphrase. A DD that walks through the implementation line-by-line has collapsed the design box to a point — it does not specify, it narrates.
  • Not a test substitute. The sibling TestSpec owns unit tests; the DD describes the contract the tests verify. Duplicating test cases inside the DD creates two places for one fact and drifts under change.
  • Not a retrofit invention. When a DD is recovered from legacy code, the observable contract and structure can be recorded; the why — rationale, considered alternatives, original intent — cannot be inferred from code. The framework records what is recoverable and marks the rest recovery_status: unknown.
  • Not Architecture. Decisions about which components exist and how they compose belong in the Architecture above. The DD lives inside one of them.

2. Best Practices

2.1 Document the why, not the what

The first discipline of a DD is knowing what belongs in it and what does not. The test is whether the information is recoverable from the code. If a skilled reader could derive a fact by reading the code alone, that fact does not belong in the DD; putting it there creates two places for one fact, which will drift.

Two rules that both apply

A DD must satisfy two rules simultaneously. They look contradictory but operate at different levels.

Rule 1 — Do not duplicate the code. The DD must not paraphrase implementation logic. If a skilled developer could derive a fact from the code, that fact is not in the DD.

Rule 2 — Be specific enough to implement and test. A junior developer must be able to produce a working implementation from the DD. A test engineer must be able to derive tests from the DD without seeing the code.

These coexist because "specific" and "duplicating code" are different things. The DD specifies what must be true of any valid implementation; the code specifies which implementation was chosen.

The test of specificity

A correctly written DD has all three properties at once:

  1. Two developers reading it would produce different code (different names, different loop styles, possibly different internal structures within the contract).
  2. Both implementations would pass the same test suite.
  3. A test engineer could write that test suite without ever seeing the code.

If property 3 fails, the DD is too vague. If property 1 fails, the DD is paraphrasing code. If property 2 fails, the contract is under-specified — two valid implementations cannot both be right.

Mental model: the box

Think of the DD as defining a box in implementation space. The walls are the contract — what is required and what is forbidden. Inside the box are many valid implementations. Outside the box are bugs.

  • Too-vague DD: no walls. Any implementation passes, including wrong ones.
  • Code-paraphrase DD: box collapsed to a point. Only one implementation fits, and the DD is just code in another language.
  • Right-level DD: walls placed where correctness, performance, or operational boundaries demand them; the interior is left open.

2.2 The 7-section artifact shape

Every DD has the same seven sections. Some will be short — a stateless leaf has no State section worth writing; a leaf with only trivial per-call behaviour has a spare Algorithms section. Short is fine; missing is not. The fixed shape gives reviewers a predictable scan path and makes absence legible: a blank State section is a deliberate assertion that the leaf is stateless, which is reviewable content.

SectionPurpose
1. MetadataID, scope, parent_architecture, governing_adrs. parent_architecture is the single structural anchor for the leaf.
2. OverviewTwo to three paragraphs: what this leaf is, what slice of its parent it realises, the approach at one level of abstraction above the code.
3. Public InterfaceEvery externally visible function, class, or type, with full contracts (§2.3).
4. Data StructuresInternal data: fields, invariants, ownership, lifetime (§2.4).
5. AlgorithmsPseudocode for the trickiest, prose for the clearer ones, decision tables for complex conditionals (§2.5).
6. StateState machines, persistent state, side effects (§2.6). Omit the body if the leaf is stateless; state the absence explicitly.
7. Error HandlingWhat can fail, how it is detected, how it propagates, what the caller receives (§2.7).

Rationale for design decisions within the DD is captured inline at the site of the decision, or — when the decision is load-bearing enough to need its own record — as a cross-cutting ADR linked via governing_adrs. See §2.8.

2.3 Contract every public function

Every entry in the Public Interface section is a contract between caller and implementation, not a signature. A contract has enough content that the caller knows its obligations, the implementation knows what it must guarantee, and the test engineer knows what to assert. A signature alone — double calculateDiscount(Order order) — is not a contract. It is a name for one.

What a complete function contract contains

Not every element applies to every function, but completeness means having consciously decided each:

  1. Signature — name, parameter types, return type.
  2. Preconditions — value ranges, state conditions, relationship constraints, units of measurement. What the caller must guarantee.
  3. Postconditions — return value range, state changes, idempotency, old-value relationships. What the implementation guarantees.
  4. Invariants — properties that hold before and after every public call.
  5. Error semantics — what happens on invalid input or inability to proceed: exception, error result, sentinel, or assertion.
  6. Nullability — explicit for every parameter and return.
  7. Side effects — what state outside the function's arguments and return may change (I/O, mutation of shared state, observable clock or environment reads).
  8. Thread-safety — immutable, thread-safe, conditionally thread-safe, thread-compatible, or thread-hostile. The category is an interface property, not an implementation detail.
  9. Complexity notes — when time or space bounds are part of the contract (a hot loop, a UI-blocking call, a bounded retry budget).

Design by Contract, plainly

The shape above is Design by Contract (Meyer 1992; Hoare 1969 for the underlying logic). Every function is a Hoare triple: if the precondition holds before the call, the postcondition holds after. Liskov & Wing (1994) give the rule for overrides: preconditions may only weaken in subtypes, postconditions may only strengthen, invariants must be preserved. Contracts that violate the Liskov rule are design bugs that become code bugs.

Unit of measure in the contract

A parameter typed double altitude with no unit is not a contract; it is an invitation to the next Mars Climate Orbiter. Name the unit in the contract text if it is not obvious from the parameter's type, and prefer a typed wrapper (Metres, Millis) when the type system can carry the distinction.

Defensive programming vs. DbC

These are philosophically opposed. DbC assigns each consistency condition to one party: either it is the caller's precondition (caller's bug if violated) or the implementation's obligation (implementation's bug if violated). Defensive programming validates everything everywhere, which blurs responsibility and multiplies checks that never fire under a correct caller. Practical synthesis: use DbC semantics in the DD; place defensive checks only at trust boundaries where an untrusted caller exists (external I/O, IPC, process edges).

How DD Public Interface relates to the parent Architecture interfaces

When a leaf component implements an interface defined in the parent Architecture, the DD's Public Interface section names the implementing function; it does not re-derive the contract. The architectural interface entry (carrying preconditions, postconditions, invariants, errors, quality attributes, authn/authz) is the single source of truth; the DD references it by ID and adds the leaf-internal detail the architecture cannot carry: the actual function name and parameter shape, internal-state preconditions that further bound when the function may legitimately be called, ownership and lifetime of returned objects, thread-safety category, and complexity notes.

The discipline matters in two directions. Upward: the DD must not strengthen a precondition or weaken a postcondition that the architecture promised callers; doing so violates the architectural contract and breaks every caller that read the architecture entry as truth. Downward: the DD may carry additional preconditions that are internal to the leaf (e.g., "this function must be called only after init(); calling it before is a programming error"). These are leaf-internal contract clauses; they are not part of the architectural contract and never appear there.

Concretely: a DD entry that implements IValidate from a parent Architecture says "Validate(set, graph, mode) implements IValidate per ARCH-IF-IValidate; thread-safety: thread-safe; internal precondition: schema cache must have been initialised by a prior loadSchemas() call" — not a re-listing of IValidate's preconditions, postconditions, and errors. The architectural entry stays load-bearing; the DD adds only the implementing-function contract.

Marker syntax inside the DD

When authoring the DD reveals a downstream gap, mark it with one of two forms.

  • [DEFER-ADR: <decision>] — a cross-cutting decision-shaped item with at least two named alternatives that the DD touches but cannot resolve. The marker reserves the ADR slot; the ADR is authored when the decision is taken.
  • Inline <TBD> — a parametric gap inside the DD itself (a numeric threshold not yet calibrated, an enumeration not yet enumerated). The DD ships when the parameter is filled; it does not get a defer marker.

A DD does not carry [DEFER-DD: …] markers about itself — the DD is the artifact those markers anticipated. Sibling DDs at the same scope are independent artifacts and are not in scope for this DD's defer markers either.

2.4 Specify data structures by invariant, not by field layout

A data-structure specification states what must be true of the data, not how the fields happen to be laid out in the chosen language. Field layout is code; invariants are design.

Every internal data structure documented in a DD has:

  • Fields — name, type, purpose. Types are expressed at one level of abstraction above the language (a "non-negative integer," not uint32_t, unless the bit-width is part of the contract).
  • Invariants — properties that hold whenever the structure is observed by any other code. The invariant is what the tests check, and what a refactor must preserve.
  • Ownership and lifetime — who constructs the structure, who may mutate it, when it is released. "Shared mutable" is a contract, not an accident; it must be named.
  • Returned-object semantics — when the structure is returned from a public function, whether the caller owns a copy, observes a live view, or receives a read-only reference. Returning a mutable object without stating this leaks ownership ambiguity into every caller.

Shared mutable state is always a contract

For any field that can be observed or mutated by more than one caller or thread, the DD states: who may read, who may write, what lock or sequencing primitive guards the field, and what happens-before relationship is established. An undocumented shared field is a design defect — the concurrency behaviour becomes whatever the first implementer happens to type, and reviewers cannot tell correctness from luck.

2.5 Algorithms: specify the result; select the algorithm only when it matters

Algorithm specification has two failure modes. Too abstract ("process the data") is untestable. Too concrete (pseudocode copied from the implementation) collapses the design box to a point and teaches the test engineer nothing the code would not already show.

The right level distinguishes behavioural requirement (what property the output satisfies) from algorithm selection (which specific algorithm is used). Most of the time the DD states the requirement and leaves the algorithm to the implementer. Sometimes the algorithm itself is the specification — when determinism, worst-case bounds, or operational constraints make a specific approach contractual. When that is the case, say so and say why.

Too vague — fails Rule 2
DesignThe function sorts the input list.

A developer does not know: ascending or descending, stable, null handling, performance envelope. A test engineer cannot derive meaningful tests.

Code paraphrase — fails Rule 1
DesignDeclare a result list, iterate from 0 to length-1, compare elements using natural ordering, swap when out of order, return the result.

This tells the developer which implementation to write — and only one. The test engineer learns nothing the code would not already show.

Right level — passes both rules
DesignReturns a list containing exactly the elements of the input, in non-descending order according to the natural ordering of the element type. Equal elements preserve their relative order from the input (stability is required because downstream processing depends on insertion order as a tiebreaker). Performance: O(n log n) worst case is required to meet the request-latency budget at the calling layer. Null inputs are rejected with IllegalArgumentException.

Two developers might produce mergesort and Timsort with different variable names. Both implementations pass the same test suite. A test engineer derives boundary, stability, performance, and null-rejection tests without seeing any code.

Specification pattern selection

The right notation depends on the shape of the behaviour. Natural language prose, pseudocode, decision tables, property specifications, and state machines are all tools — pick the one that makes the behaviour reviewable.

Behaviour typeRecommended form
Pure function, transformationPre/postcondition + property form ("for all valid inputs…")
Rule-based logic (many conditions → action)Decision table. Completeness is mechanically checkable by counting rules.
Mode-dependent, event-drivenState machine (see §2.6).
Multi-step ordered protocolScenario-based description or sequence diagram.
Complex input-domain partitioningTabular specification.

A postcondition describes the result, not the steps. "Shall iterate through the list and compare adjacent elements" is pseudocode misfiled as a postcondition; the correct postcondition names the property ("the returned list is ordered and is a permutation of the input"). Two properties, not one — omitting the permutation half is a classic specification error that lets return [] pass.

2.6 State: transitions, guards, and undefined-event behaviour

Specify dynamic behaviour when the leaf has multiple operational modes, history-dependent behaviour, concurrency, or timing constraints. Most leaves do not — say so in one line and move on. When the leaf does have state worth modelling, the State section carries:

  • State inventory with per-state invariants.
  • Transition table — source, event, guard condition, action, target state.
  • Initial state and terminal states.
  • Undefined-event handling — what happens when an event arrives in a state where it is not defined. Ignore, log, fault, or raise. Silence here is dangerous; implementers will pick a default, and the picks will differ.
  • Entry and exit actions when a state has them.

A state machine that does not fit on a page or two is almost certainly too large for reviewers to reason about. Decompose — either into hierarchical states or into sub-leaves — rather than letting a single giant diagram stand.

Thread safety is an interface property

When the leaf is used across threads, the DD names the thread-safety category per the Goetz taxonomy — immutable, thread-safe, conditionally thread-safe, thread-compatible, thread-hostile — and documents per shared field which lock guards it and what happens-before relationship is established. An "implicit" concurrency assumption, like returning a mutable object with no ownership statement, is a design defect; the concurrency behaviour in production becomes whatever the first caller does.

Timing constraints are testable statements

A timing constraint must specify: the type (deadline, period, response time), the numeric bound and its unit, the conditions under which it must hold (worst-case, nominal), the upstream requirement it derives from, and how it will be verified (static analysis, instrumented measurement, or controlled experiment). "Shall execute quickly" is not a timing constraint; it is a wish.

2.7 Errors are a design concern

Error behaviour is part of the contract. A DD that specifies only the success path leaves every error decision to the implementer — including which errors count, how they propagate, what state the leaf is in afterwards, and what the caller sees. The results diverge across implementations and cannot be tested against a spec that does not exist.

Six questions the DD must answer

  1. What errors can this leaf encounter or produce?
  2. How is each error detected?
  3. How does each error propagate, or where is it contained?
  4. What is the recovery strategy for each class?
  5. What state is the leaf left in after the error?
  6. What does the caller receive — exception, error result, sentinel, degraded value?

The error-handling matrix

A structured way to answer the six questions in one place. Each row is one error class.

ErrorDetectionContainmentRecoveryCaller receives

The rows of the matrix correspond directly to robustness test cases in the sibling TestSpec. A DD missing the matrix has no way to trace whether an error has been tested; a DD with the matrix produces the TestSpec rows almost by inspection.

Checked vs. unchecked is a contract decision, not a coding preference

A checked exception specifies that the caller must handle the condition — the handling is part of the interface. An unchecked exception indicates a programming error: the caller has a bug, and no recovery is expected. Picking one over the other is a design-level statement about who is responsible. The same logic applies to error-result types (Result<T, E>): the presence of E in the type is a specification of caller obligation.

Fail-fast, recovery, and degraded operation

For each error class, the DD states which strategy applies:

  • Fail fast — detect at the earliest boundary and surface explicitly. "Surface" means something concrete: raise, return an error, signal a safe state. Not "crash."
  • Retry — with a bounded budget, a backoff policy, and an idempotency assumption that holds. Unbounded retry is a defect.
  • Fallback — to what, under what conditions, with what quality degradation visible to the caller.
  • Compensating action — undo partial work; leave the system in a consistent state.
  • Propagate — translate at the boundary into a domain-appropriate form. A database exception should not leak to a business-logic caller in its raw form.

2.8 Capture rationale at decision time

A DD that records decisions without recording why they were made cannot be maintained — a future reader cannot tell a decision that is forced by an external constraint from one that is arbitrary. They will reverse the arbitrary ones (and sometimes the forced ones), and they will preserve the arbitrary ones unnecessarily. Both outcomes are bugs disguised as refactors.

Inline rationale for ordinary decisions

Most rationale lives inline at the site of the decision: a parenthetical explanation next to a contract element, a short note next to an unusual postcondition, a comment after an error-class recovery strategy. The test is whether a reader without the decision's context can understand why the DD says what it does. If the answer is no, add one or two sentences.

ADR for load-bearing decisions

When a decision meets the ADR threshold — load-bearing, multiple real options considered, contingent on assumptions that may change — the decision belongs in a cross-cutting ADR, and the DD references it via governing_adrs. Unit-level ADRs are appropriate (the /specs/adrs/ directory is flat across scopes); the threshold does not scale with scope depth.

Constraint categories

Design decisions are usually forced by one or more of four constraint kinds. Naming the kind makes rationale reviewable.

  • External — a standard, a platform limitation, a certification requirement, a contractual SLA.
  • Architectural — a choice made at a higher scope. "Single-threaded event loop; no blocking calls in this leaf."
  • Resource — a timing, memory, or capacity budget. "Called on the request hot path; must complete in under 2 ms."
  • Temporal — correct at the time but expected to be lifted. "Preserves the pre-v2 response shape; relaxes when v1 clients are retired."

3. Anti-Patterns

Failure modes seen at the DD layer. Each item is a short failure statement plus a concrete tell a reviewer can look for.

Interface and contract

  1. Undefined-range precondition. A parameter has no stated valid range, and the caller has no way to know which values are in contract. Tell: the contract mentions the parameter's type but not the set of values it accepts.
  2. Implicit unit. A numeric parameter carries no stated unit of measure. Tell: a double altitude, a long duration, a float temperature with no unit named nearby.
  3. Implementation-leaking interface. The public contract names a concrete collection type, exposes internal structure, or commits to a specific storage layout. Tell: the interface changes when a refactor changes the internal representation.
  4. Silent null return. A function can return null (or the equivalent) and the contract does not say so. Tell: the return type is nullable in the language but the contract does not mention the null case, leaving each caller to guess.
  5. Algorithmic postcondition. The postcondition describes the steps the implementation takes rather than the property of the result. Tell: the postcondition begins "shall iterate…" or "shall compute…" instead of "the returned value shall satisfy…."

Error handling

  1. No error strategy. The DD specifies only normal operation; each implementer decides errors independently. Tell: there is no error-handling matrix and no prose about what can fail.
  2. Exception swallowing. The contract specifies an error class, and the recovery strategy is "log and continue" with no domain action. Tell: the caller receives a success return even when the operation materially did not succeed.
  3. Exception tunneling. A low-layer infrastructure exception type appears in the contract of a high-layer domain function. Tell: the business-logic function declares throws SQLException.

State and concurrency

  1. Designing for races. A compound check-then-act operation is described as a sequence of steps without atomicity stated. Tell: the DD describes "check whether X, then update X" with no note on whether both halves are atomic, no lock, no CAS.
  2. State explosion. A single state machine with dozens of states and hundreds of transitions. Tell: the diagram does not fit on two pages; reviewers cannot reason about which transitions are reachable.
  3. Missing cancellation. A long-running operation has no specified cancellation policy — how it is signalled, whether it is cooperative, what state the operation is left in on cancellation. Tell: the contract specifies start and completion but not "can be asked to stop."

AI-era failure modes

  1. LLM confident invention. An AI-generated DD produces hallucinated constraints with false precision — timing bounds, buffer sizes, concurrency semantics — that trace to no upstream requirement and no operational data. Tell: the DD reads as crisp and complete; the constraints do not appear anywhere else in the spec tree.
  2. Code paraphrase / laundered code. A retrofit DD sophisticatedly describes what the current code does, without distinguishing intent from incident. The result is a post-hoc paraphrase presented as a design document. Tell: every contract clause could be derived from reading the implementation; nothing in the DD says why the contract is what it is.
  3. Test-as-spec inversion. The DD is written to match the assertions in the existing tests, rather than the tests being derived from the DD. Tell: the DD's error cases enumerate exactly what the test file asserts, neither more nor less, because one was back-derived from the other.
  4. Happy-path bias. The Public Interface and Algorithms sections are thorough; the Error Handling section is a one-liner. Tell: an LLM-authored DD with a full error-handling matrix for input validation and nothing about downstream failures, timeouts, or partial-success paths.
  5. Post-hoc DD. The DD is written after the code, to catch the documentation up. The option space collapsed before the DD existed, and the DD justifies the current code rather than specifying what any valid code must satisfy. Tell: the rationale for non-obvious choices reads as a defence of the current implementation rather than a record of alternatives considered.

Architecture-DD boundary failures

  1. Restating the parent architecture's interface contract. The DD's Public Interface section reproduces the preconditions, postconditions, invariants, and errors that already live on the parent Architecture's interface entry. Two places for one fact — they will drift, and the architectural contract loses its single-source-of-truth status. Tell: a DD entry whose Postconditions paragraph reads identically to the parent Architecture's interface entry; a DD that carries quality-attribute budgets that already live on the architectural interface contract. The DD references the architecture interface by ID, names the implementing function, and adds only leaf-internal detail (internal-state preconditions, ownership, complexity); it does not re-derive the contract.
  2. Treating [DEFER-DD: …] markers as a TODO list of DDs. An author concludes that only the leaves named in upstream DEFER-DD markers need DDs, leaving other leaves un-authored. Every leaf in the parent Architecture's decomposition gets a DD by mandate; markers describe gaps inside future DDs, never DD existence. Tell: a scope where the parent Architecture allocates five children but only three DDs exist, paired with three matching DEFER-DD markers from upstream — the missing two are leaves whose DDs are still mandatory.

4. Examples

4.1 Good — a leaf with all seven sections

A leaf inside the app/jobs scope: the service that claims and releases work items in the Postgres-backed job queue. Paired with the ADR example in ADR-017.

It is one of two leaves in `app/jobs`; the sibling leaf is the enqueue service. What this example gets right: every public function names the architectural interface entry it implements (ARCH-IF-IDequeue) and adds only leaf-internal detail — internal-state preconditions, side effects, thread-safety, complexity — instead of restating the architectural DbC clauses (preconditions, postconditions, errors, invariants are frozen at ARCH-app-jobs and live there as the single source of truth); the Claim data structure is specified by invariant rather than by language-idiomatic fields; the algorithm is the SQL statement because that is the contract (ADR-017 fixes the access pattern), not an implementation detail; the State section is short because there is none, and the shortness is an explicit assertion; the error matrix rows correspond one-to-one with robustness test cases the sibling TestSpec will carry.

4.2 Good — a spare DD for a simple leaf

Not every leaf needs rich content in every section. This one — an application configuration loader — is straightforward: read YAML, expose typed accessors, be immutable after load. The 7-section shape is still honoured; several sections are a few lines because there is little to say.

detailed_design.md — simple leaf--- id: DD-app-config-loader scope_tags: [app/config] parent_architecture: ARCH-app-config --- # Configuration Loader ## Overview Reads application configuration from YAML files on disk at startup and provides typed access to configuration values for the rest of the application. This leaf is the single entry point for file-based configuration; no other leaf reads config files directly. ## Public Interface ### load(path: FilePath): ConfigRegistry - Preconditions: `path` is absolute; file size is at most 1 MB. - Postconditions: returns a ConfigRegistry populated from the file's contents, with required keys present and defaults applied for optional keys. The returned registry is immutable. - Errors: on any load failure, returns a ConfigLoadResult holding the warnings/errors; no exception is thrown during normal loading. Paths that exceed the size cap raise IllegalArgumentException. ### ConfigRegistry.getString / getInt / getBoolean / … [… same shape, typed per value …] ## Data Structures ### ConfigRegistry (immutable) | Field | Type | Invariant | |----------|-------------------------|--------------------------------------------| | entries | Map<String, Value>| Populated at construction; never modified. | Ownership: returned to the caller of `load`; shared freely (immutable). Case sensitivity: keys are case-sensitive. ## Algorithms Parsing uses the host YAML library; there is no custom parse logic in this leaf. Required-key validation iterates the schema of expected keys and records any missing ones as errors. ## State Stateless beyond the immutable registry it returns. No hot reload; reconfiguration requires process restart. ## Error Handling | Error | Detection | Recovery | Caller receives | |-----------------------|-------------------|-------------------------------|--------------------------------| | File not found | Open failure | None | ConfigLoadResult with error | | Parse failure | YAML parser | None | ConfigLoadResult with error | | Missing required key | Schema check | None | ConfigLoadResult with error | | File size > 1 MB | Precondition check| Reject at boundary | IllegalArgumentException | ## Constraints - Config is loaded once, at startup. No hot reload. - No secrets in config files; secrets use environment variables loaded by a separate leaf.

Short does not mean under-specified. Every contract is complete; invariants are stated; the error table covers every failure a caller needs to anticipate. A junior developer could implement this from the DD, and a test engineer could derive the unit tests from it without ever reading the code.

4.3 Retrofit DD — fabricated vs. honest

The same team later retrofits an unfamiliar legacy leaf: a user-session token validator whose author is gone and whose commit history says only "initial import." The code shows HMAC-SHA256 signing with a 30-minute lifetime. There are no design docs, no interview subjects, and no preserved rationale.

A retrofit skill with insufficient guardrails produces this — bad:

detailed_design.md — fabricated retrofit (wrong)--- id: DD-legacy-session-validator scope_tags: [legacy/auth] recovery_status: overview: reconstructed ← Illegal on human-only fields. rationale: reconstructed ← Invented, not recovered. --- ## Overview After evaluating token formats in the 2019 platform refresh, the team selected HMAC-SHA256 with a 30-minute lifetime to balance security ← Committee-style prose; against session ergonomics for the sales operations workflow. no preserved conversation backs any of this. ## Rationale The 30-minute lifetime matches the measured median session length for the sales team, minimising re-authentication while capping the blast ← Plausible-sounding, radius of token theft. HMAC-SHA256 was chosen over JWT to avoid back-derived from what the algorithm-confusion class of attacks and to keep the verification the code does. path simple enough to audit inline.

Tells of fabrication: committee-grade prose despite no accessible deciders, no archive, no preserved conversation; rationale that could have been written in either direction (the same prose would defend a 60-minute lifetime or HS512); recovery_status: reconstructed on human-only fields — an illegal state the retrofit skill must refuse to emit.

What the retrofit should produce — good:

detailed_design.md — honest retrofit--- id: DD-legacy-session-validator scope_tags: [legacy/auth] recovery_status: overview: verified ← Observed from code. rationale: unknown ← Human-only, not recovered. Never "reconstructed". --- ## Overview Validates user session tokens. Tokens are HMAC-SHA256 over a canonical payload (see src/auth/token.py); they expire 30 minutes after the `iat` claim. Observed from code on 2026-04-19. ## Public Interface ### validate(token: String): SessionClaims - Preconditions: observed from code — none enforced on input shape; malformed tokens fail verification. - Postconditions: returns SessionClaims on valid token; otherwise raises TokenInvalid. No partial results. - Errors: TokenInvalid (signature mismatch, expired `iat`+30m, unknown issuer). - Thread-safety: observed as thread-safe in production use; no shared mutable state in the validator class. ## Rationale unknown — no preserved design notes, no accessible deciders as of 2026-04-19. The 30-minute lifetime and HMAC-SHA256 choice are observed in code and in deployment config; the forces that drove either decision are not recoverable. ## Constraints (observable) - Signing key lives in the deployment-time secret store; rotation cadence is not specified in any surviving document. - No refresh-token path exists; re-authentication is required at expiry.

Honest, useful, forward-compatible. It records what the code reveals, admits what has been lost, and does not launder the current design as inevitable. A forward change that needs the missing rationale knows it must surface it rather than assume it.

5. Quality Bar

Yes/No checklist, grouped by concern. A DD passes when every applicable item is answered Yes. The meta-gate at the bottom — the Spec Ambiguity Test — overrides every card above: a DD that ticks every box but cannot guide a junior engineer to the same implementation fails.

Structure and completeness

  • Are all seven sections present, each with either substantive content or an explicit assertion of absence ("this leaf is stateless")?
  • Does Metadata include scope_tags, parent_architecture, and governing_adrs where applicable? (parent_architecture is the single structural anchor for the leaf.)
  • Does the Overview say what slice of the parent Architecture this leaf realises, at one level of abstraction above the code?

Contracts on every public function

  • Does every public function have preconditions, postconditions, error semantics, nullability, side effects, and thread-safety stated?
  • Are numeric parameters either typed with a unit-carrying wrapper or have the unit named in the contract?
  • Are postconditions expressed as properties of the result, not as steps taken to produce it?

Data and invariants

  • Does every internal data structure list fields, invariants, ownership, and lifetime?
  • For any mutable state shared across threads or callers: is the locking, happens-before, and per-field ownership stated explicitly?
  • For any function that returns a mutable object: is the ownership transfer or sharing contract stated?

Algorithms, state, control flow

  • Where the algorithm itself is contractual (determinism, worst-case bounds, operational constraint), is it specified and is the reason it is contractual named?
  • Where it is not contractual, does the DD state the result property and leave the algorithm to the implementer?
  • If a state machine is present: are states, transitions, guards, and undefined-event behaviour named? Does it fit on two pages?

Error handling

  • Does the DD answer all six questions for each error class (what, detection, containment, recovery, end state, caller receives)?
  • Is there an error-handling matrix (or equivalent structure) that each row corresponds to a testable robustness case?
  • For every error class: is the strategy named (fail-fast, retry with bound, fallback, compensate, propagate)?

Rationale and traceability

  • Does every non-obvious decision carry inline rationale (a phrase or sentence naming the forcing constraint)?
  • Do load-bearing decisions have an ADR linked via governing_adrs?
  • Does parent_architecture name the Architecture artifact this leaf realises a slice of?

Retrofit honesty

  • If this DD was recovered from existing code: is recovery_status set correctly per field kind — code-derivable fields verified/reconstructed/unknown, human-only fields only verified/unknown?
  • Are all fields free of AI-fabricated content — no confident-sounding rationale for conversations that did not happen, no back-derived intent?
  • Is the observable-vs-inferred distinction clear ("observed from code" passages are marked as such)?

Spec Ambiguity Test — meta-gate

  • Could a junior engineer, reading only this DD, produce a correct implementation without guessing?
  • Could a test engineer, reading only this DD, write the unit-test suite without seeing the code?
  • Would an equivalent implementation in a different language (e.g., Java → Python → Go) satisfy the same DD?

If any answer is No, the DD under-specifies its leaf. Revise before accepting. This test overrides every card above: a DD that passes all of them but fails this one is not done.
