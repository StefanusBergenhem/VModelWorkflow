Level 3 — Design

What is Detailed Design?

Detailed design is the specification of individual software units — their interfaces, behavior, error handling, and internal structure — at sufficient detail that an engineer can implement them without further information and a tester can derive test cases without reading the code.

A detailed design document answers these questions for each unit:

  • What does this unit do? — its behavioral contract, inputs, outputs, and state changes
  • What does this unit promise to callers? — its interface specification with preconditions and postconditions
  • What design decisions shaped it? — the rationale, constraints, and rejected alternatives
  • What can go wrong? — error conditions, fault containment, and recovery strategy

The Fundamental Principle

A detailed design is not a description of what the code does. The code already says what it does. A detailed design captures what the code cannot say: intent, decisions, constraints, and contracts that enable independent verification. If your design document reads like prose-formatted source code, it has failed its purpose.

“The implementation documentation should never describe what happens, that does already the code for you. It should describe why it is implemented in the way it is.” — Eclipse iceoryx SWE.3 documentation guidelines

The Test Derivation Litmus Test

A detailed design is complete when an engineer who has not seen the source code can derive a meaningful set of test cases from it. If the test engineer must look at the implementation to understand what to test, the design is incomplete.

Detailed Design in the V-Model

Detailed design sits on the left side of the V at Level 3, below the Software Architecture and directly opposite the Unit Tests on the right side. It is the lowest level of specification before code.

Position in the V

SW Requirements → Architecture → Detailed Design → Source Code

The detailed design takes the software architecture as input and decomposes each architectural component into individually implementable and testable units. Each unit’s design becomes the specification against which unit tests are derived and the implementation is verified.

The Critical Principle: Design Before Code

The V-model requires that design precedes implementation. This is not bureaucracy — it is the mechanism that makes independent verification possible. If tests are derived from the design (not from the code), they can catch implementation errors. If the design is written after the code, tests derived from it will mirror the code’s behavior — including its bugs.

“If you write the detailed design after documenting your code, the point of the unit test is lost.” — UL Solutions, ASPICE SWE.3 Insights

Producing Quality Detailed Designs

This section covers the craft of writing effective detailed designs. It is organized by knowledge domain: design fundamentals, interface specification, behavioral specification, rationale, error handling, dynamic behavior, scaling strategies, and AI-assisted design. Each principle includes concrete examples.

3.1 Design Documentation Fundamentals

Before writing any design, you need to understand what makes a design document valuable. The answer is deceptively simple: a design document must contain information that the code cannot provide.

What Design Documentation Is For

Parnas and Clements established the definitive framework in their 1986 paper “A Rational Design Process: How and Why to Fake It.” Their key insight: documentation is not a record of what happened during development — it is a structured presentation of what is known about the system. Even when development is iterative and exploratory, the documentation should be written as if a rational process was followed. — Parnas & Clements, IEEE Trans. SE, 12(2), 1986

Their model requires that there be one, and only one, place for every fact in the design. Redundancy creates inconsistency under change. Every design fact belongs to exactly one document section.

What Design Adds That Code Cannot

Jack Reeves argued in 1992 that “source code is the real software design” — but explicitly warned against reading this as “no documentation needed.” His framework clarifies what documentation must add: everything that a skilled reader cannot derive from reading the code. — Reeves, “What Is Software Design?”, C++ Journal, 1992

Document This (code cannot provide)Do NOT Document This (code already shows)
Why this algorithm was chosen over alternativesThe step-by-step implementation logic
What external constraint forced this data structureVariable names and their types
Valid state space of parameters (ranges, invariants)Control flow that is clear from structure
What happens when preconditions are violatedLanguage-specific syntax
Expected performance envelope and whyInformation the type system already captures
What the module hides and why that boundary was drawnWhat the code does line-by-line

Information Hiding: What a Module Must Document

Parnas’s 1972 paper “On the Criteria To Be Used in Decomposing Systems into Modules” established that every module is characterized by a design decision it hides from all others. The design document for a module must describe: (1) what the module hides, (2) what the module reveals through its interface, and (3) why that particular boundary was drawn. — Parnas, CACM, 15(12), 1972

IEEE 1016: The Standard for Software Design Descriptions

IEEE Std 1016-2009 defines what a Software Design Description (SDD) must contain. It is applicable to both high-level and detailed design, and explicitly covers reverse engineering scenarios. The standard does not prescribe specific methodologies but establishes that an SDD must communicate design information to key stakeholders. — IEEE Std 1016-2009

3.2 Interface Specification

Interface specification is the core of detailed design. ASPICE SWE.3 BP.2 states it directly: “without this information, proper testing of the interfaces in the unit test is impossible.” Every V-model standard requires that unit interfaces be defined with enough precision to derive tests from them.

Design by Contract

Bertrand Meyer formalized Design by Contract (DbC) in the Eiffel language. Every function interface is a contract between caller (client) and implementation (supplier), structured as a Hoare triple: {P} S {Q} — if precondition P holds before operation S, then postcondition Q holds after. — Hoare, CACM, 12(10), 1969; Meyer, IEEE Computer, 25(10), 1992

Contract ElementMeaningUnit Test Role
PreconditionWhat the caller must guarantee before the callTest setup — arrange phase establishes state satisfying pre
PostconditionWhat the supplier guarantees on returnTest assertion — assert phase verifies post holds
InvariantWhat must hold before and after every public callAssertion after every mutating call

What a Complete Interface Contract Contains

Not every element applies to every function, but completeness means having consciously decided each:

  1. Signature — name, parameter types, return type (the syntactic minimum)
  2. Preconditions — value ranges, state conditions, relationship constraints, units of measurement
  3. Postconditions — return value range, state changes, idempotency, old-value relationships
  4. Invariants — properties that hold across all public calls
  5. Error semantics — what happens on invalid input: exception, error code, assertion, or undefined?
  6. Nullability — explicit @NonNull / @Nullable for every parameter and return type
  7. Thread-safety guarantee — immutable, thread-safe, conditionally thread-safe, or not thread-safe
Bad — Signature only, no contract
Java/** * Calculates the discount for an order. */ double calculateDiscount(Order order);
Good — Full interface contract
Java/** * Calculates the percentage discount for an order based on customer tier. * * @param order the order to evaluate; must not be null; must have * at least one line item; total must be > 0 * @return discount as a decimal in [0.0, 0.5]; 0.0 means no discount * @throws IllegalArgumentException if order is null or has no line items * @throws IllegalStateException if customer tier is not set on the order * * Thread-safety: this method is stateless and thread-safe. * Derived from: SWREQ-PRICING-012 */ @NonNull double calculateDiscount(@NonNull Order order);

Defensive Programming vs. Design by Contract

These are philosophically opposed. DbC assigns each consistency condition to exactly one party: either it is the caller’s precondition (caller’s bug if violated) or the supplier’s obligation (supplier’s bug if violated). Defensive programming validates everything everywhere, blurring responsibility. — Meyer, IEEE Computer, 1992; ScienceDirect DbC overview

Practical synthesis for safety-critical systems: Use DbC semantics in the interface specification. Apply defensive checks at trust boundaries (hardware interfaces, external inputs, IPC). Use assertions for internal module interfaces. — Quantum Leaps, Embedded DbC

The Liskov Substitution Principle as a Contract Rule

When a subtype overrides a method, its contract must be compatible: preconditions may only weaken (accept more), postconditions may only strengthen (promise more), invariants must be maintained. This is a reviewable property of interface specifications. — Liskov & Wing, ACM TOPLAS, 16(6), 1994

3.3 Behavioral Specification

How to specify what a unit does without paraphrasing how the code works. This is the art of describing expected behavior at a level that enables independent test derivation.

The Level-of-Detail Problem

Algorithm specification has two failure modes: too abstract (“process the data” — untestable) and too concrete (copying pseudocode from the implementation — circular, useless for independent testing). The right level constrains behavior enough to derive tests while leaving the implementer freedom to choose data structures and optimize.

DO-178C defines the target precisely: Low-Level Requirements must be detailed enough that “source code can be directly implemented without further information.” — Parasoft, DO-178C overview

Algorithm Specification

A spectrum from natural language through pseudocode to mathematical notation. The key distinction is between behavioral requirement (what property the output must satisfy) and algorithm selection (which specific algorithm to use).

Bad — Algorithm selection without behavioral requirement
DesignThe function shall use quicksort with median-of-three pivot selection.
Good — Behavioral requirement with justified algorithm constraint
DesignThe function shall produce a list containing exactly the elements of the input, in non-descending order according to the provided comparator. Stability is required (equal elements preserve their relative order) because downstream processing depends on insertion order as a tiebreaker. Performance: O(n log n) worst-case is required for real-time scheduling compliance (SYS-TIMING-003). Quicksort is excluded due to O(n²) worst case.

Decision Tables

Decision tables excel at specifying rule-based behavior. Each rule is a complete combination of conditions mapped to actions. A complete decision table gives a test suite with decision coverage by construction, and maps directly to MC/DC test case derivation. — Myers, The Art of Software Testing, 1979, Ch. 4; ISTQB Foundation Level, §4.2.3

State Machine Specification

Use a state machine when a unit has distinct operational modes — when it behaves differently based on prior history, not just current inputs. Harel statecharts (1987) solved the state explosion problem with hierarchy, concurrency, and communication. — Harel, Science of Computer Programming, 8(3), 1987

A state machine section in a design document must include: state inventory with invariants, transition table (source, event, guard, action, target), initial and terminal states, and undefined event handling (ignore, log, fault, or exception).

Specification Patterns: Choosing the Right Form

Behavior TypeRecommended Pattern
Pure function, mathematical transformationPre/postcondition + property-based
Rule-based logic (many conditions → action)Decision table
Mode-dependent, event-driven behaviorState machine (Harel statechart)

Natural Language vs. Semi-Formal vs. Formal

ISO 26262 explicitly maps notation formality to ASIL: natural language suffices for ASIL A/B; semi-formal notation (UML, state machines) is highly recommended for ASIL C/D; formal methods are highly recommended for ASIL D. DO-333 (the Formal Methods Supplement to DO-178C) provides a framework for using formal methods as an alternative verification path. — AFuzion, DO-333 overview

3.4 Design Rationale and Decisions

Rationale is what separates a design document from a code description. Without it, maintainers encounter decisions that appear arbitrary and either reverse them (breaking hidden constraints) or preserve them unnecessarily (losing improvement opportunities).

Why Rationale Matters

Kruchten argued that design decisions should be first-class entities with attributes and relationships, not implementation footnotes. Decisions have context, alternatives, justification, and connections to other decisions. — Kruchten, “An Ontology of Architectural Design Decisions”, 2004

Burge and Brown identified the costs of rationale absence: design verification becomes impossible (an auditor cannot assess appropriateness without knowing rejected alternatives), conflict mitigation fails (engineers cannot trace conflicting constraints), and knowledge transfer breaks. — Burge & Brown, “Reasoning with Design Rationale”

Architecture Decision Records for Unit-Level Design

The ADR format (Tyree & Akerman 2005; lightweight form by Nygard 2011) is directly applicable to non-trivial unit-level design decisions. Nygard’s format: Title, Status, Context, Decision, Consequences. The Context section describes the forces at play; the Consequences section records what becomes easier or harder. — Tyree & Akerman, IEEE Software, 22(2), 2005; Nygard, Cognitect blog, 2011

A unit-level decision record is appropriate when the decision is non-obvious, was constrained by factors not visible in the code, was made in preference to a plausible alternative, or has implications that a future maintainer must understand before modifying the code.

Constraint Documentation

A decision cannot be fully understood without documenting the constraints that limited the solution space. Four categories:

  • External — standards mandates, platform limitations, certification requirements. “MISRA-C compliance; no dynamic allocation after init.”
  • Architectural — decisions from a higher layer. “Single-threaded event loop; blocking prohibited in this module.”
  • Resource — timing budgets, memory limits, stack depth. “Called from interrupt context; stack bounded to 256 bytes.”
  • Temporal — correct at the time but may be lifted later. “Matches legacy interface; legacy decommissioned in Phase 4.”

Without explicit constraint documentation, the next engineer has no way to distinguish “this design is bad and should be changed” from “this design is forced by an external constraint and cannot be changed.”

3.5 Error Handling and Fault Containment

Error handling is a design decision, not an implementation detail. A design that leaves error strategy to individual implementers will produce inconsistent, untestable, and potentially unsafe error behavior.

What the Design Must Specify About Errors

For each unit, the design document must answer six questions:

  1. What errors can this unit encounter or produce?
  2. How is each error detected?
  3. How does each error propagate (or get contained)?
  4. What is the recovery strategy for each error class?
  5. What state is the unit left in after an error?
  6. What does the caller receive (exception, error code, degraded result)?

Error Propagation Strategy

Joshua Bloch’s Effective Java Items 69–77 establish the canonical guidance for Java exception design. Key design decisions: — Bloch, Effective Java, 3rd ed., Addison-Wesley, 2018

  • Checked vs. unchecked exceptions — a checked exception is a specification of caller obligation (the caller must handle it). An unchecked exception indicates a programming error (the caller has a bug). This is a design-level decision about who is responsible.
  • Exception translation at boundaries — lower-layer exceptions should be caught and translated to domain-specific exceptions at component boundaries. A database SQLException should not leak through to business logic.
  • Exception hierarchy — keep hierarchies shallow. Separate domain exceptions (business rule violations) from technical exceptions (infrastructure failures).

Fault Containment and Isolation

Michael Nygard’s Release It! formalized stability patterns that are design-level decisions: — Nygard, Release It!, 2nd ed., Pragmatic Bookshelf, 2018

  • Bulkhead pattern — isolate resource pools so that failure in one subsystem does not drain resources from others. The design must specify the isolation boundaries.
  • Circuit breaker — a three-state machine (closed, open, half-open) that prevents cascading failures. The design must specify: failure threshold, timeout period, health check mechanism, and fallback behavior.

Robert Hanmer’s Patterns for Fault Tolerant Software extends this with error containment barriers and quarantine patterns specifically for safety-critical systems. — Hanmer, Patterns for Fault Tolerant Software, Wiley, 2007

Fail-Fast vs. Recovery vs. Degraded Operation

Jim Shore defined “fail fast” as: “When a problem occurs, it fails immediately and visibly. If something unusually or unexpectedly occurs, let the software fail immediately.” In safety-critical contexts, fail-fast means detect-and-surface, not crash — the design must specify what “surface” means (safe state, alert, shutdown sequence). — Shore, “Fail Fast”, IEEE Software, Sep/Oct 2004

The design must specify, for each error class, which strategy applies: retry (with backoff?), fallback (to what?), compensating action, safe state, or propagation to caller.

Error Handling Matrix

Safety-Critical Error Handling

DO-178C requires robustness testing — tests that verify behavior under abnormal and erroneous inputs. These tests can only exist if the design specifies what “abnormal behavior” means for each unit. Undocumented error handling creates an untestability gap: the test engineer cannot write a robustness test without knowing what the correct error response is.

ASPICE SWE.3 requires that error handling be traceable to requirements, not ad-hoc defensive code. ISO 26262 defines specific fault tolerance mechanisms and requires that degradation concepts be designed, not improvised.

Anti-Patterns

  • No error strategy — each implementer handles errors differently, producing inconsistent behavior across the component
  • Catch and ignore — silently swallowing exceptions masks defects and violates every safety standard
  • Catch Exception — over-broad catching masks programming errors (NPE, ClassCast) alongside expected conditions
  • Exception tunneling — low-level exceptions propagate through layers that don’t understand them, leaking implementation details
  • Missing error specification in interfaces — the function signature says nothing about what can go wrong; callers must guess

3.6 Dynamic Behavior and Concurrency

ASPICE SWE.3 BP.3 requires: “Evaluate and document the dynamic behavior of and the interaction between relevant software units.” The key qualifier is “relevant” — not every unit has dynamic behavior worth modeling. Specify dynamic behavior when the unit has multiple states, history-dependent behavior, concurrency, or timing constraints.

State Machine Testing Coverage

State-based testing has an established coverage hierarchy: state coverage (every state visited) → transition coverage (every transition exercised) → round-trip path coverage (every cycle covered) → sneak-path testing (verify illegal transitions cannot be triggered). Sneak-path testing is critical for safety: it exercises the system with events undefined for the current state to verify correct rejection. — Coverage Criteria for State-Based Testing, systematic review

Thread Safety as a Design Contract

Thread safety is an interface property, not an implementation detail. Brian Goetz established the standard taxonomy for Java: — Goetz et al., Java Concurrency in Practice, 2006

LevelMeaningCaller Obligation
ImmutableState cannot change after constructionNone — inherently safe
Thread-safeInternal synchronization sufficientNone — safe for concurrent access
Conditionally thread-safeSome method sequences need external syncMust synchronize documented sequences
Thread-compatibleNot internally synchronizedMust provide all synchronization externally
Thread-hostileCannot be made safeAvoid in multi-threaded contexts

Every shared mutable field must be documented at design level: which field, which lock guards it, what happens-before relationship is established, and what invariants must hold across all threads. An undocumented shared field is a design defect. — SEI CERT, CON52-J

Timing and Performance Constraints

A timing constraint must specify: constraint type (deadline, period, response time), numeric bound with unit, conditions (worst-case, nominal), source requirement, and verification method (static analysis, instrumented measurement, formal proof). Vague timing specifications (“shall execute quickly”) are untestable.

Good — Well-formed timing constraint
DesignprocessFrame() shall complete within 5 ms on target hardware (ARM Cortex-M4 @ 168 MHz) under worst-case input conditions. Derived from: SYS-TIMING-003. Verification: WCET analysis via static tool + instrumented measurement.

Concurrency Anti-Patterns in Design

  • Check-then-act without atomicity — two operations described as sequential that must be atomic but are not specified to be atomic
  • Missing cancellation policy — a long-running task without specification of how, when, and what happens on cancellation — Goetz, JCIP, Ch. 7
  • Over-synchronization — synchronizing code that does not need protection; lock nesting without consistent ordering
  • Implicit concurrency assumptions — returning mutable objects that may be shared, without specifying ownership semantics

Safety-Critical Concurrency

DO-178C addresses concurrency through ARINC 653 partitioning: spatial isolation (hardware-enforced memory regions) and temporal isolation (fixed time windows in a major frame). ISO 26262 defines freedom from interference (FFI) across three categories: timing/execution, memory, and information exchange. Design documents for safety-critical software must include an FFI analysis for every interface between components of different assurance levels. — ARINC 653; ISO 26262 FFI

3.7 Scaling: What Needs a Design and How Much Detail

A 300,000-line codebase cannot have 300,000 lines of design documentation that paraphrase the code. The question is: which units warrant full detailed design, which need only interface documentation, and which are adequately covered by their code and tests alone?

The Economics

Capers Jones found that on defense/aerospace projects, document production costs can be twice the cost of writing the code. Yet a systematic mapping of 69 empirical papers (Zhi et al. 2015) found only 6 addressed documentation cost at all — there is no reliable model to calculate documentation ROI per unit. — Zhi et al., JSS, Vol. 99, 2015

Stale documentation is worse than none — it creates false confidence. An engineer reading a design document that has drifted from the code believes they understand the system when they do not. — Rios et al., “Documentation Technical Debt”, ACM, 2019

Standards-Supported Tiering

StandardTiering Support
IEC 62304Explicit. Detailed design required only for Class C. Classes A and B exempt.
ISO 26262Notation-based. Natural language (ASIL A/B), semi-formal (ASIL C/D recommended), formal (ASIL D highly recommended).
DO-178CNo internal tiering within a DAL. All software at a given DAL treated uniformly. But LLR content naturally scales with complexity.
ASPICENo formal tiering. All components in scope. Can be combined with ISO 26262 ASIL levels to justify differentiated rigor.

Practical Tiering: Three Levels

Tier 1 — Full detailed design when any of the following apply: safety-critical functionality, complex algorithm, state machine, concurrency/synchronization, high defect history, high change frequency, complex interface contracts.

Tier 2 — Interface documentation when the unit performs well-understood operations, implements standard patterns without deviation, and has behavior fully specified by its interface and comprehensive tests. The iceoryx header-file model is proven viable for ASPICE SWE.3. — iceoryx SWE.3 guidelines

Tier 3 — Component-level summary for DTOs, value objects, simple utilities, and data classes with no significant logic. Named in the component inventory with one-line description only.

Metrics for Prioritization

  • Cyclomatic complexity > 10 — strong candidate for full design documentation. NIST SP 500-235 recommends 10 as the baseline limit. — Watson & McCabe, NIST SP 500-235, 1996
  • High fan-in — shared infrastructure used by many callers. A defect propagates widely. Prioritize interface documentation regardless of internal complexity.
  • High change frequency — hotspot analysis using VCS history. About 1–2% of a codebase accounts for up to 70% of development work. — Tornhill, Your Code as a Crime Scene, 2nd ed., 2024
  • Defect density — units with above-average historical defects have demonstrated unclear design intent.

Legacy Retrofit: The Post-Hoc Design Problem

When retrofitting design documentation onto existing code, the design-before-code chain is broken. Acknowledge this honestly to assessors. Demonstrate disciplined analysis (not just code-comment extraction). Implement design-before-code for all new changes going forward. Use the retrofit as a gap-finding exercise — undocumented behavior and missing test coverage are legitimate findings.

3.8 AI-Assisted Detailed Design

LLMs can produce design-shaped text quickly and fluently. Whether that text corresponds to correct, complete, and verifiable design intent is a different question — and the evidence base does not support optimism.

Forward Design Generation

LLMs can take requirements and produce interface specifications, behavioral descriptions, and state machine descriptions that look like detailed design. The primary risks are hallucinated constraints (plausible but untraced), missing edge cases, and over-generic designs that appear complete but omit the specifics that make them testable. — “Leveraging LLMs for Formal Software Requirements”, 2025

A 2025 study found that LLMs “often fail to properly understand existing design patterns and coding styles of a project,” leading to generated artifacts that require extensive revision. — “Do Code LLMs Understand Design Patterns?”, 2025

Reverse-Engineering Design from Code (Legacy Retrofit)

For legacy codebases, the question is whether AI can produce useful design documentation from existing code. The critical distinction:

  • Code paraphrase — the AI describes what the code does in English. This is not design documentation; it is a different encoding of the same information, with the same gaps.
  • Recovered design intent — the AI identifies abstractions, boundaries, invariants, and contracts that the code implements, expressed above the implementation level. This is what design documentation should contain.

The Post-Hoc Paraphrase Trap

The central failure mode: AI produces a sophisticated description of what the code does, not a design document that reveals intent, decisions, and constraints independently of the code. This is the exact anti-pattern that standards warn about — amplified by AI’s ability to produce authoritative-sounding prose.

Missing rationale is a structural failure, not a fixable omission. Rationale is not in code. Any AI-generated rationale for a legacy system is invention, not recovery.

Safety-Critical Implications

No published study as of 2026 has evaluated whether LLM-generated design documentation can pass a V-model assessment at any assurance level. FAA (July 2024 roadmap) and EASA (NPA 2025-07) are actively building AI guidance for aviation, but neither has produced specific requirements for AI-generated lifecycle data. DO-178C independence requirements (reviewer cannot be the author) create an unresolved ambiguity for AI-authored design documents. — FAA AI Safety Roadmap, 2024; EASA NPA 2025-07

Mitigation Strategies

  • Structured templates as guardrails — constrain AI output to match valid design document schemas; omissions become visible as empty fields
  • AI as analyst, not author — use AI for code comprehension and pattern identification; human authors supply design intent and judgment — Thoughtworks Technology Radar
  • Bidirectional traceability enforcement — every design claim must trace to a requirement; gaps and hallucinated constraints are caught mechanically
  • Human-in-the-loop at every decision point — not superficial approval of plausible-looking output, but substantive review of content correctness

V-Model Specific Considerations

Traceability Requirements

Every V-model standard requires bidirectional traceability through the detailed design layer:

SW Requirements ↔ Architecture ↔ Detailed Design ↔ Source Code ↔ Unit Tests

Each design element must trace upward to the architectural component or requirement it satisfies, and downward to the source files and test cases that implement and verify it. Orphan design elements (no parent requirement) and orphan requirements (no child design element) are both compliance findings.

Derived Requirements

Design decisions often generate new requirements that were not in the original specification. These derived requirements must be explicitly identified, justified, and traced. Common sources: choice of synchronization mechanism creates timing constraints; error handling strategy creates recovery requirements; interface design creates compatibility constraints. ASPICE requires traceability of derived requirements back to the design decision that created them.

Design Review Independence

DO-178C requires independent review for DAL A/B — the reviewer must not be the author. For DAL C, review can be by a peer. This has implications for AI-assisted design: if an LLM generates the design, the review must be performed by a human with domain expertise who can assess correctness, not merely structural completeness.

Code-Can-Be-the-LLR Question

Can source code serve as its own detailed design? The general advice from DO-178C practitioners is no. AdaCore’s compliance guidance states: “the general advice is to avoid using a programming language as the medium for expressing — even in part — the software design” and “requirements are properties to be verified by the code and are not the code itself.” Exception: Ada’s specification/body separation can serve design purposes. Model-based development (DO-331) is another path where the model IS the design. — AdaCore DO-178C Compliance Analysis

ASPICE SWE.3: Common Assessment Failures

  • Post-hoc design — writing design after code. Assessors can tell from document metadata and commit history.
  • Source code markup as design — using code comments or Javadoc as the sole design artifact, when the content merely describes what the code does rather than serving as a prior contract.
  • Missing interface detail — interfaces documented without types, units, ranges, or constraints. BP.2 requires these explicitly.
  • Missing dynamic behavior — no state machine or interaction documentation for units that clearly have modal behavior (BP.3).

Framework Integration

Design Artifact Input

The detailed design takes the software architecture as its primary input. Each architectural component is decomposed into units with individual design specifications. The architecture defines component boundaries, responsibilities, and inter-component interfaces; the detailed design specifies how each unit fulfills its component’s responsibilities.

Design Artifact Format

Detailed designs are Markdown files with YAML frontmatter (.md). This makes them readable by humans and assessors while remaining parseable by agents and the traceability engine. The schema is at schemas/artifacts/detailed-design.schema.yaml.

Trace Links

Each detailed design artifact links to:

  • Upward: trace_from in the frontmatter lists architecture or SW requirement IDs this design derives from
  • Downward: source files implement the design; unit test cases verify it. The traceability engine validates bidirectional coverage and detects staleness via content hashes.

Design-to-Test Pipeline

The detailed design is the direct input for unit test derivation. Every behavior rule (B1, B2, …) and error handling rule (E1, E2, …) maps to at least one test case. The derive-test-cases craft skill reads the design specification and produces test cases using four strategies: requirement-based testing, equivalence class partitioning, boundary value analysis, and fault injection. See the Unit Test derivation strategies documentation.

Schema: The Layer Model

The detailed design schema uses an additive layering model. Higher layers build on lower layers — you never skip a layer. The layer a document reaches determines how much design evidence it contains.

LayerWhat It ContainsWhen It’s Sufficient
Layer 1
Component Design		 Purpose, external interfaces, unit inventory, shared patterns, constraints. The map of what exists and how it connects. QM / ASIL A / IEC 62304 Class A. Components with only simple, well-understood units.
Layer 2
Unit Interface Contracts		 Per-unit: purpose, interfaces (inputs/outputs tables), behavior rules, error handling. The level from which unit tests can be derived. ASIL A–B / IEC 62304 Class B / DO-178C DAL D–E. Standard units with testable behavior.
Layer 3
Deep Design Evidence		 Design rationale, dynamic behavior (state machines, diagrams), concurrency model, negative requirements. ASIL C–D / DO-178C DAL A–C / IEC 62304 Class C. Safety-critical or high-complexity units.

How Layers Map to Section 3.7 Tiering

The schema’s layers implement the documentation tiering discussed in Section 3.7. A Layer 1 document is the Tier 3 “component-level summary.” Layer 2 adds the Tier 2 “interface documentation.” Layer 3 adds the Tier 1 “full detailed design” with rationale and formal notation.

File Organization

Two artifact ID patterns:

  • CD-nnn — Component Design document. Always starts at Layer 1. May embed Layer 2/3 content for its units as subsections.
  • DD-nnn — Standalone Unit Design document. Used when a unit is complex enough to warrant its own file. References the parent CD-nnn.

Both live in docs/artifacts/detailed-design/. When units are embedded in the component document, no separate DD-nnn files are needed.

YAML Frontmatter

Every detailed design file starts with YAML frontmatter between --- delimiters:

YAML frontmatter--- artifact_id: CD-001 # or DD-017 for standalone unit artifact_type: detailed-design version: "1.0.0" status: draft # draft | in_review | approved | baselined | superseded component: fuel-control # references sw-architecture component id layer: 3 # highest layer present in this document trace_from: # upward traceability - ARCH-004 - SWREQ-FUEL-001 assurance_level: 2 # optional: 1=highest (DAL A/ASIL D), 5=lowest unit_id: FuelRateLimiter # only for standalone DD-nnn documents ---

Layer 1: Component Design (Always Present)

Every detailed design document starts with these headings. They provide the component-level map.

HeadingContent
# {Component Name}Document title. The component’s human-readable name.
## PurposeWhat this component does and why it exists. 1–3 paragraphs. Not a restatement of the architecture — focus on this component’s specific role.
## External InterfacesHow this component communicates with other components. Table: Direction | Partner Component | Mechanism | Data | Description.
## Unit InventoryAll units listed with their design layer. Table: Unit ID | Layer | Brief Description. Every unit must appear — even trivial ones.
## Shared PatternsDesign decisions that apply across multiple units: error strategy, threading model, resource management, naming conventions, common data types. Document once here rather than repeating per unit.
## ConstraintsComponent-wide non-functional requirements: performance budgets, memory limits, certification requirements, deployment constraints.

Layer 2: Unit Interface Contracts (Per Unit)

Added per unit, either as subsections within the component document (using ### headings under a ## {Unit Name}) or as separate DD-nnn files. This is the level from which tests are derived.

HeadingContentID Convention
### PurposeWhat this specific unit does. 1–2 sentences.—
### InterfacesThe unit’s inputs and outputs with full type information.—
#### InputsTable: ID | Name | Type | Unit | Constraints | DescriptionI1, I2, I3…
#### OutputsTable: ID | Name | Type | Unit | Constraints | DescriptionO1, O2, O3…
### BehaviorHow inputs become outputs. Each row is one testable rule. Table: ID | Condition | ResultB1, B2, B3…
### Error HandlingEvery foreseeable error and the unit’s response. Table: ID | Condition | ResponseE1, E2, E3…
### Configuration(optional) Parameters set at init/deploy time. Table: Name | Type | Default | Description—
### Internal State(optional) State retained between invocations. Table: Name | Type | Description—
### Constraints(optional) Unit-specific non-functional requirements (timing, memory, thread-safety).—

How Section 3 Principles Map to Layer 2 Tables

  • DbC preconditions (Section 3.2) → the Constraints column in the Inputs table
  • DbC postconditions (Section 3.2) → the Result column in the Behavior table + Output constraints
  • Error semantics (Section 3.2, 3.5) → the Error Handling table
  • Thread-safety guarantee (Section 3.6) → the unit-level Constraints section
  • Behavioral specification (Section 3.3) → the Behavior table (one row per testable rule)

Layer 3: Deep Design Evidence (Critical Units)

Added for safety-critical or high-complexity units. These sections provide the evidence that separates design from code paraphrase.

HeadingContent
### Design RationaleFor each significant decision: what was decided, alternatives considered, justification. Table or prose. Maps to Section 3.4.
### Dynamic Behavior(optional) State machines, sequence diagrams, timing diagrams. Use Mermaid or PlantUML code blocks. Maps to Section 3.6.
### Concurrency Model(optional) Thread-safety level, shared state with guards, synchronization mechanisms. Maps to Section 3.6.
### Negative Requirements(optional) What the unit explicitly must NOT do. Prevents scope creep and clarifies responsibility.

Traceability ID Conventions

Every table row has an ID column. These IDs enable fine-grained traceability between design elements and test cases.

ElementID FormatExample
Component design documentCD-{nnn}CD-001
Standalone unit designDD-{nnn}DD-017
InputI{n}I1, I2
OutputO{n}O1, O2
Behavior ruleB{n}B1, B2, B3
Error handling ruleE{n}E1, E2

Use dot notation to reference specific elements across documents:

  • DD-001.B3 → Behavior rule 3 in design DD-001
  • DD-001.E1 → Error handling rule 1
  • CD-001.FuelRateLimiter.B3 → When unit is embedded in component doc

These references are used by trace links and test case mappings. A test case traces to a specific behavior or error rule, not to the entire design document.

Example: Fuel Control Component (Layer 3)

A complete component design with an embedded Layer 3 unit. This is what a detailed design looks like for a safety-critical component in an engine control system.

CD-001.md — Complete component design--- artifact_id: CD-001 artifact_type: detailed-design version: "1.0.0" status: draft component: fuel-control layer: 3 trace_from: - ARCH-004 - SWREQ-FUEL-001 - SWREQ-FUEL-002 - SWREQ-FUEL-003 assurance_level: 2 --- # Fuel Control ## Purpose The fuel-control component computes the fuel delivery rate for a single engine. It receives a raw demanded rate from the engine control law and clamps it to mode-specific safety bounds before passing it to the fuel metering hardware interface. If this component were deleted, the engine would receive unvalidated fuel demands — potentially exceeding structural limits during transients or producing negative rates during shutdown sequencing. ## External Interfaces | Direction | Partner Component | Mechanism | Data | Description | |-----------|-------------------|----------------|-----------------|------------------------------------------------| | Consumes | engine-control | function call | RawFuelDemand | Demanded rate from control law (kg/h) | | Consumes | mode-manager | shared state | EngineMode | Current engine operating mode | | Provides | fuel-metering | function call | ClampedFuelRate | Validated rate after safety clamping (kg/h) | | Provides | health-monitor | event | FuelRateAlert | Emitted when clamping activates (>5% delta) | ## Unit Inventory | Unit ID | Layer | Brief Description | |-------------------|-------|----------------------------------------------------| | FuelRateLimiter | 3 | Clamps fuel rate to mode-specific bounds | | FuelRateLogger | 1 | Logs rate history to circular buffer for diagnostics| | FuelDemandAdapter | 1 | Adapts raw demand from control law interface format | ## Shared Patterns **Error strategy:** Units in this component throw `FuelControlException` (unchecked) for programming errors (null inputs, invalid enum values). For operational errors (sensor out of range, mode transition during computation), units return a safe fallback value and emit a `FuelRateAlert` event. No silent error swallowing. **Threading model:** Single-threaded. All calls originate from the main control loop at 100 Hz. No unit in this component may block or spawn threads. **Units of measurement:** All fuel rates are in kg/h as `double`. All temperatures are in Kelvin as `double`. No imperial units anywhere in this component. ## Constraints - WCET budget: entire component must complete within 2 ms per cycle (allocated from SYS-TIMING-001). - No heap allocation after initialization (MISRA-C heritage rule, retained for predictability). - Component is DAL B / ASIL C — independent review required. --- ## Fuel Rate Limiter (FuelRateLimiter) ### Purpose Clamps a raw fuel demand to the safety bounds defined for the current engine operating mode. This is the primary safety function in the fuel-control component. ### Interfaces #### Inputs | ID | Name | Type | Unit | Constraints | Description | |----|----------------|---------|------|------------------------------------|------------------------------------------| | I1 | rawDemand | float | kg/h | ≥ 0.0, ≤ 500.0 | Raw fuel demand from engine control law | | I2 | mode | enum | — | {STARTUP, IDLE, CRUISE, MAX, SHUTDOWN} | Current engine operating mode | | I3 | ambientTemp | float | K | ≥ 200.0, ≤ 330.0 | Outside air temperature for density comp | #### Outputs | ID | Name | Type | Unit | Constraints | Description | |----|----------------|---------|------|------------------------------------|------------------------------------------| | O1 | clampedRate | float | kg/h | Within mode bounds (see B1–B5) | Validated fuel rate after clamping | | O2 | wasClipped | boolean | — | — | true if clamping changed the raw demand | | O3 | clipDelta | float | kg/h | ≥ 0.0 | Absolute difference between raw and clamped | ### Behavior | ID | Condition | Result | |----|------------------------------------|--------------------------------------------------------------------------| | B1 | mode = STARTUP | O1 = min(I1, 50.0). Startup clamped to 50 kg/h max. | | B2 | mode = IDLE | O1 = clamp(I1, 5.0, 30.0). Idle range: 5–30 kg/h. | | B3 | mode = CRUISE | O1 = clamp(I1, 20.0, 300.0). Cruise range: 20–300 kg/h. | | B4 | mode = MAX | O1 = clamp(I1, 100.0, 500.0). Max thrust range: 100–500 kg/h. | | B5 | mode = SHUTDOWN | O1 = max(I1 × 0.5, 0.0). Halve demand, floor at zero. | | B6 | I3 < 233.15 (below −40°C) | O1 = O1 × 1.05. Cold-start density compensation (+5%). | | B7 | always | O2 = (O1 ≠ I1). O3 = |I1 − O1|. | ### Error Handling | ID | Condition | Response | |----|------------------------------------|--------------------------------------------------------------------------| | E1 | I1 < 0.0 | Return O1 = 0.0, emit FuelRateAlert(NEGATIVE_DEMAND). | | E2 | I1 > 500.0 | Clamp to 500.0, emit FuelRateAlert(DEMAND_EXCEEDS_MAX). | | E3 | I2 is null or not a valid mode | Throw FuelControlException("invalid mode"). Programming error. | | E4 | I3 outside [200.0, 330.0] | Use I3 = 288.15 (ISA standard), emit FuelRateAlert(TEMP_OUT_OF_RANGE). | ### Internal State This unit is **stateless**. Output depends only on current inputs. No state is retained between invocations. ### Constraints - Thread-safety: not required (single-threaded control loop). - WCET: must complete within 0.3 ms (allocated from component budget). - No floating-point exceptions: all divisions guarded, no NaN/Inf propagation. ### Design Rationale | Decision | Alternatives Considered | Justification | |---------------------------------------|--------------------------------------------|-------------------------------------------------------------------------| | Clamp rather than reject out-of-range | Reject and return error; saturate silently | Clamping maintains engine operability. Silent saturation would hide issues from the health monitor. Rejection would cause engine flameout on transient spikes. SWREQ-FUEL-002 requires continued operation under sensor faults. | | Per-mode bounds as constants | Configurable bounds from parameter file | Bounds are safety-critical and change only during certification updates. Runtime configurability creates a configuration management risk that SWREQ-FUEL-003 prohibits. | | Cold-start compensation as +5% factor | Lookup table by temperature; ignore | Linear factor is simple, verifiable, and sufficient per flight test data (SYS-PERF-007). Lookup table adds complexity without measurable benefit in the −40°C to 0°C range. Ignoring would violate SYS-PERF-007. | ### Dynamic Behavior This unit is stateless and does not have modal behavior. The mode input (I2) selects bounds but does not create internal states. No state machine is needed. If a future requirement adds rate-of-change limiting (requiring memory of previous output), this unit will need a state machine and this section must be updated. ### Negative Requirements - Must NOT perform I/O (logging is FuelRateLogger’s responsibility). - Must NOT cache results between invocations. - Must NOT allocate heap memory. - Must NOT modify the mode input or any global state.

Example: Configuration Loader (Layer 1 Only)

A Layer 1 component design for a simple, non-critical component. No per-unit behavioral specification — the unit inventory and shared patterns are sufficient because all units are straightforward.

CD-008.md — Layer 1 component design--- artifact_id: CD-008 artifact_type: detailed-design version: "1.0.0" status: draft component: config-loader layer: 1 trace_from: - ARCH-012 --- # Configuration Loader ## Purpose Reads application configuration from YAML files on disk and provides typed access to configuration values for other components. This component is the single entry point for all file-based configuration — no other component reads config files directly. If this component were deleted, all runtime-configurable parameters would need to be hard-coded or passed as command-line arguments. ## External Interfaces | Direction | Partner Component | Mechanism | Data | Description | |-----------|-------------------|---------------|------------------|----------------------------------------| | Consumes | filesystem | file read | YAML files | Config files from deployment directory | | Provides | all components | function call | typed values | Typed getters: getString, getInt, etc. | | Provides | health-monitor | function call | loadStatus | Whether config loaded successfully | ## Unit Inventory | Unit ID | Layer | Brief Description | |------------------|-------|-----------------------------------------------------| | YamlFileReader | 1 | Reads and parses a single YAML file to a map | | ConfigRegistry | 1 | Holds parsed config; provides typed getters | | ConfigValidator | 1 | Validates required keys are present after loading | | DefaultsProvider | 1 | Supplies default values for optional config keys | ## Shared Patterns **Error strategy:** Config loading errors (file not found, parse failure, missing required key) are collected into a `ConfigLoadResult` object with a list of warnings and errors. No exceptions thrown during loading. Callers check the result object. Missing optional keys return the default silently. **Immutability:** Once loaded, the configuration is immutable. No setter methods. Thread-safe by construction. Reload requires creating a new ConfigRegistry instance. ## Constraints - Configuration is loaded once at startup. No hot-reload. - Maximum config file size: 1 MB (reject larger files). - All keys are case-sensitive. - No sensitive data (passwords, keys) in config files — those use environment variables via a separate component.

Why Layer 1 Is Sufficient Here

Every unit in this component (YamlFileReader, ConfigRegistry, ConfigValidator, DefaultsProvider) is a simple, well-understood operation: read a file, store values in a map, check for required keys, provide defaults. The shared patterns section documents the error strategy and immutability contract. A developer can implement these units from the component description and the shared patterns alone. A tester can derive integration tests from the external interfaces table.

If a unit became more complex (e.g., ConfigValidator gained complex validation rules with conditional logic), it would be promoted to Layer 2 in the Unit Inventory table and given its own Interfaces / Behavior / Error Handling sections.
