Every-Scope Artifact

1. V-Model Context

A TestSpec states how a scope is verified: which behaviours, interfaces, flows, and error conditions are under test; which derivation strategies the cases were drawn from; which spec elements each case verifies; what coverage bar the test code must clear. One TestSpec per scope — root, branch, leaf. Same envelope at every layer; different weight. At root it specifies end-to-end scenarios against Product Brief outcomes, root Requirements, and root Architecture Composition; at a branch it specifies integration behaviour against both that branch's Requirements (behavioural intent) and that branch's Architecture (Composition entries + interface DbC clauses); at a leaf it specifies unit verification against the leaf's Detailed Design. The TestSpec is the derivation-discipline layer between spec and test code: it is not a replacement for the test code and it is not a duplicate of it.

Position on the V

TestSpec is the only artifact in the framework that lives on the right-hand ascent of the V. Every other spec artifact describes design intent as the V descends; TestSpec describes verification intent as the V ascends. Each TestSpec sits directly opposite the layer's specs — two upstream sources at non-leaf scopes, not one (Position C, TARGET_ARCHITECTURE §5.3): root TestSpec ↔ Product Brief + root Requirements + root Architecture Composition; branch TestSpec ↔ branch Requirements + branch Architecture (Composition + interface DbC); leaf TestSpec ↔ that leaf's Detailed Design. The horizontal connection is strict — a leaf TestSpec does not derive its cases from root Requirements, and a root TestSpec does not derive its cases from leaf Detailed Designs. Each layer tests its own layer. Upward traceability is built from each TestSpec's verifies links, not from tests reaching across layers.

The core rule that falls out of this positioning: tests derive from the layer's specs, never from the code below. A TestSpec written by reading the implementation and enumerating what the code already does is a tautology in artifact form — it will always pass and will never find a bug. It also destroys the V. If tests derive from code, the code is the de-facto specification and the spec above it is documentation theatre. Every failure mode the framework is designed to catch — AI happy-path bias, retrofit fabrication, silent requirement drift — is a direct consequence of derivation source confusion. The rule is simple to state, hard to hold under pressure, and non-negotiable.

Root TestSpec draws from three co-equal sources: Product Brief outcomes (user-visible scenarios), root Requirements (behavioural intent), and root Architecture Composition (orchestration / integration shape). Branch TestSpec draws from two: branch Requirements (behavioural cases — functional, boundary, error, state-transition) and branch Architecture (composition cases — integration, contract, property-when-about-wiring). Architecture is a first-class verification target at non-leaf scopes: Composition entries describe emergent runtime behaviour; interface DbC clauses (preconditions, postconditions, invariants) describe boundary behaviour. Decomposition entries are not verification targets — they are structural metadata that propagates intent down to child specs. Leaf TestSpec draws exclusively from the leaf's Detailed Design. Crossing these lanes — e.g., deriving unit cases from root Requirements, or system scenarios from leaf DDs — is a structural error and makes the test suite unreviewable.

Inputs and outputs

A TestSpec consumes two kinds of inputs and produces three. Inputs: the layer's spec artifacts (Product Brief + root Requirements + root Architecture Composition at root; branch Requirements + branch Architecture at branch; Detailed Design at leaf), and the derivation strategies applied (see §2.3). Outputs: a flat list of test cases each with a non-empty verifies link; a coverage matrix mapping every spec element to at least one case; and a declared coverage bar (structural and mutation-score) that the test code is expected to hit. The TestSpec is not the test code. Test code lives in the project's source tree; the TestSpec lives in /specs/ alongside the other spec artifacts and is authored before the test code is written.

“Tests as spec” vs TestSpec

A mature TDD practitioner will push back on the idea of a separate TestSpec artifact. “The tests are the specification,” they will say — well-named JUnit methods, Arrange-Act-Assert structure, specific assertions, the test suite as executable documentation (Beck, Meszaros, Adzic). They are not wrong about the test code. They are wrong about the conclusion.

A TestSpec is not a test-case catalogue duplicating what the code already expresses. It is the derivation-discipline layer. It records which equivalence classes were picked, which boundary values were chosen, which error cases were included, why those and not others. That information is not in the test code. The test code shows what the author thought to test; the TestSpec shows what the layer's spec said needed to be tested. Without a TestSpec you lose four distinct review leverage points:

  • Derivation choices are invisible. “We have seven tests that pass” is not the same claim as “every equivalence class, every boundary, every error path named in the DD is covered.” The second claim requires a TestSpec to compare tests against.
  • AI target disappears. An LLM writing tests without a TestSpec either mirrors the DD prose (derivation implicit) or mirrors the code (the canonical failure mode). Empirically, LLM test suites routinely hit 100% line coverage with 4% mutation score (see §2.7). A TestSpec pins the derivation so the AI has something concrete to aim at and a human can review the aim.
  • Traceability anchor collapses. Requirements and Detailed-Design elements need to trace to test artifacts; test artifacts need stable IDs at case granularity. JUnit method names are not stable artifact IDs.
  • Completeness gate fails. The Quality Bar rule “every spec element has at least one verifying test” cannot be enforced without a TestSpec element to enforce against.

The framework's resolution matches the TDD intuition exactly where it is right: at leaf scope, the TestSpec is thin by design. A leaf TestSpec case might be one line — TC-006 – rejects negative idle_ttl_seconds, verifies DD-expiry.error.negative_idle_seconds — and the assertion detail lives in the test code. Double-bookkeeping is actively prevented. At branch and root scope the TestSpec carries more, because the test code carries less of the load (fixtures, test doubles, scenarios, environments are not self-documenting the way a well-named JUnit method is). Per-layer weight matches the per-layer self-documentation density of the test code itself.

BDD, Gherkin, Given-When-Then

Most software engineers encounter test-case specification first through Given-When-Then — the Gherkin syntax from Cucumber and SpecFlow, the BDD tradition popularised by Dan North and Liz Keogh. A Gherkin scenario and a TestSpec case are substantively the same object expressed differently:

  • Gherkin Given ↔ TestSpec preconditions
  • Gherkin When ↔ TestSpec inputs + steps
  • Gherkin Then ↔ TestSpec expected
  • Gherkin feature file ↔ TestSpec artifact
  • Gherkin scenario ID (tag) ↔ TestSpec case id + verifies

The differences matter at the edges. Gherkin is optimised for stakeholder-readable acceptance scenarios; its vocabulary is deliberately loose (“a customer with three items in their cart”) and its tooling ties scenarios to executable step definitions. A TestSpec is a spec artifact first and an executable input second; its vocabulary is precise (enumerated equivalence classes, named boundary values, specified error types). Gherkin works naturally at root scope where scenarios map to Product Brief outcomes; below that it gets verbose. TestSpec uses one shape at every layer and lets per-layer weight differ. A team that already writes Gherkin acceptance tests is not expected to discard them — the root-scope TestSpec can cite feature-file scenarios as its steps, with Gherkin tags acting as the case IDs. What the framework requires is that the derivation discipline, verifies links, and coverage matrix exist around the scenarios, which Gherkin alone does not provide.

TDD, ATDD, and DRTDD

TDD (Test-Driven Development, Beck) is code-level red-green-refactor: write a failing test, write the minimum code to pass, refactor. TDD operates inside a leaf; it produces unit tests and unit code in lockstep. The leaf TestSpec is what TDD's tests derive from — the DD names the contracts, the TestSpec derives the cases, the engineer writes the failing test first. TDD without a TestSpec is TDD-as-you-remember-it: the cases are whatever the engineer thought to type. TDD with a TestSpec is TDD-with-a-review-gate.

ATDD (Acceptance Test-Driven Development; also Specification by Example, Adzic 2011) applies the same discipline at the acceptance layer: write the acceptance test first, from the requirement, in stakeholder-readable form. ATDD is the practice that TestSpec formalises at root and branch scope: system-level cases written from Product Brief outcomes and Requirements before any code exists. ATDD + Gherkin + a discipline of verifies links is functionally a root-scope TestSpec.

DRTDD (Design-Requirement-Test Driven Development) is the framework's umbrella extension of TDD: REQUIRE → DESIGN → TEST(red) → IMPLEMENT(green) → REFACTOR → VERIFY. Each phase produces traceable artifacts; human gates separate phases; the TestSpec is the artifact at the TEST(red) phase. DRTDD is TDD with upstream design explicit and downstream verification tracked. A project that runs DRTDD end-to-end produces a TestSpec per scope as a byproduct of normal work.

Test pyramid, honeycomb, trophy

“How many of each test level should a project have?” has three answers in the industry and no consensus. The TestSpec artifact is agnostic about the shape — it specifies per-layer derivation, not per-layer quantity. The three shapes you will encounter:

Pyramid (Cohn) — many fast unit tests, fewer integration, very few e2e. Fits codebases where units are rich in domain logic and integration surface is small.

Honeycomb (Schaffer, Spotify) — many integration tests, fewer unit, few e2e. In service-oriented architectures, most behaviour lives at component boundaries; integration tests of component contracts catch the real defect population.

Trophy (Dodds) — many static checks, many integration, some unit, some e2e. Targets front-end codebases where type systems and lint catch a surprising fraction of classical unit-test bugs.

The framework takes no position on which shape is right for a project. It does require that whatever shape is chosen, every scope's TestSpec derives cases from the layer's own spec artifact, and every spec element has at least one verifying case. The framework's shape discipline is horizontal (same envelope, different weight per layer); the industry's shape debate is vertical (how much of each layer). They are orthogonal concerns.

This page teaches craft that holds across domains

Examples throughout this page use a web-service domain (a session-management service, implementable in Java / Spring Boot but written so the patterns are language-agnostic). This is a choice of shared vocabulary, not a scope limitation. Derivation strategies, coverage bars, test-double strategies, the envelope shape, and the Quality Bar are invariant across domains. What changes is the vocabulary: “integration test” at a branch scope is a Spring Boot @SpringBootTest in a web service, a hardware-in-the-loop test on a microcontroller, a cross-page navigation test on a mobile app, a pipeline-stage test on a batch ETL, a contract test between two gRPC services in a platform. The per-layer weight discipline (thin at leaf, heavier at root) holds everywhere.

What a TestSpec is not

  • Not a test code file. A TestSpec is a spec artifact. The test code is a separate output that satisfies the TestSpec. They do not have the same shape, do not have the same file type, and are not maintained together in the same place.
  • Not a test-case catalogue duplicating the test code. A TestSpec records derivation discipline, not every assertion the test code makes. At leaf scope a TestSpec case is often one line; the JUnit method is allowed to be the specificity.
  • Not a coverage report. Coverage is measured against running test code, retroactively. The TestSpec declares the coverage bar the test code must clear; it does not report what the code did.
  • Not a QA plan or a test strategy document. A test strategy document answers “how does this organisation do testing?” (tooling choices, environments, release gates). A TestSpec answers “what cases verify this scope?” The test strategy is pre-pivot test-planning ceremony; the TestSpec is the per-scope derivation artifact.
  • Not a Gherkin feature file alone. Gherkin scenarios can live inside a root TestSpec's case steps, but a feature file on its own lacks the derivation matrix, verifies links, and per-case strategy tags that the TestSpec requires.

Retrofit posture

A TestSpec retrofitted from existing code is a mixed-evidence document. Cases observed in the existing test suite can be lifted with recovery_status: reconstructed and explicit source (test file path and method name); the coverage matrix is built by mapping existing tests against DD / Architecture / Requirements elements; gaps are recorded as unknown coverage-matrix entries that drive the gap report. The hard rule: a test case's verifies field is not retrofittable by AI inference. A retrofit tool can propose a link (“this test name mentions `expiry`, the DD has `compute_expiry`”) but the assertion that the test actually verifies that element is a human call — retrofit output must mark proposed links as unknown until a human confirms. Fabricating verifies links is the sharpest TestSpec retrofit anti-pattern and the skill layer refuses to emit it. An adjacent hazard: test-as-requirement inversion — a retrofit tool reads the existing tests, generates a synthetic TestSpec that matches them exactly, and then “verifies” the tests against the synthesised spec. That is not a retrofit; that is a loop that asserts existing behaviour is correct because it exists. The retrofit skill must break the loop by deriving TestSpec cases from the DD / Architecture / Requirements independently, then flagging existing tests that do not map as gaps or over-reach.

Why the framework treats this layer as non-negotiable

The TestSpec is where the V closes. Without it, the left-hand descent — Product Brief, Requirements, Architecture, Detailed Design — is design intent with no verified claim that the code satisfies any of it. Code coverage reports show that tests executed lines, not that tests verified intent. The TestSpec is the only artifact that answers, per layer: did we test what this layer's spec said needed testing? It is the last place before deployment where a human can review the verification discipline rather than the verification output. The Spec Ambiguity Test has its sharpest bite here: a TestSpec where a reviewer cannot tell whether every equivalence class, every boundary, every error path was considered — or whether the author stopped when they ran out of ideas — is a TestSpec that fails the meta-gate regardless of how many cases it lists.

2. Best Practices

Nine disciplines. Pre-code specification discipline first, because everything downstream depends on the test artifact existing before the code does. Per-layer emphasis next, because the same envelope carries three different weights and getting the weight wrong produces either double-bookkeeping at leaf or under-specification at root. Then the derivation toolbox and the traceability discipline, the case-quality rules, the test-double strategy, coverage (structural and mutation), the integration-and-system section that carries the content unique to those layers, and finally the envelope shape — the YAML a TestSpec actually renders to.

Frontmatter and shape

A TestSpec is a single Markdown file per scope, colocated with the other spec artifacts at that scope: /specs/testspec.md (root), /specs/{branch}/testspec.md, /specs/{branch}/{leaf}/testspec.md. YAML front-matter identifies the artifact; the body holds a brief overview, a coverage matrix, and a flat list of test-case entries. Canonical front-matter:

---
id: TS-session-store-expiry-calculator
scope: session-store/expiry-calculator
parent_scope: session-store
level: unit                           # unit | integration | system
verifies: [DD-session-store-expiry-calculator]
derived_from: DD-session-store-expiry-calculator
governing_adrs: []
recovery_status: reconstructed        # omit on greenfield
---

The mandatory fields are id, scope, level, verifies, and derived_from. verifies is mandatory non-empty at the TestSpec level and at every case. At root scope scope is null and parent_scope is null; at a branch scope is the branch path; at a leaf scope is the full leaf path and parent_scope is the containing branch. The level field is scope-derived (root → system; non-leaf non-root → integration; leaf → unit) and is carried explicitly so downstream tooling does not have to re-derive it.

2.1   Pre-code specification discipline

A TestSpec is authored before the test code is written, and before the implementation code is written. This is the non-negotiable ordering. The reasons compound across the V.

It forces the spec above to be testable. If a Detailed Design says “the expiry calculator returns the correct expiry for any session,” the TestSpec author will find it cannot derive cases because “correct” is not specified. The TestSpec work exposes the ambiguity before any code locks it in. This is the same discipline TDD claims at the code level — testability is a design quality gate — promoted one layer up to the spec.

It breaks the code-to-test loop. If the implementation exists first, the path of least resistance for an engineer (and the guaranteed path for an AI) is to read the implementation and enumerate cases that match what the code already does. The resulting test suite passes and verifies nothing — the “tautological tests” failure mode named empirically by Yang et al. (2024) and by Meta's ACH deployment (Foster et al., 2025). Authoring the TestSpec first removes the code from the derivation path entirely.

It makes AI authoring reviewable. An AI given a DD and asked to produce a TestSpec has a constrained task: enumerate equivalence classes, boundaries, and errors from the DD, and declare the derivation. The output is short, structured, and human-reviewable before a line of test code is written. An AI given a DD and asked to “write tests” has no such constraint: it will produce plausible-looking JUnit that hits high line coverage and low mutation score, and the review requires reading dozens of test methods and reasoning about what was not tested. The TestSpec is the layer where human-to-AI review leverage is highest.

It anchors traceability before tests exist. The verifies link is declared on the TestSpec case; the test code inherits it by reference. If the TestSpec is authored after the code, traceability becomes an archaeology task: for every JUnit method, work out which DD element it was trying to verify. Authored-first TestSpecs have traceability by construction.

Exception: a pure-retrofit pass on an existing codebase. In that case the code exists and there is no “before.” The retrofit flow derives the TestSpec from the DD / Architecture / Requirements independently, then maps the existing test suite against it, then emits gaps. The code is still not the derivation source — the spec is — and the resulting TestSpec is a specification of what should be verified, independent of what the existing tests happen to cover.

2.2   Per-layer emphasis

One envelope, three weights. The same YAML shape renders a one-line leaf case and a multi-paragraph root scenario. The distinction is not structural — the structural fields (id, title, type, verifies, preconditions, inputs, steps, expected) exist at every layer. The distinction is in which fields carry detail and which are one-liners or omitted.

ScopeLevelDerivation sourceField weight
Leaf unit Detailed Design of this leaf inputs + expected carry the case; steps usually trivial or omitted; preconditions typically one line. Case is often one line; the test code carries the specificity.
Branch integration Architecture of this branch preconditions grows: fixtures, test containers, test doubles, seed data. steps is usually explicit (sequence of cross-child interactions). expected names observable cross-child state, not just return values.
Root system / e2e Product Brief outcomes + root Requirements Full scenario weight. preconditions names environment, tenants, test data, feature flags. steps is an ordered narrative mapping to user journeys or operational flows. expected names outcomes in the Product Brief's vocabulary (not internal state).

Leaf: thin by design, not thin by laziness. The discipline is that the TestSpec's derivation strategy per case is declared (the type field: functional, boundary, error, …), and the verifies link names the DD element under test. The case title plus the type plus the verifies link is usually enough at leaf. The JUnit method name mirrors the TestSpec case title; the AAA block carries the assertion detail; double-bookkeeping is prevented. The TestSpec at leaf is the derivation-discipline layer: which classes were partitioned, which boundaries were named, which errors were enumerated. That is not in the test code and cannot be inferred from it.

Branch: double the weight, more than double the value. At a branch, most of the testing apparatus lives in the preconditions — a test-container with Postgres, a fake upstream, seed data, an auth token with a specific role. The test code cannot self-document this; the precondition block has to. The verifies link at branch scope references either branch Requirements (behavioural cases), Architecture interface DbC clauses (boundary cases), or Architecture composition entries (cross-child / orchestration cases) — multiple targets are common in one case. A branch TestSpec case of the form “TC-005: cleanup-worker and session-api do not race on expired entries, verifies ARCH-session-store.composition.cleanup-api-coordination” is the review gate: it states the composition invariant being tested and names the Architecture element under test. Without the link, a reader has no way to tell whether the case verifies a composition decision or an accident of the current wiring.

Root: written in the Product Brief's vocabulary. A root-scope TestSpec case is readable by a product owner. Its preconditions names environment and persona. Its steps reads as a user journey or an operational scenario. Its expected is phrased as the Product Brief phrased the outcome — “session TTL enforced across devices,” not “Redis TTL key expires.” Root scope is where acceptance happens; the TestSpec is where the acceptance shape is authored. A root TestSpec that reads as a list of API smoke tests has failed the layer; Product Brief outcomes are not API smoke tests.

Quantitative ratios vary; the weight rule is invariant. The industry debate over pyramid / honeycomb / trophy (see §1) is about how many tests of each level a project should have. The per-layer weight rule is about how much the TestSpec case carries at each level. A honeycomb-shaped project has many integration cases and few unit cases; each integration case still carries more envelope weight than each unit case. The weight rule does not constrain the quantity ratio.

2.3   Derivation strategies

The toolbox. Eight strategies, applied in combination. A well-derived TestSpec cites at least one strategy per case via the type field, and the coverage matrix at the top of the TestSpec names which strategies were used overall. The strategies are not ranked — each targets a different failure mode, and stopping at one of them is the characteristic error.

1. Requirement-based testing (RBT). One case per behaviour rule named in the spec. If the DD says “the expiry calculator returns the minimum of the idle expiry and the absolute TTL,” RBT produces at least one case where idle is smaller, one where absolute is smaller, and one where they are equal. If the Architecture says “the cart publishes a PriceChanged event when the pricing component updates,” RBT produces at least one case that asserts the event is published with the expected shape. RBT is the spine of every TestSpec; the other strategies fill in what RBT alone would miss.

2. Equivalence class partitioning (ECP). Partition the input space into classes such that any input from a class should produce “the same kind of” output. Test one representative per class. For the expiry calculator taking an idle_ttl_seconds parameter: one class for positive, one for zero (if the spec accepts it), one for negative (the error class). For a string input: one class for ASCII, one for Unicode BMP, one for Unicode supplementary plane, one for empty, one for null. The discipline is to enumerate the classes at TestSpec time, not to pick examples ad-hoc.

3. Boundary value analysis (BVA). At every input-space boundary, test on the boundary, one below, one above. For idle_ttl_seconds ∈ [1, 86400]: tests at 1, 0, 86400, 86401. BVA catches off-by-one errors, inclusive-vs-exclusive confusion, and silently-different behaviour at transition points. Every numeric range, every array-index bound, every TTL, every retry-count has a boundary. ECP tells you which classes to test; BVA tells you where in each class the bugs hide. Applied together they are empirically the highest-yield pair in test case derivation (cited in multiple safety standards and empirical fault-detection studies).

4. Decision tables. When behaviour is governed by a combination of conditions — “if the session is expired and the user has the refresh-token cookie and the cookie is still valid, reissue; otherwise fail” — a decision table enumerates the condition combinations and the expected outcome per combination. The table is a direct test-case source: each row is a case. For N binary conditions a full table has 2N rows, often prunable with don't-care markers. Decision tables also enable modified condition/decision coverage (MC/DC) derivation: for N conditions, N+1 cases are sufficient to exercise each condition's independent effect. Use a decision table whenever the DD or Architecture has nested conditionals or a compound predicate; do not test compound predicates with ad-hoc ECP alone, because the combinations are where the bugs hide.

5. State-transition testing. For any unit with persistent state — a session, a connection, a circuit breaker, a workflow — the DD names the states and the transitions between them. State-transition testing generates: state coverage (every state reached), transition coverage (every named transition exercised), round-trip coverage (loops back to starting state tested), and sneak-path testing (unspecified transitions attempted, verifying the unit rejects them). A session state machine with states ACTIVE / IDLE / EXPIRED / REVOKED and transitions activate, touch, expire, revoke needs cases for every valid transition, at least one sneak-path case (e.g., touch from REVOKED should reject), and a loop case if one exists.

6. Error paths and fault injection. Every error condition the DD or Architecture names gets a case. Additionally, common implicit fault classes get cases regardless of whether the spec names them: invalid-input variants of every parameter, null / absent / malformed variants, timeout from every external dependency, partial failure from every cross-component interaction, clock skew, resource exhaustion. Fault injection differs from error-path testing in where the fault originates: error-path tests drive invalid input into the unit; fault injection replaces a dependency with a failing fake and checks the unit's response. Both belong in the TestSpec; the type field distinguishes them (error vs fault-injection).

7. Property-based testing. When a unit has an invariant that should hold for any input in a domain, property-based testing (QuickCheck, Hypothesis, fast-check, jqwik) generates random inputs from the domain and asserts the invariant holds. For the expiry calculator: for any valid inputs, the returned expiry is ≥ now and ≤ now + absolute_ttl. A single property-based test covers an input range that ECP+BVA would hit only at representative points. Property-based testing earns its keep when: (a) the invariant is easy to state, (b) the input domain is large, (c) the output is easy to check algorithmically. It does not replace RBT+ECP+BVA — random generation misses rare edge cases those enumerate explicitly — but it catches the “I didn't think of that input” class of bug that enumeration misses. Declared in a TestSpec case with type: property, a generator spec, and the invariant as the expected predicate.

8. Error guessing / experience-based. The lowest-priority strategy and still worth including explicitly. An experienced engineer reading a DD will spot failure modes the other strategies miss: “this retry-with-jitter logic will thundering-herd under partial failure,” “this ID generator will collide under high concurrency within the same millisecond.” Error guessing captures those as explicit cases. It is the least systematic strategy and the easiest to miss on retrofit (the hunches are in the original author's head, not in the artifacts). Its value is as a complement, not a substitute. A TestSpec case derived from error guessing should declare type: error-guessing and carry a note pointing at the hypothesised failure mode, so a reviewer can evaluate whether the guess is well-founded.

Stopping criterion. Coverage is complete when the coverage matrix maps every spec element to at least one case, every applicable strategy has been applied (RBT always; ECP+BVA for any input range; decision tables for any compound predicate; state-transition for any stateful unit; error paths for every named error; fault injection for every dependency), and the Spec Ambiguity Test meta-gate passes (a reviewer reading only the TestSpec can tell whether coverage is adequate). Stopping when the clock runs out is the characteristic failure.

2.4   Traceability and verifies

The verifies field is mandatory non-empty on every TestSpec case. This is the anti-orphan rule: a test case without a traceable intent is either misplaced (verifying something that is not in the scope's spec) or testing unspecified behaviour (which means either the spec is incomplete or the case is pinning an implementation accident). Both are defects. The tooling refuses to accept a TestSpec case with an empty verifies list.

What verifies points at. A stable artifact ID at the layer's spec: a Requirement ID (REQ-042), an Architecture child or interface (ARCH-cart.children.pricing, ARCH-cart.interfaces.price-changed), a Detailed Design element (DD-expiry.behavior.compute_expiry, DD-expiry.error.negative_idle_seconds, DD-expiry.invariants.2). A Product Brief outcome (PB.outcomes.secure-session-lifecycle) at root scope. Never a commit SHA, never a test file path, never a Jira ticket — those are ephemeral. The framework's traceability model (TARGET_ARCHITECTURE §7) specifies artifact-level default granularity with field-level allowed specifically for verifies, which is why a case can point at a specific invariant or error field rather than just the DD.

Granularity at leaf. At leaf scope, verifies should resolve to a specific DD field, not the whole DD. A case pointing at DD-expiry is less reviewable than a case pointing at DD-expiry.behavior.compute_expiry. If a DD has ten behaviour rules and twenty cases all verify the whole DD, the coverage matrix cannot distinguish well-covered rules from uncovered ones.

Granularity at branch. At branch scope, verifies references branch Requirements (REQ-{branch}-{n}), Architecture interface DbC clauses (ARCH-{branch}.interface.{name}.{clause}), or Architecture composition entries (ARCH-{branch}.composition.{name}). Decomposition entries are not valid targets — they are structural, not behavioural. Multiple targets are common because integration tests often exercise both behavioural intent (a REQ) and a specific architectural surface (an interface or composition entry) in one scenario. The list may be long; it must still be specific.

Granularity at root. At root scope, verifies references root Requirements and Product Brief outcomes. An end-to-end case typically verifies multiple Requirements and one outcome. 2.5   Case quality: F.I.R.S.T., AAA, assertion specificity

The TestSpec declares the case; the test code implements it. The case-quality disciplines below are properties of the test code, declared as expectations in the TestSpec so the test-authoring step (human or AI) has a concrete bar to hit.

F.I.R.S.T. (Robert C. Martin). Tests are Fast (milliseconds for unit, seconds for integration, minutes at most for e2e), Independent (no test depends on another's state or order), Repeatable (deterministic — inject clocks, seed random, control external sources), Self-validating (a single boolean pass/fail from the test runner, not a log line a human inspects), Timely (authored close in time to the code, ideally before it). Declared on the TestSpec as default expectations at each level, with exceptions explicit. A test that pauses for 30 seconds to “wait for cache to warm” violates Fast and Repeatable and is a Quality Bar failure regardless of whether it technically passes.

Arrange-Act-Assert (AAA), aka Given-When-Then. Every test has three visually separated blocks: Arrange (set up inputs, state, dependencies); Act (call the unit under test exactly once); Assert (verify outcome matches expected). The one-Act rule is load-bearing: a test that calls the unit twice is testing two behaviours, and when it fails, diagnosing which call failed requires re-reading the test. Parameterised tests are fine (JUnit 5 @ParameterizedTest, pytest @pytest.mark.parametrize) — they run the same AAA block with different data. Gherkin's Given-When-Then is AAA under a different name and is directly equivalent.

Assertion specificity. Every test asserts a specific expected value, not a vague property. assertEquals(expected, actual) with an enumerated expected, not assertNotNull(actual). assertEquals("2026-04-21T12:30:00Z", result.expires_at), not assertTrue(result.expires_at.isAfter(now)). Vague assertions pass for incorrect implementations — a function that always returns the current time plus one second satisfies isAfter(now) for every input. The TestSpec case's expected block must name the specific value; if it cannot (because the spec does not specify the value), the spec has a gap that the TestSpec surfaces. Empirical: weak-assertion tests are the second-largest failure mode in LLM-generated test suites (Yang et al., 2024); the fix is on the author side, not the tool side.

One logical concept per test. Multiple assertions on the same outcome are fine (assert the session's id, then expires_at, then principal on the same returned object). Multiple assertions on unrelated outcomes are not — a test that asserts both “expiry is computed correctly” and “the logger emitted a debug line” is testing two behaviours, and when it fails, the failure message is ambiguous. The TestSpec case's expected block scopes the outcome under test; a case that needs two expected blocks is two cases.

Test naming. Test names describe the scenario, not the method being tested. rejects_negative_idle_ttl_seconds over testComputeExpiry. The JUnit method name mirrors the TestSpec case title. At read time, the list of test method names in a file should read as an informal spec of the unit's behaviour. This is where TDD's “tests as living documentation” claim actually earns itself — in the test code. The TestSpec is not trying to replace it; the TestSpec is the layer above.

2.6   Test doubles strategy

A test double is any object substituted for a real collaborator in a test. Gerard Meszaros's canonical taxonomy names five types, ranked by how much they do:

  • Dummy — filler, never exercised. Used to satisfy a parameter list.
  • Stub — returns canned answers. Used to supply inputs from an indirect source.
  • Spy — records calls for later inspection. Used when the interaction itself is observable behaviour.
  • Mock — pre-programmed with expectations; fails if the expected calls do not occur. Used for interaction-based verification.
  • Fake — working but simplified implementation (in-memory database, fake HTTP client). Used when a real collaborator is too heavy for fast tests.

Preference order: real > fake > stub > mock. Prefer the real collaborator whenever it is deterministic, fast, and cheap to set up. When it is not, prefer a fake (Google's canonical preference; Martin Fowler's “Mocks Aren't Stubs” endorses the same hierarchy). Prefer a stub to a mock whenever the test can verify outcome (state) rather than interaction (calls made). Mocks are a last resort, used only when the interaction itself is the observable behaviour — “the notification was sent” is an interaction; “the discount was applied” is a state.

Don't mock what you don't own. Freeman and Pryce's rule (Growing Object-Oriented Software, Guided by Tests, 2009). If a third-party API is hard to test against, wrap it in an adapter your code owns, then mock the adapter. Mocking the third-party's interface directly couples your tests to the third-party's API shape, which changes out from under you.

Contract-test every fake. A fake is a liability if it drifts from the real implementation. The discipline: the fake and the real share a contract-test suite — the same test cases run against both. Whenever the real's behaviour changes, the contract-test suite catches the fake's drift. Applies across the boundary: if your service uses a fake Redis in tests, run the same contract tests against the real Redis in CI at a lower frequency. Contract-tested fakes are the highest-leverage form of test double; uncontracted fakes are a delayed bug.

Over-mocking is a design smell. If a unit requires more than two or three test doubles to test, the unit is doing too much or is wired too tightly to its collaborators. The TestSpec case signals this: if the preconditions block names five doubles, the authoring skill surfaces the coupling as a design issue back to the author rather than writing the test. Fixing the coupling in the DD produces a simpler unit with a simpler TestSpec; mocking through the coupling produces tests that pass in isolation and break in production (“all my mocks agreed; production disagreed” is the canonical symptom).

Per-layer usage. Heaviest at leaf (unit tests use doubles for anything beyond the unit itself); bounded at branch (integration tests use real instances of the children being integrated, doubles only for out-of-scope dependencies like external services); minimal at root (system tests run against production-like environments with real instances of everything, doubles only where a real third-party would be unsafe to exercise — a payment gateway's test mode qualifies; a real payment gateway does not).

2.7   Coverage: structural and mutation

Coverage measures the test code against the production code, retroactively. The TestSpec declares the coverage bar: the minimum percentages the test code must clear. Two independent dimensions matter, and measuring only the first is the single largest silent-failure mode in software testing.

Structural coverage is the familiar metric set: statement coverage (every line executed), branch coverage (every branch taken both ways), decision coverage (every boolean outcome reached), modified condition/decision coverage (MC/DC — every condition independently exercised within every decision). They form a strict hierarchy: MC/DC subsumes decision, decision subsumes branch, branch subsumes statement. Tools like JaCoCo, Istanbul, Coverage.py report them; CI systems enforce thresholds. Structural coverage is necessary but insufficient: it answers “did my tests execute the code?” — which is a lower bar than “did my tests verify the code is correct?” It is also not a derivation method: writing tests to hit a coverage number produces tests that execute the code they were written against, without verifying anything else. A TestSpec declares a structural coverage bar (branch coverage ≥ 85%, say) as a floor; hitting it does not mean the tests are good.

Mutation coverage closes the gap. A mutation is a small, systematic change to the production code: flip a conditional, change an arithmetic operator, remove a statement, replace a boolean literal. The mutation testing tool runs the full test suite against each mutated version; if the tests still pass, the mutation survived — the tests did not detect the change. The mutation score is the fraction of mutations killed. A mutation score close to 100% means the test suite is precise; a low mutation score with high line coverage means the tests execute code without verifying its behaviour. PIT (Pitest) for JVM, Stryker for JavaScript/TypeScript, mutmut for Python are the canonical tools. Meta's ACH industrial deployment (Foster et al., FSE 2025) ran mutation-guided test generation across 10,795 classes with 73% engineer acceptance; MutGen (the same programme's mutation-guided LLM approach) achieved 89.5% mutation score against a naive-LLM baseline of ~4% on the same targets.

Why this is load-bearing. Yang et al. (ASE 2024) evaluated five LLMs on unit-test generation: 34–62% of generated tests were syntactically invalid (did not compile or threw at load time). Among the tests that did compile, tautological tests were the dominant failure mode: the test recomputed the expected value using the same logic the implementation uses, asserted equality, and passed vacuously. The canonical pathology: a test suite hits 100% line coverage with a 4% mutation score. The code was executed; nothing was verified. GitHub Copilot in an uncontrolled setting: 7.55% passing tests; with prompt engineering (context about the spec), 45.28%. Diffblue and similar specialist tools: ∼100% compilation but lower mutation scores than human-authored tests. The empirical consensus across the 2023–2026 literature is that coverage alone is a misleading metric for AI-generated tests and mutation testing is the viable complement.

Thresholds. The TestSpec declares both a structural coverage bar and a mutation-score bar. Defensible defaults: branch coverage ≥ 85%, mutation score ≥ 60% as the minimum-viable floor, 80%+ as the target. These numbers are load-bearing per the empirical literature above; they are not institutional preference. A project that declares only a line-coverage bar is declaring it is comfortable shipping tautological tests. A project that declares a mutation-score bar and enforces it in CI has the only technical defence against the coverage-mutation gap. Declare bars per layer: leaf TestSpecs usually carry the highest bars (the test code is cheapest and the mutation testing is most tractable); integration and system TestSpecs often carry lower structural bars because the test surface is wider.

Adoption. Mutation testing is CPU-intensive (each mutation runs the suite; a codebase with 10k mutations is 10k test runs). Standard patterns: run mutation testing nightly rather than per-commit; run it only on changed files per PR; run it on a sampled subset per commit with the full sweep nightly. The TestSpec bar does not mandate the CI frequency; it mandates the floor the mutation score must clear when measured. A nightly mutation run below the declared bar is a failing build the next morning.

2.8   Integration and system testing specifics

Branch and root TestSpecs carry disciplines that leaf TestSpecs do not encounter. Four families: contract testing between components, test-environment specification, end-to-end scenario authoring, and the specialised test categories (performance, security, accessibility) that live in the same envelope with some extra fields.

Contract testing. When a branch decomposes into children that talk across an interface, the interface is a contract. Integration tests verify the contract from both sides: the provider honours its advertised postconditions, the consumer obeys its stated preconditions. In a distributed system where producers and consumers evolve independently, consumer-driven contract testing (Pact, Spring Cloud Contract, schemathesis against OpenAPI) is the mechanism that keeps them honest: consumers declare the subset of the provider's contract they actually depend on; the provider runs the consumer-supplied contract tests against its implementation; the contract is violated when either side drifts. In the TestSpec, a contract test is an integration-level case with type: contract, verifies pointing at the Architecture interface entry, and preconditions naming the contract artifact (Pact file, OpenAPI spec version, Avro schema) under test. A branch TestSpec that omits contract tests between its children will admit interface drift silently.

Test environment specification. Integration and system tests run against environments: composed infrastructure that stands in for some or all of production. The TestSpec at branch and root scope names the environments its cases require. Canonical forms: in-process (all children loaded in one JVM / process; fastest, least faithful), test-containers (real infrastructure in containers, spun up per-test-run; medium speed, high faithfulness for data stores and queues), shared staging (long-lived environment shared across test runs; slow, highest faithfulness for deployments), production-like preview (ephemeral environment per PR; slowest, closest to production). The TestSpec's preconditions block names the required environment and its essential shape (Postgres 15 with schema X loaded, Redis 7 with AOF enabled, three service instances behind a load balancer). A case that runs in the wrong environment is not the case it claims to be.

End-to-end scenario authoring. At root scope, cases are narratives: a user does X, then Y, then Z; the system's state at the end satisfies the Product Brief outcome. The authoring discipline is to keep the narrative at the stakeholder level — the steps block describes what a user does, not which API endpoints the test harness calls; the expected names outcomes in the Product Brief's vocabulary. When a scenario has multiple stakeholder actors (admin, customer, system), the case names them explicitly in preconditions. Production-like data matters: a root-scope case running against an empty database is testing plumbing; the same case running against a realistic data volume is testing the system. The TestSpec at root scope declares the data shape required.

Performance testing. Same envelope; the type field is performance, and the expected block names thresholds (p99 latency ≤ 200ms at 1000 req/s; throughput ≥ 5000 writes/s; memory steady-state ≤ 2GB over a 1-hour soak). preconditions names the load profile and duration. Performance tests at branch scope target internal interfaces (the cart service's addItem path); at root scope target user-facing flows (login-to-first-page latency). Declared in the TestSpec with the same verifies discipline — pointing at a Requirements quality attribute or a Product Brief outcome. A performance test with no verifies link is a benchmark, not a verification.

Security testing. Same envelope; type is security and the cases derive from a threat model (STRIDE, or an organisation-specific model) applied to the scope. Examples: authentication bypass attempts on each externally visible endpoint; authorisation boundary tests for each role; input fuzzing for injection classes (SQL, LDAP, command, XSS); session fixation and CSRF; rate-limit enforcement. Static analysis (SAST) and dynamic scanning (DAST) are tool-level complements; they live outside the TestSpec per se, but a TestSpec can reference their thresholds (preconditions: DAST clean on OWASP-top-10 classes). Security tests are derivable from a threat-model artifact; if the project has one, cases verifies its threat IDs. If the project does not have one, the Requirements quality attributes cover the floor and the TestSpec can only verify what is named.

Accessibility testing. Same envelope; type is accessibility; derivation is from WCAG criteria or an equivalent internal standard. Cases verify keyboard-only navigation, screen-reader label completeness, focus management, colour-contrast ratios, ARIA landmark correctness. Automated tools (axe-core, pa11y, Lighthouse) cover a subset; the rest requires human verification, which a TestSpec can describe as an accessibility review case with a checklist rather than an automated assertion. Like security, accessibility tests derive from Requirements quality attributes; they are not a separate artifact family.

Closing gaps left by leaf-level test doubles. The discipline: whatever a leaf test double elided (“the unit test used a fake database, so database-real behaviour is not verified here”) must be picked up in a branch or root case against the real thing. The TestSpec records this as an explicit trace — a leaf TestSpec note (“DB interactions verified at branch TestSpec case TC-branch-007”) and the branch case with verifies back to the same DD element. Closing the loop is how test doubles stay safe to use: a double is a promise that the real thing is verified somewhere else, and the TestSpec records where.

3. Anti-Patterns

Thirteen named failure modes. Each is a specific tell the authoring or review skill looks for. The numbering is stable across revisions; skills cite them by number.

  1. Code-to-test derivation. TestSpec cases enumerated by reading the implementation and describing what it does. The tell: case expected values match the implementation's output exactly even where the spec is silent; cases exist for behaviour the DD does not mention. The tests always pass; no bug can be found by this suite. Fix: derive from the spec, not the code.
  2. Tautological tests. The test recomputes the expected value using the same logic as the implementation, then asserts equality. The tell: the test body looks like a copy-paste of the production function with assertEquals around it. Empirically the dominant failure mode in LLM-generated test suites (Yang 2024). Fix: hard-code expected values from the spec; never compute them in the test.
  3. Test-as-requirement inversion. Retrofit tool reads the existing tests, synthesises a TestSpec that matches them, then claims the tests verify the spec. The loop asserts existing behaviour is correct because it exists. The tell: every existing test has a perfect verifies target; no gaps surface; every behaviour in the synthetic spec is test-mirrored. Fix: retrofit derives TestSpec from DD / Architecture / Requirements independently, then maps existing tests against it, surfacing mismatches as gaps or over-reach.
  4. Happy-path bias. Cases cover the primary behaviour and a few obvious errors. Boundaries, fault injection, concurrency, resource exhaustion, unusual input encodings, and long-running behaviour are missing. The tell: the error-path-to-happy-path case ratio is below 1:2. Empirically systematic in AI-generated suites and common in human ones under time pressure. Fix: apply ECP+BVA+error-paths+fault-injection explicitly; measure the ratio during review.
  5. Weak assertions. assertNotNull, assertTrue(true), instanceof, “does not throw” as the sole assertion. Passes for any implementation that returns a non-null same-type object, including a wrong one. The tell: search for assertNotNull with no other assertion on the same object; count the fraction of cases whose expected is qualitative rather than an enumerated value. Fix: every case asserts a specific expected value named in the TestSpec's expected block.
  6. Over-mocking. A leaf case's preconditions names four or more doubles; the resulting test is a puppet show in which the mocks agree with the mocks. The tell: any unit that requires more than two or three doubles to test; any verify()-heavy test suite at the leaf layer. Fix: revisit the DD for coupling; prefer fakes over mocks; move interaction verification to branch scope where real collaborators live.
  7. Mystery guest / hidden fixtures. Cases depend on data or state whose shape is not visible in the TestSpec — a database fixture referenced by name with no schema, a file in test/resources/ referenced but not described. The tell: a reviewer cannot understand the expected without loading the external fixture. Fix: data and fixtures named with their essential shape inline or in a referenced data block within the TestSpec.
  8. Ice-cream-cone coverage. Many e2e tests, few unit tests. Tests are slow, flaky, expensive to maintain; defects land in production because the ones that should have been caught by fast unit tests were never written. The tell: leaf TestSpecs thin or absent; root TestSpec covers half of what unit tests should cover. Fix: push verification to the lowest layer that can reach it; a behaviour verifiable in a unit test does not belong in an e2e test.
  9. Coverage-as-quality-metric. The CI pipeline enforces a line-coverage threshold; the TestSpec declares no mutation bar. The test suite clears the line threshold with tautological tests and ships. The tell: the TestSpec's coverage bars name only structural metrics; no mutation threshold exists. Empirically 100% line / 4% mutation is a reachable state and catastrophic. Fix: declare and enforce a mutation-score bar in parallel with structural.
  10. Unbounded negative tests. A case asserts “the function never errors” or “the session never leaks.” Unbounded negatives are infinite in input space and cannot be verified. The tell: expected contains never, always, or for all without a bounded domain. Fix: bound the negative — “for inputs in this enumerated class, the function does not error” — so the test becomes finite and exhaustive over its class, not over the universe.
  11. Flaky tests left in place. A test that passes most of the time and fails intermittently without a real bug. The team learns to rerun failed tests; the signal is lost; real regressions land unnoticed. Google data: 84% of pass-to-fail transitions are flaky, not regressions. The tell: any case that cannot be run independently and repeatably. Fix: delete (Kent Beck's prescription) or fix the determinism root cause (inject clocks, seed random, isolate shared state, bound async with explicit waits, never real-network from unit tests).
  12. Orphan tests (verifies empty or unresolvable). Cases exist without a traceable intent, or verifies points at an ID that no longer exists. The tell: traceability validator reports cases with empty or broken verifies. Either the case is testing unspecified behaviour (which means the spec is incomplete or the case is pinning an implementation accident) or the referenced spec element was renamed / removed and the case was not updated. Fix: every case has a non-empty verifies resolving to a live ID; tooling enforces at submit.
  13. Fabricated retrofit intent. In a retrofit TestSpec, a case's title or notes claims the test was written to verify X, where X was inferred by an AI from the test name or body. The tell: every existing test has a plausible-sounding stated intent; retrofit output exhibits no unknown or unmapped cases. Hard rule violation: test intent is human-only in retrofit; AI proposals carry recovery_status: unknown until a human confirms. Fix: the retrofit skill refuses to emit stated intent on human-only fields; reviewer must supply or mark unknown.

Case bodies are short; the DD carries the behaviour, the TestSpec carries the derivation discipline, the JUnit method will carry the assertion syntax. 4.5   Decision-table derivation walkthrough

From the DD for session creation: accept if credentials valid AND principal not rate-limited AND principal has fewer than 5 active sessions; otherwise reject with a specific error code. Three conditions, three independent predicates, compound output. Decision table:

| Rule | credentials_valid | not_rate_limited | under_5_sessions | Outcome                               |
|------|-------------------|------------------|------------------|---------------------------------------|
| R1   | T                 | T                | T                | accept, 201 — issue session token       |
| R2   | T                 | T                | F                | reject, 429 — too many concurrent       |
| R3   | T                 | F                | —                | reject, 429 — rate limited (don't-care)|
| R4   | F                 | —                | —                | reject, 401 — invalid credentials       |

4.6   Bad vs good — code-to-test vs spec-to-test

Two versions of a leaf case for the expiry calculator. The DD says “compute_expiry returns the earlier of last_touched + idle_ttl_seconds and created_at + absolute_ttl_seconds.”

Bad — code-to-test:

- id: TC-101
  title: compute_expiry returns correct result
  type: functional
  verifies: [DD-session-store-expiry-calculator]
  inputs:
    last_touched: "2026-04-21T11:45:00Z"

    created_at: "2026-04-21T10:00:00Z"
    absolute_ttl_seconds: 86400
  expected:
    expires_at: "<whatever the implementation returns>"   ← computed from code, not spec
    is_expired: "<whatever the implementation says>"       ← same
  notes: "Derived by running the implementation with these inputs and recording the output."

Good — spec-to-test:

- id: TC-001
  title: idle-based expiry when idle_ttl is earlier than absolute_ttl
  type: functional
  verifies: [DD-session-store-expiry-calculator.behavior.earliest_of_idle_and_absolute]
  inputs:
    last_touched: "2026-04-21T11:45:00Z"
    idle_ttl_seconds: 1800
    created_at: "2026-04-21T10:00:00Z"
    absolute_ttl_seconds: 86400
  expected:
    expires_at: "2026-04-21T12:15:00Z"   ← derived from DD: last_touched + 1800s
    is_expired: false                     ← now (12:00) is before expires_at
  notes: "Derived from DD. idle_ttl yields 12:15; absolute_ttl yields next day; DD says earlier wins."

The bad version is vague about what the expected value depends on; its verifies points at the whole DD rather than a specific behaviour rule; its notes admit the provenance is the implementation. The test will always pass because the expected equals what the code produces, whatever the code is. The good version derives the expected from the DD's stated rule, cites the specific rule, and will fail if the implementation gets the “earlier of the two” logic wrong in any direction. Both cases superficially look like tests. Only one verifies anything.

4.7   Bad vs good — fabricated retrofit vs honest retrofit

A retrofit pass on an existing Java test in a legacy codebase. The original JUnit method:

@Test

The bad version reads smoother because an AI invented the stated intent. 5. Quality Bar

Yes/No checklist, grouped by concern. A TestSpec passes when every applicable item is answered Yes. Two cards carry a visual accent: Coverage-mutation independence (where measuring only structural coverage is the single-largest silent-failure mode in software testing) and the Spec Ambiguity Test (the meta-gate overriding everything above).

Derivation discipline

  • Are the derivation sources named explicitly in derived_from and do they match the layer per Position C (PB + root Req + root Arch at root; branch Req + branch Arch at branch; DD at leaf)?
  • Was the TestSpec authored before the test code and before the implementation code (or, in retrofit, independently of the existing tests)?
  • Does every case declare a type from the derivation-strategy vocabulary (functional / boundary / error / fault-injection / state-transition / property / contract / performance / security / accessibility / error-guessing)?
  • Has every applicable strategy been applied: RBT always; ECP+BVA for every input range; decision tables for every compound predicate; state-transition for every stateful unit; error paths for every named error; fault injection for every dependency?
  • Is the error-path-to-happy-path case ratio ≥ 1:2?

Case shape

  • Does every case have a non-empty verifies resolving to live spec IDs at this layer?
  • Is every case's expected a specific value or a bounded invariant, not a qualitative non-negative assertion?
  • Does every case carry a title that describes the scenario rather than the method under test?
  • Is every case independent, repeatable, and deterministic (F.I.R.S.T.), with clocks and random seeds named where relevant?
  • Is the one-Act-per-test discipline declared and the steps block respecting it (at layers where steps is present)?

Per-layer weight

  • At leaf: are cases thin (title + type + verifies + inputs + expected, with steps usually omitted)?
  • At branch: do preconditions name fixtures, doubles, and environment, and does steps enumerate the cross-child interactions?
  • At root: do cases read as user journeys, with expected phrased in the Product Brief's vocabulary rather than internal API terms?
  • Are integration/system cases that close leaf-level test-double gaps explicitly traced (leaf case notes the closer case; closer case verifies back)?

Traceability and completeness

  • Is every spec element at this layer covered by at least one case in the coverage matrix?
  • Are uncovered elements recorded as gaps, not hidden?
  • Does verifies granularity match the layer (DD element at leaf; REQ / Architecture interface DbC clause / Architecture composition entry at branch and root; PB outcome additionally at root)?
  • Has the traceability validator returned zero orphan cases and zero unresolved references?

Coverage-mutation independence — load-bearing

  • Does the TestSpec declare a structural coverage bar (branch ≥ 85% is a defensible default) enforced in CI?
  • Does the TestSpec declare a mutation-score bar (60% minimum viable / 80% target) enforced in CI at a named frequency?
  • Is the mutation-testing tool named (PIT, Stryker, mutmut, …) with its scope (files / modules / per-PR subset)?
  • Have weak-assertion and tautological-test audits been run (e.g., grep for assert-only-not-null; ratio of hard-coded expected values vs computed ones)?
  • Is it understood and documented that 100% line coverage with a 4% mutation score is an achievable state that ships no verification?

Test-double discipline

  • Does every test double in a leaf case's preconditions have a named type (dummy / stub / spy / mock / fake) and a reason?
  • Does every fake have a contract-test suite declared against the real implementation, run at a named frequency?
  • Is no leaf case using more than two doubles (or, if more, has the coupling been flagged as a design issue)?
  • Is interaction verification (mocks, spies) reserved for cases where the interaction itself is the observable behaviour?

Integration and system specifics

  • At branch scope: does every child-to-child interface have at least one contract or integration case?
  • At branch / root scope: does every case name the environment shape its preconditions require?
  • Are specialised cases (performance, security, accessibility) present where the Requirements quality attributes call for them?
  • Do specialised cases declare thresholds, derivation source (threat model, WCAG, SLO budget), and verification method?
  • Is contract-testing active where the scope has external consumers (Pact / OpenAPI / Avro), and are contract versions pinned in preconditions?

Retrofit discipline

  • In retrofit, are TestSpec cases derived from the DD / Architecture / Requirements first, and existing tests mapped against the result afterwards?
  • Are proposed verifies links for observed tests marked recovery_status: unknown until a human confirms?
  • Is stated intent (title, notes) limited to observed behaviour when the original intent is not recoverable?
  • Is the gap report populated (uncovered spec elements, tests not mapping to any element, observed-but-suspect cases)?

Spec Ambiguity Test — meta-gate

  • Could a junior engineer or mid-tier AI, reading only this TestSpec (plus the layer's spec artifact and governing ADRs), write test code that implements every case as specified, and — crucially — would a reviewer reading only the TestSpec be able to tell whether every equivalence class, every boundary, every error path was considered, without having to read the implementation or the test code?

If the answer is No, the TestSpec under-specifies its derivation or misses applicable strategies. Revise before accepting. This test overrides every box above: a TestSpec that passes structural checks but fails this one has not done the job TestSpec exists to do.
