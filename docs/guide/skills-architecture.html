Skills & Workflow Architecture

How the framework's 18 skills and 3 worker agents compose into the two coupled workflows — Spec Flow (humans + AI authoring specifications) and Build Flow (AI dispatching code + tests against those specifications). This page is the navigational map: what each piece does, what it reads, what it writes, and how a user runs the framework end-to-end on a real project.

The framework follows three load-bearing rules:

Workflows are independent

Spec Flow, Build Flow, and Retrofit are loosely coupled through artifacts only. No workflow calls, subscribes to, or has runtime knowledge of another. The Spec Flow emits artifacts; the Build Flow consumes them; there is no "spec-complete" event — Build can start whenever it has enough.

Skills are the canonical procedure

Skills hold the procedure, references, templates, and HALT conditions. They are what a human reads to understand what the framework does. Agents are thin dispatch shells that exist to give workers an isolated context window and enable parallelism — they delegate everything substantive to a skill.

Handovers are file-based

Every boundary — author ↔ review, render ↔ implement ↔ review-execution, orchestrator ↔ worker — is a YAML or Markdown file on disk under .vmodel/. No in-memory state survives a tool boundary. Any session can crash and resume from the last written file.

Where this fits in the framework

The framework has four independent components: documentation, templates & schemas, traceability, and AI skills. This page documents the AI skills component and how the artifacts and workflows tie together at runtime. For per-artifact craft (what a good Requirements doc looks like, what a good ADR looks like) see the per-artifact pages in the sidebar. For the architectural rationale behind every decision, see docs/plan/TARGET_ARCHITECTURE.md.

Skill / Agent / Tool Split

Three categories with different jobs. Knowing which is which is how the framework keeps each piece small and replaceable.

CategoryJobWhat it knowsLives at
Skill Interpretive, judgment-driven procedure (LLM-driven). Procedure, references, templates, HALT conditions, hard refusals. .claude/skills/<skill>/SKILL.md + references/ + templates/
Agent Thin dispatch shell run via Task tool. Loads one canonical skill in an isolated context. Only the dispatch envelope shape (worktree path, task id, mode). No procedure. .claude/agents/<agent>.md
Tool Mechanical, deterministic check (no LLM). Same input → same output. Schema validation, link integrity, ID encoding, defer-index aggregation, etc. scripts/check-*.py, scripts/index-deferred-items.py, plus the vmodel-core external tool product.

Why agents wrap skills

The build flow needs parallel per-task workers with isolated context windows. Three things follow from that:

  • Isolated context. A worker reading one leaf's DD + tests + source should not pollute its sibling worker's context. The Task tool gives each invocation a fresh context.
  • Parallelism. The orchestrator emits multiple Task tool calls in a single tool-use message — the runtime executes them concurrently, bounded by build.parallel.max_concurrent.
  • Failure isolation. A worker crash does not affect siblings or the orchestrator.

The agent file itself is < 50 lines — it just declares the dispatch envelope and tells the runtime to load the canonical skill. All procedure lives in the skill. The three workers (render-tests, implement-leaf, review-execution) are wrapped this way; retrospect-build stays a Skill invocation because it's a single end-of-run call with no parallelism need.

Why spec-side skills are not wrapped

Spec authoring is interactive — the human discusses, the skill drafts, the reviewer verdicts, the human approves or pushes back. There is no parallelism story to enable, and the human wants the work in their own context. So spec skills are user-invoked directly (/vmodel-skill-author-architecture), not dispatched as agents.

Lifecycle distinction. Spec-side review verdicts (.vmodel/.reviews/) are committed — they are a forensic record of design-time decisions and have lasting value. Build-side runtime state (.vmodel/.build/ tasks, progress, pipeline state) is gitignored — it is transient operational state. The two committed exceptions on the build side are escalations/ (spec-gap evidence) and lessons.yaml (cumulative learning).

Every artifact carries its outgoing links in front-matter. Reverse relationships (realized_by, verified_by, derives) are computed by tooling — never stored — to avoid the DOORS-era "central table drifts from documents" problem.

Per-task file contract (build flow)

Each leaf build task owns a directory under .vmodel/.build/tasks/<task-id>/. The three workers (render-tests, implement-leaf, review-execution) communicate exclusively through files in that directory. Every file has exactly one producer.

FileProducerConsumerPurpose
current-task.yaml orchestrate-build All three workers Task contract — acceptance_criteria, context_to_load, out_of_scope copied verbatim from tasks.yaml; carries mode (greenfield/fix/resume), attempt, amendments_used, optional resume_from_step.
render-report.yaml render-tests Orchestrator (manifest) Cases rendered, cases skipped, compile and red-phase confirmation.
build-progress.yaml implement-leaf (overwrite-on-update) Orchestrator (resume only) Per-gate journal. last_step across red / green / refactor / lint / coverage / self-check / review-ready-written. Drives silent-recover / resume-at-gate / restart on resume.
build-blocked.yaml implement-leaf Orchestrator Scope-expansion HALT. blocker_type, needed_writes / needed_reads, suggested_resolution. Routed to auto-amend (within budget) or escalate.
review-ready.yaml implement-leaf (sole producer) review-execution Implementation handoff: files changed, contracts implemented, lint / coverage / test status.
feedback.yaml review-execution (REJECTED only) implement-leaf (fix mode) failures[] with type, location, description, required_fix, spec_ref. Taxonomy: contract-violation, scope-violation, missing-implementation, wrong-assertion-is-impl-bug, integration-failure-impl-bug, regression.
ESC-NNN.yaml review-execution (ESCALATE only) Orchestrator → spec-side author skill Layer-typed escalation. Mirror copy lives in .vmodel/.build/escalations/ (committed).

APPROVED writes no file. review-execution emits a single stdout line APPROVED <task-id>; the orchestrator infers APPROVED from the absence of feedback.yaml and any new ESC-NNN.yaml after dispatch. The implementation's review-ready.yaml remains as the record of what was implemented.

Mode B — author with review feedback

When an author skill is invoked on an existing artifact, it reads the latest review file at .vmodel/.reviews/<artifact-id>-*.yaml (lexically last). Each finding is addressed in the revision — applied, pushed back with rationale, or deferred via an explicit marker. The revision narrative names which findings were addressed and how.

Build Pipeline Shape

The pipeline runs bottom-up the V — leaves first (unit tests + code), then branches (integration tests), then root (system tests). Within a stage, tasks run in parallel up to build.parallel.max_concurrent.

Pipeline plan-build (user-invoked) └─► writes .vmodel/.build/tasks.yaml orchestrate-build (user-invoked) ├─► Stage: leaf builds (parallel within stage, sequential across stages by dependency) │ for each leaf task: │ Task → render-tests (worker agent, isolated worktree) │ Task → implement-leaf (worker agent, isolated worktree) │ Task → review-execution (worker agent, isolated worktree) │ ├─► Stage: branch integration (bottom-up by tree depth) │ for each branch: │ Task → render-tests (branch TestSpec → integration tests) │ Task → review-execution (branch verdict) │ ├─► Stage: root system │ Task → render-tests (root TestSpec → system / acceptance tests) │ Task → review-execution (root verdict) │ └─► retrospect-build (skill, single end-of-run)

Resume modes

If a worker crashes mid-attempt, the orchestrator reads build-progress.yaml on resume and picks one of three modes:

  • Silent-recover — last_step == review-ready-written AND review-ready.yaml exists. Proceed directly to review-execution; attempt counter unchanged.
  • Resume-at-gate — last_step ∈ {self-check-passed, coverage-met, lint-clean, refactored, green-passing} AND review-ready.yaml absent. Re-dispatch with mode: resume + resume_from_step; skip earlier work, re-validate from the resume point.
  • Restart — earlier last_step or build-progress.yaml absent. Fresh attempt, increment counter, prior worktree contents kept as starting code.

Scope-expansion HALT

When implement-leaf cannot proceed without violating its contract — needing to read a file outside context_to_load, write a file outside files_to_touch, or cross an out_of_scope boundary — it emits build-blocked.yaml and halts. The orchestrator routes:

  • Auto-amend — when suggested_resolution: amend-contract AND amendments_used < build.auto_amend.max_auto_amendments (default 1) AND scope is within the leaf's source directory: append needed paths and re-dispatch same-attempt.
  • Escalate — otherwise: write ESC-NNN.yaml with target_layer derived from suggested_resolution.

Escalation Routing

When the build flow finds a spec gap, it does not silently work around it. review-execution emits a typed ESC-NNN.yaml tagged with target spec layer and confidence. The orchestrator routes:

target_layerRoutes toTrigger
detailed-design vmodel-skill-author-detailed-design (mode B) DD ambiguous, contract incomplete, missing function spec
testspec vmodel-skill-author-testspec (mode B) Test contradicts DD (wrong-assertion), missing case for an edge surfaced during impl
architecture vmodel-skill-author-architecture (mode B) Branch integration failure pointing at decomposition / interface contract
adr vmodel-skill-author-adr (mode B) Decision needed that meets the three-condition threshold; was deferred but now blocks
requirements vmodel-skill-author-requirements (mode B) Root system test failure pointing at requirement (not implementation)
product Human (no skill yet — re-elicit or hand-edit) Root acceptance failure pointing at product intent (rare)

Confidence tagging

  • High confidence — auto-routes to the owning spec author skill; no human interrupt. The author skill runs in mode B and surfaces the result via a fresh review verdict.
  • Low confidence — surfaces to human gate before the spec layer is touched. The human decides which layer is actually responsible.

No spec artifact is silently modified by the build flow. All escalations are filed under .vmodel/.build/escalations/ (committed) before routing.

Workflow Handover — Spec → Build

The spec workflow has no "spec-complete" event. The handover is purely artifact-driven:

  1. Human runs plan-build. The skill walks the spec tree.
  2. For each leaf scope with both detailed_design.md and testspec.md present and structurally valid, plan-build emits a leaf task.
  3. For each branch scope with architecture.md and testspec.md, it emits a branch integration test task.
  4. For the root scope with testspec.md, it emits a root system test task.
  5. Dependencies are inferred from interface usage in DDs and from ADR governing_adrs ordering. Topological sort produces stages.
  6. For each leaf task, plan-build populates the task contract mechanically:
    • acceptance_criteria — one entry per TestSpec case, format "<TS-id>.<case-id>: <case title>"
    • context_to_load — read-only allowlist: own DD + TS, parent ARCH, governing ADRs, dependency-leaf DDs/TSs (interface readability), .vmodel/references/**
    • out_of_scope — declarative do-not list (no spec mutation, no test weakening, no gold-plating, no sibling-leaf source mutation)

Partial spec trees are first-class. The framework explicitly supports starting Build before the entire spec tree is authored. plan-build emits tasks for whatever leaves and branches are spec-complete; missing spec artifacts simply do not appear in the DAG. This lets a project run a vertical slice end-to-end before broadening.

Build → Spec back-channel

The reverse handover happens through escalations (covered above). The shape:

YAML # .vmodel/.build/escalations/ESC-007.yaml schema_version: 1 id: ESC-007 date: 2026-05-10 source_task: leaf-app-checkout-discount-calc target_layer: detailed-design target_artifact: specs/app/checkout/discount-calc/detailed_design.md confidence: high trigger: "DD postcondition for discount(orderTotal, code) does not specify behaviour when code is expired" evidence: - "TestSpec case TC-007 references DD-…discount-expired but DD function has no expired-handling postcondition" recommended_action: "Add postcondition to discount() for expired-code outcome; re-render TC-007 if oracle changes"

The orchestrator copies the file to .vmodel/.build/escalations/ (committed; forensic record) and routes to the named author skill in mode B. The author revises; the human reviews the new draft; review-{layer} verdicts; the loop closes when the new spec is APPROVED. Then the build task can be re-attempted with the corrected spec.

User Workflow 1 — Greenfield from Scratch

You have an idea. You want to build it the V-model way, with AI doing the bulk of the writing under your design review. Step-by-step:

Phase 0 — Project setup (one-time)

  1. Create the project directory and cd into it.
  2. Run vmodel-init. The skill creates .vmodel/config.yaml, copies the framework's reference defaults into .vmodel/references/, and creates the .reviews/ and .build/ placeholders. Interactive checkpoints let you set project name, language, root-product type (PB / needs / PD).
  3. Answer the root-product choice.
    • Product Brief (PB) if you have stakeholders, an NFR matrix, real business context.
    • needs.md if you have stakeholder narrative but not yet a structured PB — you'll run elicit-needs next.
    • Product Description (PD) if it's an internal tool or dogfood project — you'll run elicit-pd next.

Phase 1 — Author the root product

  1. If PD: invoke /vmodel-skill-elicit-pd. Answer the question-prompted blind-spot scan. The skill writes specs/product_description.md.
  2. If needs: invoke /vmodel-skill-elicit-needs. Provide stakeholder narrative (transcript, draft, or interactive). The skill applies anti-assumption / explanation / gap-finding / readback discipline and writes specs/needs.md.
  3. If PB: hand-author specs/product_brief.md against the schema and craft doc (no skill yet — deferred per Phase 5 closeout).

Phase 2 — Author the spec tree, top-down

Recurse from root through every scope. At each scope:

  1. Author Requirements. /vmodel-skill-author-requirements at this scope. Reads root product (or parent Requirements). Writes {scope}/requirements.md.
  2. Review Requirements. /vmodel-skill-review-requirements. Reads the verdict from .vmodel/.reviews/. Iterate until APPROVED.
  3. Author Architecture. /vmodel-skill-author-architecture. Decompose into children with responsibilities and DbC interface contracts. Composition section names the runtime pattern. May emit [DEFER-ADR: …] markers when decisions surface.
  4. Capture ADRs. For each [DEFER-ADR] marker: /vmodel-skill-author-adr. Then /vmodel-skill-review-adr.
  5. Review Architecture. /vmodel-skill-review-architecture. Iterate.
  6. Author TestSpec at this scope. /vmodel-skill-author-testspec. At a non-leaf scope, derives integration cases from Architecture (composition + interface DbC). Then /vmodel-skill-review-testspec.
  7. Recurse into each child scope defined in the Architecture decomposition. Branches repeat steps 1 – 6; leaves go to step 7.
  8. At each leaf scope: author Detailed Design. /vmodel-skill-author-detailed-design. Reads the parent Architecture's leaf-allocation. Writes {leaf}/detailed_design.md with seven mandatory sections.
  9. Review DD. /vmodel-skill-review-detailed-design. Iterate.
  10. Author leaf TestSpec. /vmodel-skill-author-testspec at the leaf — derives unit-level cases from the DD. Then /vmodel-skill-review-testspec.

Tip — vertical slice first. You don't have to author the entire tree before starting Build. A common pattern is: author root product, root Requirements + Architecture + ADRs + root TestSpec, then drill straight down one branch to one leaf, get that leaf to passing code, then come back and broaden. plan-build handles partial trees natively.

Phase 3 — Build

  1. Run /vmodel-skill-plan-build. Walks the spec tree, emits .vmodel/.build/tasks.yaml with leaf, branch, and root stages.
  2. Run /vmodel-skill-orchestrate-build. The orchestrator dispatches workers via the Task tool, stage by stage, parallel within stage. Each leaf goes through render-tests → implement-leaf → review-execution. Branches and root run their integration / system test stages.
  3. If escalations fire (high-confidence), the relevant author skill is auto-invoked in mode B. The fresh draft surfaces in your conversation; you review and approve, then re-run orchestrate-build to retry the affected task.
  4. On run completion, retrospect-build emits retrospective.md and updates lessons.yaml.

Phase 4 — Iterate

You now have committed code + tests for the slice. To extend, broaden the spec tree (Phase 2 for new branches / leaves), then re-run Phase 3. The build is idempotent on already-completed tasks.

User Workflow 2 — Updating an Existing VModel Repo

The repo already has a spec tree and (probably) committed code. You want to add a behaviour, fix a bug, swap a technology. The principle is enter at the highest layer that must change.

Step 1 — Decide the entry layer

Change sourceEntry layerSkill to invoke
New stakeholder, outcome, scope expansionRoot productHand-edit PB, or re-run elicit-needs / elicit-pd
New behaviour under an existing outcomeRequirements (at the scope where it applies)author-requirements + review-requirements
Re-allocation of existing behaviour to a different componentArchitectureauthor-architecture + review-architecture
Technology swap / pattern changeADR + Architecture Compositionauthor-adr + author-architecture (mode B)
Algorithm or state-machine changeDetailed Design (leaf)author-detailed-design + review-detailed-design
Bug fix where spec was correct, code was wrongBuild flow onlyRe-run orchestrate-build targeting the affected leaf
Bug fix where spec was ambiguous or wrongThe layer whose spec was wrongAuthor skill at that layer + downstream propagation

Step 2 — Author the change at the entry layer

  1. Invoke the author skill at the entry layer. The author skill reads the latest review verdict at .vmodel/.reviews/ if one exists.
  2. Revise the artifact. The author skill addresses any prior findings, applies the new change, and may emit new [DEFER-ADR] / [DEFER-DD] markers.
  3. Run the matching review skill. Iterate until APPROVED.

Step 3 — Propagate downstream

Pending — candidate-set walker. TARGET_ARCHITECTURE §8.3 describes a tooling-driven candidate-set walker that, given an entry-layer change, produces a list of downstream artifacts that may need updates. This walker is not yet implemented as a skill or tool. Workaround for now: manually walk the forward links (derived_from, allocates, governing_adrs, verifies) from the changed artifact and visit each affected child. The change is incomplete until every candidate is either updated or explicitly closed as "no change needed" with a brief reason.

For each downstream artifact that must change:

  1. Invoke the author skill in mode B (it reads the latest review verdict and the upstream change).
  2. Run the matching review skill.
  3. If the change creates a [DEFER-ADR], capture the ADR.

Step 4 — Rebuild affected leaves

  1. Run /vmodel-skill-plan-build. The new task DAG reflects the changed specs.
  2. Run /vmodel-skill-orchestrate-build. Already-passing leaves whose specs did not change may be skipped (idempotent on no-op). Affected leaves go through render-tests → implement-leaf → review-execution.
  3. Branch integration tests and root system tests re-run if their TestSpec or upstream architecture changed.

Step 5 — Retrospect

End-of-run retrospect-build updates lessons.yaml. If the same kind of escalation surfaced multiple times, that's signal worth reviewing.

User Workflow 3 — Retrofitting Legacy Code

You have an existing codebase (e.g., a 100k-line Java legacy app with ~10% test coverage and no documentation). You want an honest floor of documentation from which forward development can proceed, plus a gap report driving remediation priorities.

Principle. Retrofit is the Specification workflow run bottom-up with a topology-discovery phase prepended. Same artifact set, same Quality Bar, same Spec Ambiguity Test. Same author/review skills — they have a retrofit mode that enforces no fabrication.

Phase 0 — Project setup

  1. vmodel-init in the legacy repo (or migrate mode if a partial .vmodel/ already exists). The skill scaffolds .vmodel/ non-destructively.
  2. Choose the root product type. Most retrofits start with no product narrative — defer the root-product authoring to Phase 4 below.

Phase 1 — Topology discovery

Pending — automated topology discovery (Phase 7). The framework backlog (BACKLOG.md §3.7) tracks topology discovery as a Phase 7 deliverable: a tool that proposes a candidate scope tree from dependency / package / module analysis. Workaround for now: hand-author the scope tree by inspecting the codebase's directory structure, package boundaries, and dependency graph. AI assistance is fine but the human owns the validation. Output: an agreed scope tree before any artifact authoring.

Phase 2 — Leaf recovery (per leaf)

  1. For each leaf identified in Phase 1, invoke /vmodel-skill-author-detailed-design in retrofit mode (the skill auto-detects when source code exists at the scope but no DD).
  2. The DD author reads source files at the leaf scope. Code-derivable fields (signatures, data structures, observable behaviour) are populated with recovery_status: verified | reconstructed. Human-only fields (rationale, intent, "why this leaf exists") default to recovery_status: unknown — never fabricated.
  3. Human supplies missing intent. Iterate.
  4. Run /vmodel-skill-review-detailed-design. The review skill enforces retrofit honesty (no reconstructed on human-only fields).
  5. Author the leaf TestSpec. Existing tests, if present, become the seed. /vmodel-skill-author-testspec in retrofit mode preserves existing tests as cases with recovery_status: reconstructed on the verifies field; gaps surface as [NEEDS-TEST: …] markers.
  6. Capture leaf-scope ADRs for visible technology choices (frameworks, libraries, persistence). /vmodel-skill-author-adr in retrofit mode — context / alternatives / rationale default to recovery_status: unknown unless human supplies.

Phase 3 — Branch + root recovery (bottom-up)

  1. Going up the tree, for each branch scope:
    • /vmodel-skill-author-architecture in retrofit mode — derives Composition from child DDs / Architectures, decomposition from observed coupling, interfaces from observed call sites. recovery_status applies as for DD.
    • /vmodel-skill-author-requirements in retrofit mode — derives requirements from child contracts and externally-supplied QAs. Rationale defaults to unknown.
    • /vmodel-skill-author-testspec at the branch — integration cases derived from Architecture composition and child TestSpecs.
    • Capture ADRs as decisions surface.
    • Review pass per artifact.
  2. At root: same pattern. Root TestSpec verifies Requirements + (deferred) root product.

Phase 4 — Root product (stakeholder-driven, optional)

Requires conversations with product owners / users. If stakeholders are accessible, run /vmodel-skill-elicit-needs or /vmodel-skill-elicit-pd. If they aren't, defer — the system runs on everything below without a root product. This is a correct outcome for retrofit, not a partial failure.

Phase 5 — Gap report

Pending — gap report aggregator (Phase 7). The framework specifies six gap categories (unintentioned leaf, test coverage gap, ambiguous spec, lost rationale, structural drift, ADR without decision record — see TARGET_ARCHITECTURE §8.2). The aggregator that compiles them into a single retrofit gap report is a Phase 7 deliverable. Workaround for now: the per-artifact recovery_status: unknown fields plus the .vmodel/defer-index.md aggregate (regenerated by scripts/index-deferred-items.py) together cover most of the gap surface. Hand-compile a gap report from these signals.

Phase 6 — Forward development from the retrofit floor

Once the retrofit baseline is committed, switch to Update Workflow (above) for ongoing changes. Retrofit is read-only on code — it produces specs and a gap report. Code modifications are a separate follow-up activity at that point.

What is missing for retrofit-mature use

  • Topology discovery tool (Phase 7) — currently human-driven.
  • Gap report aggregator (Phase 7) — currently a manual sweep.
  • Dedicated retrofit author skill for code-structure analysis and behaviour characterisation — deferred to Phase 7. Per-artifact author skills already encode retrofit honesty (hard refusal A in ADR pair; equivalent in DD), so the framework is usable for retrofit today; the dedicated skill would lift specific recurring patterns.
