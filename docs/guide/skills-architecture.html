Skills Architecture

A detailed design for the AI skills, agents, and orchestration that drive V-model compliant development. This document defines what each component does, how they compose, and where the boundaries are.

The skill system is organized into three layers, each with a distinct concern:

Layer 1 — Skills

Atomic knowledge units. Craft skills encode best practices independent of any framework. Framework skills are thin adapters that introduce VModelWorkflow templates, traceability, and tool checks. Each skill is a single .md file with optional reference files.

Layer 2 — Agents

Execution environments that compose skills into specialized roles. Each agent runs as a subagent with isolated context, scoped tool access, and a structured input/output contract. Agents do the work; skills provide the knowledge.

Layer 3 — Orchestration

Pipeline control. A lean orchestrator reads state files, decides what happens next, and spawns agents. All handoffs between pipeline phases are document-based — any session can restart from the last written artifact.

Why Three Layers?

Separation of concerns. A craft skill doesn’t know it’s being used inside a V-model pipeline. An agent doesn’t know which orchestrator spawned it. The orchestrator doesn’t know the domain knowledge — it just routes contracts between agents. This means:

  • Craft skills are independently usable — anyone can use vmodel-skill-develop-code without the framework
  • Agents are recomposable — swap in a different review skill without changing the pipeline
  • Orchestration is V-model-layer-agnostic — the same pipeline pattern works for detailed design, architecture, requirements
  • Context windows are protected — each agent gets a fresh context; no pollution from prior phases

Document-Based Handoffs

Every boundary between pipeline components is a file on disk, not an in-memory transfer. This is a deliberate architectural decision for resilience:

  • If a session runs out of context, restart from the last written artifact
  • If an unforeseen problem consumes context, exit the pipeline and resume later
  • Human can inspect any handoff document to understand pipeline state
  • Subagents communicate through contracts and verdicts, never through direct messaging

Design Principles

Context Window as Scarce Resource

Every design decision optimizes for context window preservation:

  • Research/plan runs in its own session — the back-and-forth with the human can consume significant context. The output is a clean artifact; the exploration history is discarded.
  • Task decomposer runs as a subagent — reads the plan, produces contracts, releases its context.
  • Each developer/reviewer is a subagent — fresh context per task. No pollution from prior tasks.
  • The orchestrator is lean — reads state files, makes routing decisions, spawns agents. Minimal context consumption.
  • Skills are short — under 500 lines per SKILL.md (research finding: small models follow early instructions better, context dropout increases with length).

SOLID for Skills

Skills follow the same decomposition principles as code:

  • Single Responsibility: Each skill teaches one concern. develop-code doesn’t know about testing. derive-test-cases doesn’t know about code style.
  • Open/Closed: Skills are extended by adding reference files, not by modifying the SKILL.md. New V-model layers add new skills; existing skills don’t change.
  • Dependency Inversion: Agents depend on skill interfaces (what the skill does), not on skill internals (how the prompt is structured).

Framework-independent best practices. Each skill is derived from the corresponding artifact documentation page. The skill adds value only where models don’t already follow best practice by default — the delta that matters.

vmodel-skill-review-code

Review source code and unit tests together against design specification and quality criteria. Produces a structured verdict.

Status
To build
Derived from
Source Code (Section 3.5: Code Review) + Unit Test (Section 3.5: Test Smells)

Output

vmodel-skill-retrofit-dd

Reverse-engineer a detailed design from existing source code. Captures intent and decisions, not just structure.

Status

Output

Extensibility

As we move up the V-model, new framework skills will be added for each layer’s template format (e.g., vmodel-skill-sw-req-template, vmodel-skill-arch-template). The traceability and tool-checks skills are layer-agnostic and will not need duplication.

Agent Definitions

Agents are subagents — isolated execution environments with their own context window, scoped tool access, and a structured input/output contract. Each agent composes relevant skills and runs one task to completion.

Agent — Producer

vmodel-agent-tdd-developer

Takes a task contract and produces code + tests using red-green-refactor TDD. Runs all configured tool checks before declaring done.

Skills loaded
vmodel-skill-develop-code, vmodel-skill-derive-test-cases, vmodel-skill-traceability, vmodel-skill-tool-checks
Input
Task contract (from decomposer) + detailed design document
Output
Source code files, test files, trace links, tool check results
Tool access
Read, Write, Edit, Bash, Glob, Grep (full file system access within scope)
Process
  1. Read task contract, load context files
  2. Read design document for the units in scope
  3. Red: Derive test cases from design (invoke derive-test-cases knowledge). Write tests. Confirm they fail.
  4. Green: Implement code from design (invoke develop-code knowledge). Write minimum code to pass tests.
  5. Refactor: Clean up without changing behavior. Apply code quality checks.
  6. Create/update trace links (invoke traceability knowledge)
  7. Run tool checks from config (invoke tool-checks knowledge)
  8. If all checks pass: write completion artifact. If checks fail: fix and retry (max 3 attempts, different approach each time).
Escalation
HALTs to pipeline controller if: design is ambiguous, scope exceeds contract, 3 consecutive check failures
Agent — Reviewer

vmodel-agent-code-reviewer

Reviews developer output against design specification, quality criteria, and tool check results. Produces a verdict.

Skills loaded
vmodel-skill-review-code, vmodel-skill-traceability, vmodel-skill-tool-checks
Input
Task contract + design document + developer output (code, tests, traces, check results)
Output
Review verdict: APPROVED, REJECTED (with specific findings + fix instructions), or DESIGN_ISSUE (escalate to human)
Tool access
Read, Glob, Grep (read-only — reviewer cannot modify code)
Process
  1. Read task contract and design document
  2. Verify tool checks were run and passed
  3. Review code against checklist (code quality section)
  4. Review tests against checklist (test quality section)
  5. Cross-check: every design element has implementation + test
  6. Check trace link completeness and correctness
  7. Produce structured verdict with evidence for each finding
DESIGN_ISSUE
If the review reveals a problem that cannot be fixed without changing the design (wrong decomposition, missing interface, contradictory requirements), the reviewer writes a DESIGN_ISSUE verdict. This escalates to human rather than cycling back to developer.
Agent — Producer

vmodel-agent-dd-developer

Creates detailed design documents from requirements/architecture. Uses forward or retrofit skill based on context.

Skills loaded
vmodel-skill-develop-dd OR vmodel-skill-retrofit-dd (selected by orchestrator), vmodel-skill-dd-template, vmodel-skill-traceability, vmodel-skill-tool-checks
Input
Task contract + requirements/architecture (forward) OR existing code (retrofit)
Output
Detailed design document (VModelWorkflow format), trace links, tool check results
Tool access
Read, Write, Edit, Glob, Grep (needs to read code for retrofit, write design docs)
Process
  1. Read task contract, load context
  2. Apply develop-dd or retrofit-dd craft knowledge
  3. Produce design in VModelWorkflow template format (dd-template knowledge)
  4. Create trace links to upstream artifacts
  5. Run tool checks, fix issues, write completion artifact
Retrofit note
When in retrofit mode, additional HALT conditions apply: ambiguous intent flags human review rather than guessing.

Agent — Planning

vmodel-agent-task-decomposer

Reads the implementation plan and produces scoped task contracts that fit within a single agent context window.

Skills loaded
(none — this is an architectural reasoning task, not a craft task)

Pipeline & Handoffs

The pipeline is a state machine where every transition produces a document on disk. The orchestrator is a lean skill that reads state, decides the next step, and spawns the appropriate agent.

End-to-End Flow

Phase 1 — Own Session
Research & Plan
Human-driven, interactive. Gathers context, discusses approach, makes design decisions.
→ implementation-plan.yaml
Phase 2 — Human Gate
Task Decomposition
Subagent reads plan, produces contracts. Human approves scope.
→ task-contracts/*.yaml
Phase 3 — Automated Loop
Build → Review
Per task: developer agent produces, reviewer agent validates. Feedback loop (max 3).
→ code, tests, traces, verdicts
Phase 4 — Human Gate
Final Review
Human reviews all output. Approves or rejects.
→ approved artifacts

Phase 1: Research & Plan

Runs in its own session. The human provides context and goals; the AI gathers information, analyzes impact, and proposes approaches. This is a back-and-forth discussion that may take significant context. The output is a clean artifact that captures all decisions — the exploration history is discarded.

Output: Implementation Plan

YAML # .vmodel/pipeline/implementation-plan.yaml plan_id: "PLAN-001" created: "2026-04-06" v_model_layer: "detailed-design" # or "code+unit-test" mode: "forward" # or "retrofit" scope: description: "Detailed design for Fuel Control subsystem" components: - "fuel-rate-limiter" - "fuel-pump-controller" upstream_artifacts: - "docs/requirements/fuel-control-sw-req.yaml" - "docs/architecture/fuel-control-arch.yaml" decisions: - id: "D1" topic: "Rate limiting algorithm" decision: "Token bucket with configurable burst size" rationale: "Matches timing requirements in SWREQ-042; simpler than leaky bucket for our use case" - id: "D2" topic: "Error handling strategy" decision: "Fail-safe: default to minimum fuel rate on sensor failure" rationale: "Safety requirement SR-107 mandates fail-safe behavior" constraints: - "Must be thread-safe (called from ARINC 653 partition)" - "No dynamic memory allocation after initialization" - "Cycle time budget: 10ms worst case" context_files: - "src/fuel-control/existing-pump-controller.java" - "docs/safety/fuel-control-fmea.xlsx" notes: | Human confirmed token bucket approach during discussion. Existing pump controller code should be referenced for interface conventions but not modified in this scope.

Phase 2: Task Decomposition

The task decomposer runs as a subagent (preserves orchestrator context). It reads the plan and produces one task contract per scoped unit of work. The human reviews and approves the decomposition before the automated loop begins.

Output: Task Contract

YAML # .vmodel/pipeline/tasks/TASK-001.yaml task_id: "TASK-001" plan_ref: "PLAN-001" status: "pending" # pending | in_progress | review | approved | rejected | blocked description: "Implement FuelRateLimiter unit with token bucket algorithm" v_model_layer: "code+unit-test" design_refs: - artifact: "DD-FRL-001" file: "docs/design/fuel-rate-limiter.md" sections: ["interface", "behavior", "error-handling"] files_to_touch: - "src/fuel-control/FuelRateLimiter.java" - "src/fuel-control/FuelRateResult.java" - "src/fuel-control/ClampingReason.java" - "test/fuel-control/FuelRateLimiterTest.java" context_to_load: - "src/fuel-control/FuelPumpController.java" # interface conventions - "docs/design/fuel-rate-limiter.md" # full design - ".vmodel/config.yaml" # tool checks acceptance_criteria: - "All design behavior rules (B1-B7) have corresponding tests" - "All error conditions (E1-E4) have tests with fault injection" - "Tool checks pass: unit-tests, static-analysis, coverage ≥ 80%" - "Trace links created: DD-FRL-001 → source files, DD-FRL-001 → test files" trace_links_expected: - source: "DD-FRL-001" targets: ["FuelRateLimiter.java", "FuelRateLimiterTest.java"] link_types: ["implemented-by", "verified-by"] decisions_from_plan: - ref: "D1" summary: "Token bucket with configurable burst size" - ref: "D2" summary: "Fail-safe: default to minimum fuel rate"

Phase 3: Build → Review Loop

For each task contract, the pipeline controller spawns a producer agent, then a reviewer agent. If the reviewer rejects, the task goes back to the producer with specific feedback. Maximum 3 attempts with different approaches. On the 3rd failure or on DESIGN_ISSUE, escalate to human.

State Machine ┌─────────────┐ │ pending │ └──────┬──────┘ │ pipeline picks up ┌──────▼──────┐ ┌────►│ in_progress │ (developer agent) │ └──────┬──────┘ │ │ developer done │ ┌──────▼──────┐ │ │ review │ (reviewer agent) │ └──────┬──────┘ │ │ │ ┌──────┴──────┐ │ │ │ REJECTED │ APPROVED ──► done (attempt │ < 3) │ DESIGN_ISSUE ──► human escalation │ │ └─────┘ │ attempt ≥ 3 ──► human escalation

Feedback Document (on REJECTED)

YAML # .vmodel/pipeline/tasks/TASK-001-feedback.yaml task_ref: "TASK-001" attempt: 1 verdict: "REJECTED" reviewer: "vmodel-agent-code-reviewer" findings: - severity: "must-fix" category: "error-handling" location: "FuelRateLimiter.java:47" finding: "Null returned on sensor timeout instead of fail-safe default" fix_instruction: "Return FuelRateResult.failSafe(MIN_RATE) per design rule E2" - severity: "must-fix" category: "test-completeness" finding: "No test for concurrent access (design rule B6: thread-safety)" fix_instruction: "Add test verifying thread-safe rate calculation under concurrent calls" - severity: "suggestion" category: "naming" location: "FuelRateLimiter.java:23" finding: "Parameter 'ts' should use domain vocabulary" fix_instruction: "Rename to 'sensorTimestamp'"

Retry Discipline

  • Attempt 1: Developer implements from contract
  • Attempt 2: Developer receives feedback, must use a different approach to fix each finding
  • Before attempt 3: Root-cause tracing applied — diagnose why previous fixes didn’t work
  • After attempt 3 failure: HALT. Escalate to human with full history (all feedback documents preserved)

Configuration

Each project has a .vmodel/config.yaml that defines language, tool checks, and component-level overrides. Producer agents read this to know what checks to run. The configuration is project-specific but follows a standard schema.

Configuration Schema

Formal schema: schemas/core/vmodel-config.schema.yaml. Example:

YAML # .vmodel/config.yaml project: name: "fuel-control-system" description: "Embedded fuel control for engine management unit" defaults: language: "java" test_framework: "junit5" build_tool: "gradle" checks: code: - name: "unit-tests" command: "./gradlew test" required: true - name: "static-analysis" command: "./gradlew spotbugsMain" required: true - name: "coverage" command: "./gradlew jacocoTestReport" required: true - name: "traceability" command: "vmodel-trace validate" required: true detailed-design: - name: "schema-validation" command: "vmodel-schema validate docs/design/" required: true - name: "traceability" command: "vmodel-trace validate" required: true # Component-level overrides — most specific path match wins components: ml-pipeline/: language: "python" test_framework: "pytest" checks: code: - name: "unit-tests" command: "pytest ml-pipeline/tests/" required: true - name: "type-check" command: "mypy ml-pipeline/" required: true - name: "coverage" command: "pytest --cov=ml-pipeline --cov-fail-under=80" required: true frontend/: language: "typescript" test_framework: "vitest" checks: code: - name: "unit-tests" command: "npm test -- --run" required: true - name: "lint" command: "npm run lint" required: true - name: "type-check" command: "npx tsc --noEmit" required: true

Resolution Order

When an agent needs to determine which checks to run for a file:

  1. Match the file path against component prefixes (most specific match wins)
  2. If no component matches, use defaults
  3. Component config fully replaces defaults for that section (no merging) — keeps behavior predictable

Check Execution Contract

Every check command must:

  • Exit 0 on success, non-zero on failure
  • Write output to stdout/stderr (agent captures and logs to /tmp)
  • Be idempotent (safe to re-run)
  • Complete within a reasonable time

Pipeline State

YAML # .vmodel/pipeline/pipeline-state.yaml plan_ref: "PLAN-001" phase: "executing" # planning | decomposing | executing | human-review | done current_task: "TASK-002" tasks: TASK-001: { status: "approved", attempts: 1 } TASK-002: { status: "in_progress", attempts: 2 } TASK-003: { status: "pending", attempts: 0 } blocked: [] escalated: []

The pipeline controller reads this file on startup to determine where to resume. If the file doesn’t exist, the pipeline starts fresh. If the session crashed mid-task, the task’s status tells the controller whether to re-run the developer or the reviewer.

Shared Reference Files

Reference files that are used by multiple skills are duplicated into each skill’s references/ directory. This is deliberate — each skill must be self-contained (no cross-skill file references). When a shared reference changes, all copies must be updated. A future meta-skill or build step may automate this synchronization.

Alternative considered: Symlinks or a shared references/ directory. Rejected because the agentskills.io spec expects each skill to be self-contained, and symlink behavior varies across platforms and tools.

Current Scope & Build Order

This architecture covers the lower V-model layers: detailed design, source code, and unit tests. Higher layers (architecture, requirements, system test) will follow the same pattern with layer-specific skills added.

Build Order

Bottom-up, craft skills first, then framework, then agents, then orchestration:

  1. Craft skills: review-code, develop-dd, retrofit-dd, review-dd (rename existing skills to vmodel-skill-* convention)
  2. Framework skills: dd-template, traceability, tool-checks
  3. Agents: tdd-developer, code-reviewer, dd-developer, dd-reviewer, task-decomposer
  4. Orchestration: research-plan skill, pipeline controller skill
  5. Configuration: config schema, project setup (.vmodel/config.yaml)

Each step is independently useful. After step 1, anyone can use the craft skills standalone. After step 3, agents can be manually invoked. Step 4 wires everything together.
