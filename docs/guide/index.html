The V-Model: What & Why

The V-model is a software development lifecycle that pairs every specification activity on the left with a corresponding verification activity on the right. The “V” shape shows this relationship visually.

The Core Idea

Traditional waterfall development writes requirements, then designs, then codes, then tests. The problem: testing happens last, and by then you've built something that may not match what was needed. Bugs found late are expensive to fix.

The V-model fixes this by planning verification at the same time as specification. When you write a requirement, you also define how it will be tested. When you design the architecture, you also define the integration tests. This front-loading of verification thinking catches problems early.

The Shape

The "V" has two arms and a bottom point:

Left arm (going down): Specification and design, from abstract to concrete.

System Requirements → SW Requirements → SW Architecture → Detailed Design → Implementation

Bottom point: Implementation — where code is actually written.

Right arm (going up): Verification, from concrete to abstract.

Unit Tests → Integration Tests → SW Verification → System Tests

Why the V-Model Exists

Safety-critical industries (aviation, automotive, medical devices) cannot afford to discover in production that a requirement was missed or a design was wrong. The cost of failure ranges from product recalls to loss of life.

The V-model provides:

  • Traceability — every requirement can be traced to its verification evidence
  • Completeness checking — orphan detection reveals gaps before release
  • Change impact analysis — when a requirement changes, you know exactly which designs and tests are affected
  • Audit evidence — certification bodies can inspect the artifact chain

Not Just for Safety-Critical

While the V-model originated in safety-critical domains, its principles benefit any software project that values quality. The rigor can be scaled: a startup might use lightweight artifacts and informal reviews, while an aviation project uses formal verification with independent review boards. The structure is the same — only the formality changes.

Common Misconceptions

“The V-model is waterfall”

No. The V-model defines what artifacts you need and how they relate. It does not mandate that you produce them in a strict top-to-bottom sequence. You can work iteratively, incrementally, even use agile sprints — as long as the artifact relationships are maintained. VModelWorkflow explicitly supports incremental, module-by-module development.

“It’s only for big projects”

The V-model scales down. A small feature might have 3 requirements, 1 architecture artifact, 2 detailed designs, and corresponding tests. The trace links still provide value: you know what's verified and what isn't.

“It generates too much documentation”

Bad V-model implementations do. Good ones produce exactly the documentation needed for traceability and nothing more. In VModelWorkflow, artifacts are concise YAML files — not 50-page Word documents.

Left Side: Specification & Design

The left arm of the V moves from the abstract (“what does the system need to do?”) to the concrete (“what does this function do, exactly?”).

Level 0: System Requirements

The highest level. Defines what the overall system must do, often derived from customer needs, safety analyses, or regulatory requirements. In a calculator app, this might be: “The system shall perform arithmetic operations on user-provided numerical inputs.”

System requirements are allocated to software and hardware. The software portion becomes the input for the next level.

Level 1: SW Requirements

Defines what the software must do. These are derived from system requirements and decomposed to be individually testable. Each SW requirement has a defined verification method (test, analysis, inspection, demonstration, or review).

A good SW requirement is:

  • Unambiguous — one interpretation only
  • Testable — you can write a test for it
  • Traceable — linked to the system requirement it comes from
  • Atomic — one requirement, one “shall”

Level 2: SW Architecture

Decomposes the software into components, defines their responsibilities, and specifies the interfaces between them. The architecture answers: “How is the software organized to satisfy all the SW requirements?”

SW requirements are allocated to architecture components. This allocation is a trace link managed by the traceability layer.

Level 3: Detailed Design

Describes individual units (functions, classes, modules) at a level of detail sufficient for implementation and test derivation. Each unit belongs to exactly one architecture component.

The detailed design defines: inputs, outputs, behavior rules, and error handling. A developer reading the detailed design should be able to implement the code. A tester reading it should be able to derive test cases.

Level 4: Implementation

The bottom of the V. Actual source code, written to satisfy the detailed design. In the V-model, implementation is not freewheeling — it follows a specification.

Right Side: Verification

The right arm of the V verifies that each specification level was correctly implemented. Each right-side activity corresponds to a left-side activity at the same depth.

Level 3: Unit Tests

Verify the Detailed Design. Each behavior rule and error handling case in the design becomes a test case. Unit tests are the most granular verification — they test individual functions or classes in isolation.

Level 2: Integration Tests

Verify the SW Architecture. Integration tests exercise the interfaces between components. They answer: “Do the components work together as the architecture defined?”

Level 1: SW Verification

Verifies that SW Requirements are satisfied. This is often called “qualification testing” or “software acceptance testing.” It operates at the SW level (not individual components) and checks that the assembled software meets its requirements.

Level 0: System Tests

Verifies that System Requirements are satisfied. In the target environment (or a faithful simulation of it). This is the highest-level verification — does the complete system do what was originally required?

The Pairing Principle

The crucial insight: each left-side artifact is verified by the right-side artifact at the same depth. This is not coincidence — it's the structural guarantee that nothing falls through the cracks.

System Req ← verified by → System Test
SW Req ← verified by → SW Verification
Architecture ← verified by → Integration Test
Detail Design ← verified by → Unit Test

Traces & Links

Traces are the connective tissue of the V-model. They link artifacts together so you can answer: “Is everything specified? Is everything verified? Is anything stale?”

What is a Trace?

A trace is a typed, directional link between two artifacts. For example:

SWREQ-042 → allocated-to → ARCH-001

This says: “Software requirement SWREQ-042 is allocated to the architecture defined in ARCH-001.”

Trace Direction

Traces are stored separately from artifacts (not embedded). Each trace file has a source artifact and one or more target links. The source is always the artifact being traced from.

Why Separate?

Keeping traces in their own files (rather than embedding them in artifacts) is a deliberate design choice:

  • Artifacts remain clean and focused on their own content
  • Traces can be validated independently by tooling
  • Multiple teams can work on artifacts and traces without merge conflicts
  • The traceability layer (Pillar 2) can evolve independently from the artifact layer (Pillar 1)

Vertical vs Horizontal Traces

Vertical (down the V)

Link artifacts at different V-levels: a detailed design is derived-from a SW requirement. These traces show how high-level needs decompose into concrete designs.

Horizontal (across the V)

Link specification to verification at the same level: a detailed design is verified-by a unit test. These traces show that each specification has verification evidence.

For the full list of link types with rules and examples, see Link Types.

Gates & Reviews

In the V-model, artifacts don’t automatically progress. A human must review and approve each artifact before it becomes part of the verified baseline.

Artifact Lifecycle

Every artifact follows this lifecycle:

draft ──→ in_review ──→ approved ──→ baselined ↑ │ │ └───────────┘ ↓ (rejected) superseded
  • draft — work in progress, not yet ready for review
  • in_review — submitted for review, awaiting verdict
  • approved — reviewed and accepted by a qualified reviewer
  • baselined — frozen in a configuration baseline; changing it requires a new version
  • superseded — a newer version has replaced this one

Why Human Gates?

Even with automated validation (schema checks, trace completeness), human judgment remains essential. Automated tools can verify structure but not correctness. A requirement can be syntactically perfect EARS format and still be wrong.

Human gates ensure:

  • Technical correctness (does the requirement actually describe what we need?)
  • Completeness (did we miss any requirements?)
  • Consistency (does this conflict with other artifacts?)
  • Appropriateness (is this the right level of detail?)

Review Records

Reviews are not just approval stamps — they are artifacts themselves. A review record captures what was reviewed, who reviewed it, what findings were raised, and what the verdict was. This provides audit evidence that due diligence was performed.

Artifact Envelope

Every V-model artifact shares a common envelope. The envelope provides identity, versioning, and lifecycle tracking. Type-specific content goes in the body field.

Fields

FieldTypeDescription
artifact_id string required Human-readable ID. Format: TYPE-nnn (e.g., SWREQ-042, ARCH-001, DD-017)
artifact_type string required References an artifact type schema (e.g., sw-requirement, sw-architecture, detailed-design)
version string required Semantic version (1.0.0). Major = breaking interface change, Minor = refinement, Patch = correction.
status enum required draft | in_review | approved | baselined | superseded
body object required Type-specific content. Structure defined by the artifact type schema.
assurance_level integer optional 1 (highest rigor) to 5 (lowest). Maps to DAL A–E or ASIL D–QM depending on domain.
content_hash string auto SHA-256 prefix (8 chars) of the body. Used for staleness detection. Machine-generated.

Example

yaml artifact_id: "SWREQ-042" artifact_type: "sw-requirement" version: "1.0.0" status: "draft" assurance_level: 3 body: title: "Addition operation" statement: "The calculator shall compute the sum of two numerical inputs." verification_method: "test" rationale: "Core arithmetic operation required by system requirement SYSREQ-001."

What’s Explicitly Not in the Envelope

created_by — Whether a human or an AI agent wrote the artifact is irrelevant to compliance assessment. Quality is determined by review, not authorship.

reviewed_by — A name on a field is insufficient evidence. Review evidence is captured in separate Review Record artifacts linked via traceability.

trace_up / trace_down — Traceability is managed by Pillar 2 in separate trace files. Artifacts don't know about their links.

Two-Tier Identification

Artifacts have two ID forms:

  • Human-facing: SWREQ-042 — readable, stable, used in conversations and documents
  • Machine-facing: SWREQ-042@a3f7c2d1 — includes content hash for change detection

The hash changes when the body content changes, enabling automatic staleness detection in trace links.

SW Requirement

Left Side · Level 1

Defines what the software must do. Derived from system requirements, decomposed to be individually testable.

What is a SW Requirement?

If system requirements say “what the whole machine needs to do,” then SW requirements say “what the software’s share of that job is.”

Think of it this way: a system requirement might say “the coffee machine must brew coffee in under 3 minutes.” That’s a job for both hardware (the heater, the pump) and software (the control logic). The SW requirement zooms in on just the software’s part: “The software shall activate the pump within 200ms of receiving a brew command.”

Each SW requirement must be individually testable — you should be able to write a test case for it. If you can’t, it’s too vague and needs to be refined.

Why bother?

SW requirements are the bridge between what the customer wants and what developers build. Without them:

  • Developers guess what to implement — and guess differently
  • Testers don’t know what “correct” looks like
  • There’s no way to prove the software does what was asked

In the V-model, SW requirements are Level 1 — one step below system requirements. They are derived from system requirements (traced via Pillar 2) and allocated to architecture components. On the right side of the V, SW Verification tests verify that every SW requirement is satisfied.

Schema Fields

FieldTypeDescription
titlestringrequiredShort descriptive name
statementstringrequiredThe requirement text. Free-form — any syntax is valid at the schema level.
verification_methodenumrequiredtest | analysis | inspection | demonstration | review
rationalestringoptionalWhy this requirement exists. Especially valuable for derived requirements.

Good Example

Good Requirement
yaml artifact_id: "SWREQ-001" artifact_type: "sw-requirement" version: "1.0.0" status: "draft" body: title: "Addition of two numbers" statement: > When the user provides two numerical inputs and selects the addition operation, the calculator shall display the arithmetic sum of the two inputs. verification_method: "test" rationale: > Core arithmetic operation. Derived from SYSREQ-001 which requires the system to perform basic arithmetic.

This requirement is testable (clear inputs, operation, expected output), unambiguous (one interpretation), and atomic (one “shall”).

Bad Requirement
yaml body: title: "Calculations" statement: "The calculator should do math correctly and quickly." verification_method: "test"

Problems: vague (“math” means what?), compound (“correctly and quickly” are two concerns), uses “should” instead of “shall” (ambiguous obligation), and not testable (how do you test “quickly” without a threshold?).

Trace Connections

SYSREQ-nnn → derived-from → SWREQ-nnn → allocated-to → ARCH-nnn
SWREQ-nnn → verified-by → SW Verification Tests

SW Architecture

Left Side · Level 2

Decomposes the software into components, defines their responsibilities, and specifies the interfaces between them.

What is a SW Architecture?

If SW requirements say “what the software must do,” then the architecture says “how the software is organized to do it.”

Think of it like designing a house. The requirements say you need a kitchen, two bedrooms, and a bathroom. The architecture is the floor plan — it decides where each room goes, how big it is, and how they connect (doors, hallways). It doesn’t specify the color of the walls or the type of tiles — that’s detailed design.

In software, the architecture breaks the system into components (the rooms), assigns each one a clear responsibility (what it does), and defines the interfaces between them (how they talk to each other).

Why bother?

Without architecture, software grows organically into a tangle where everything depends on everything else. Architecture prevents this by:

  • Dividing the problem — each component can be designed, built, and tested independently
  • Defining boundaries — interfaces make it clear what each component promises to others
  • Enabling integration testing — on the right side of the V (Level 2), integration tests verify these interfaces work

SW Architecture is Level 2 in the V-model. SW requirements are allocated to architecture components (traced via Pillar 2). Each component is then fleshed out in detailed designs at Level 3.

Example: Calculator Architecture

yaml artifact_id: "ARCH-001" artifact_type: "sw-architecture" version: "1.0.0" status: "draft" body: title: "Calculator Application Architecture" description: > Decomposes the calculator into input parsing, arithmetic engine, and output formatting components with clean interfaces. components: - id: "input-parser" description: "Parses and validates user input into structured operands" responsibilities: - "Parse numerical input strings into numeric values" - "Validate input is within acceptable range" - "Report parse errors with descriptive messages" - id: "arithmetic-engine" description: "Performs arithmetic operations on validated operands" responsibilities: - "Execute addition, subtraction, multiplication, division" - "Handle edge cases (division by zero, overflow)" - "Return structured results with operation metadata" - id: "output-formatter" description: "Formats computation results for display" responsibilities: - "Format numeric results with appropriate precision" - "Format error messages for user display" interfaces: - from: "input-parser" to: "arithmetic-engine" mechanism: "function call" data: "Parsed operands (left: number, right: number, operator: enum)" - from: "arithmetic-engine" to: "output-formatter" mechanism: "function call" data: "Computation result (value: number, operation: string) or error" design_rationale: > Separated parsing from computation to allow independent testing and potential reuse of the arithmetic engine. The output formatter is isolated so display format can change without affecting logic.

Detailed Design

Left Side · Level 3

Describes components and their units at a level of detail sufficient for implementation and test derivation. Written as Markdown with YAML frontmatter.

What is a Detailed Design?

If the architecture is the floor plan, the detailed design is the blueprint for each room. It starts at the component level — the map of what exists, why, and how it connects — then adds unit-level detail where needed: inputs, outputs, behavior rules, and error handling, specific enough that a developer can implement from it and a tester can derive test cases from it.

Why bother?

This is where many teams skip straight to coding and regret it later:

  • Developers get a spec, not a guess — no ambiguity about what to build
  • Testers get test cases directly — each behavior rule (B1, B2...) and error condition (E1, E2...) becomes a test
  • Code reviews have a reference — does the code match the design?
  • Future maintainers understand intent — why the code does what it does, not just how
  • Assessors see proportional effort — detail scales with criticality, not one-size-fits-all

Detailed Design is Level 3 in the V-model — the last step before implementation. On the right side, unit tests (also Level 3) verify that each unit behaves exactly as its detailed design specifies.

Design must exist before code. If you write the detailed design after implementing the code, the unit test becomes meaningless — you’re just documenting what already exists, not verifying that code fulfils a design. This is the #1 assessment failure in ASPICE SWE.3.

Layered Design: Scale with Criticality

Not every unit needs the same level of documentation. Detailed design uses an additive layering model — higher layers build on lower layers, you never skip a layer:

LayerWhat It CoversWhen Required
Layer 1
Component Design		 Purpose, external interfaces, unit inventory, shared patterns, constraints. The map of what exists. Always. Every component gets a Layer 1 regardless of criticality.
Layer 2
Unit Contracts		 Per-unit interfaces (typed inputs/outputs), behavior rules, error handling, configuration, internal state. ASIL A+ / DAL D+ / IEC 62304 Class B+ — or any non-trivial unit you want proper test derivation for.
Layer 3
Deep Evidence		 Design rationale, state machines/sequence diagrams, concurrency model, negative requirements. ASIL C-D / DAL A-C / IEC 62304 Class C — or any safety-critical or highly complex unit.

Layer 2 and 3 content for units can either be embedded in the component document (practical when a component has few units) or written as separate unit documents (better when units are complex enough to warrant their own file). Both approaches are equally valid for compliance.

Document Format

Detailed designs are Markdown files (.md) with YAML frontmatter. This makes them readable by humans and assessors while remaining parseable by agents and the traceability engine.

Layer 1 Sections: Component Design

Every detailed design document starts with these sections. They provide the component-level overview — the map that gives context to any unit-level detail below.

SectionWhat To WriteCommon Mistakes
## Purpose What this component does and why it exists in the architecture. Ask yourself: “If this component were deleted, what would break?” 1–3 paragraphs. Not a restatement of the architecture description — focus on this component’s specific role. Copying the architecture description verbatim. Too vague (“handles fuel stuff”). Too long (implementation details leaking in).
## External Interfaces How this component communicates with other components. Table with columns: Direction | Partner Component | Mechanism | Data | Description. This is the component’s boundary as seen from outside — not internal unit-to-unit calls. Listing internal interfaces. Missing the mechanism (function call? message queue? shared memory?). Describing data format instead of data meaning.
## Unit Inventory All units in this component, each with its design layer and a one-line description. Table with columns: Unit ID | Layer | Brief Description. This is the table of contents for the component’s internals. Every unit must appear — even trivial ones. Forgetting trivial units (DTOs, config holders). Layer column missing or wrong. Descriptions that are too vague to distinguish units.
## Shared Patterns Design decisions that apply across multiple units in this component. Document once here instead of repeating in every unit. Typical patterns:
  • Error strategy — How do units report errors? Exceptions? Result types? Error codes?
  • Threading model — Single-threaded? Thread-per-request? What about shared state?
  • Resource management — Who owns lifecycle of shared resources (connections, file handles)?
  • Common data types — Shared enums, value objects, or structs used across units.
If a unit deviates from a shared pattern, document the deviation in that unit’s section, not here.		 Writing generic best practices (“we use clean code”) instead of concrete component decisions. Omitting this section because “it’s obvious” — assessors can’t verify what isn’t written. Putting unit-specific decisions here.
## Constraints Non-functional requirements that apply to the component as a whole: performance budgets, memory limits, certification requirements, deployment constraints. Only component-wide constraints — unit-specific constraints go in the unit’s section. Mixing functional and non-functional. Listing constraints that only apply to one unit.

Layer 2 Sections: Unit Interface Contracts

Added per unit that needs detailed specification. When embedded in the component document, use ## Unit Name (UnitId) as the heading with ### for subsections. When in a separate file, the unit name is the document title and subsections use ##.

SectionWhat To WriteID Convention
### Purpose What this specific unit does. 1–2 sentences. Clear enough that someone can decide whether a given code change belongs in this unit or not. —
### Interfaces
 #### Inputs		 Table: ID | Name | Type | Unit | Constraints | Description. Type uses language-agnostic names (float, integer, enum, etc.). Constraints column: valid ranges, enum values, lengths. I1, I2, I3...
 #### Outputs Same table format. For union/variant outputs, add a row per variant with the variant name in Description. O1, O2, O3...
### Behavior Table: ID | Condition | Result. Each row is one testable rule. Be specific enough to write an assertion: “Returns error” is too vague; “Returns Error(INVALID_SYNC, offset 0)” is testable. For sequential algorithms, use “after B1”, “after B2 succeeds” in Condition. B1, B2, B3...
### Error Handling Table: ID | Condition | Response. Every foreseeable error condition. If the unit has no error cases, state that explicitly with one row. E1, E2, E3...
### Configuration Table: Name | Type | Default | Description. Parameters set at init/deployment time. Only present if the unit has configurable parameters. —
### Internal State Table: Name | Type | Description. State retained between invocations. Omit entirely for stateless units. —
### Constraints Unit-specific non-functional requirements (thread-safety, timing, determinism). Only what’s specific to this unit — component-wide constraints stay in Layer 1. —

Layer 3 Sections: Deep Design Evidence

Added for safety-critical or highly complex units. Required at ASIL C-D / DAL A-C / IEC 62304 Class C.

SectionWhat To Write
### Design Rationale WHY this design, not just what it is. For each significant decision: what was decided, what alternatives were considered, why this approach was chosen. This is what separates design from code paraphrase. Can be a table (Decision | Alternatives | Justification) or structured prose.
### Dynamic Behavior State machines, sequence diagrams, timing diagrams for units with complex runtime behavior. Use Mermaid or PlantUML code blocks. Only needed when the Behavior table alone doesn’t capture temporal or state aspects.
### Concurrency Model Thread-safety guarantees, synchronization mechanisms, shared state access patterns. Required when the unit may be accessed from multiple threads or contexts.
### Negative Requirements What this unit explicitly must NOT do. Prevents scope creep and clarifies responsibility boundaries. Examples: “Must not allocate heap memory”, “Must not perform I/O”, “Must not modify input parameters.”

Traceability IDs

Every table row has an ID column. These IDs enable fine-grained traceability between design elements and test cases:

System Requirement

Left Side · Level 0

Defines what the system must do — before anyone thinks about software, hardware, or how to build it.

What is a System Requirement?

A system requirement is the answer to: “What does this thing need to do?” — before anyone thinks about software, hardware, or how to build it.

Imagine you’re building a coffee machine. Before anyone writes code or designs circuits, someone needs to write down: “It must brew coffee in under 3 minutes,” “It must not exceed 85 dB noise,” “It must shut off automatically after 30 minutes.” Those are system requirements — they describe what the whole system must do, in terms a customer or regulator would understand.

Why bother?

Everything in the V-model flows down from system requirements:

  • SW requirements are derived from system requirements
  • Architecture decisions are justified by system requirements
  • Tests at the top of the right side verify system requirements

If you skip them or get them wrong, everything below is built on sand. You might build software that passes all its tests but doesn’t actually satisfy what the customer needed — because nobody wrote down what that was.

In regulated industries this isn’t optional. Auditors will ask: “Show me the system requirement. Show me how it traces down to software. Show me how it traces back up to verification.” If you can’t, you don’t certify.

Where it sits in the V

    System Requirements  ← YOU ARE HERE
           ↓
     SW Requirements
           ↓
     SW Architecture
           ↓
     Detailed Design
           ↓
      Implementation

System requirements are Level 0 — the top of the left side. They are origin artifacts: they don’t derive from other artifacts in the framework, they derive from the real world (customer needs, safety analysis, regulations).

This requirement is testable (clear threshold and timing), unambiguous (specific numbers), atomic (one “shall”), and classified (safety + regulatory flags help auditors filter).

Bad System Requirement
yaml body: title: "Fuel Safety" statement: "The system should handle fuel safely and efficiently." requirement_type: "safety" verification_method: "review"

Problems: vague (“safely and efficiently” means what?), compound (two concerns in one statement), uses “should” instead of “shall” (ambiguous obligation), and not testable (no measurable threshold).

System Test Case

Right Side · Level 0

Defines how a system requirement is verified at the system level — the top of the right side of the V.

What is a System Test Case?

A system test case answers: “How do we prove the system does what the requirement says?”

Remember the coffee machine example? You wrote “It must brew coffee in under 3 minutes.” Now you need to define: put water in the tank, insert a pod, press start, measure time until coffee is dispensed. That procedure, together with the expected result (“coffee dispensed in under 3 minutes”), is your system test case.

Why a YAML artifact?

Lower on the V, tests are code — unit tests, integration tests, qualification tests. They live as real test files and are linked via Pillar 2 trace files. But system-level testing is different:

  • Often not code — manual procedures, hardware-in-the-loop scripts, bench tests, field demonstrations
  • Multiple verification methods — test, analysis, inspection, demonstration, review
  • External environments — may require simulators, rigs, or target hardware

Because system tests often can’t live as executable code, they need their own artifact format. Each step combines the action with the expected result — avoiding the common V-model confusion of splitting procedure and expected results into separate documents.

Where it sits in the V

System Requirements  ────  System Test Cases  ← YOU ARE HERE
         ↓                         ↑
   SW Requirements      SW Verification Tests
         ↓                         ↑
   SW Architecture       Integration Tests
         ↓                         ↑
   Detailed Design          Unit Tests
         ↓                         ↑
         └──── Implementation ────┘

System test cases are Level 0 — the top of the right side. They mirror system requirements on the left.

FieldTypeDescription

objectivestringrequiredWhat this test verifies and why.
verification_methodenumrequiredtest | analysis | inspection | demonstration | review

step_number (integer) — sequential, starting at 1

Good Example

Good System Test Case
yaml artifact_id: "SYSTEST-001" artifact_type: "system-test-case" version: "1.0.0" status: "draft" body: title: "Arithmetic Operations — End-to-End" objective: > Verify that the calculator correctly performs all four arithmetic operations with user-provided inputs and displays the result. verification_method: "test" preconditions: - "Calculator application is running" - "No previous calculation state (fresh session)" steps: - step_number: 1 action: "Enter 12 as the first operand and 4 as the second operand, select addition" expected_result: "Display shows 16" - step_number: 2 action: "Enter 12 as the first operand and 4 as the second operand, select subtraction" expected_result: "Display shows 8" - step_number: 3 action: "Enter 12 as the first operand and 4 as the second operand, select multiplication" expected_result: "Display shows 48" - step_number: 4 action: "Enter 12 as the first operand and 4 as the second operand, select division" expected_result: "Display shows 3" postconditions: - "Calculator returns to ready state after each operation" environment: "Host platform, release build"

Each step has a specific action and an unambiguous expected result. A tester can execute this and determine pass/fail without interpretation. The objective explains what is being verified without requiring a lookup of the linked requirement.

Bad System Test Case
yaml body: title: "Calculator Test" objective: "Test the calculator" verification_method: "test" preconditions: [] steps: - step_number: 1 action: "Use the calculator" expected_result: "It works correctly"

Problems: vague objective (test what about the calculator?), empty preconditions (what state is assumed?), non-specific action (“use” how?), and untestable expected result (“works correctly” is not a pass/fail criterion).

Staleness Detection

When an artifact changes, its trace links may become stale. Content hashing detects this automatically.

How It Works

Every artifact has a content_hash — the first 8 characters of the SHA-256 hash of its body content. When a trace link is created or rebased, both the source hash and target hash are recorded in the trace file:

yaml source: "DD-001" source_hash: "a3f7c2d1" # hash at time of last verification links: - type: "implemented-by" target: "src/ArithmeticEngine.java" target_hash: "b8e4f123" # hash at time of last verification

Detection Flow

The traceability tool (Pillar 2) compares stored hashes against current hashes:

  1. Compute current hash of DD-001's body
  2. Compare against stored source_hash (a3f7c2d1)
  3. If different → the source artifact changed since the trace was last verified
  4. Same process for each target

Staleness States

Fresh

All hashes match. The trace link is still valid.

Source-Stale

The source artifact changed. The link may no longer be accurate. Needs re-verification.

Target-Stale

The target changed (code was modified, test was rewritten). The link may no longer reflect reality.

Both-Stale

Both sides changed. Highest priority for review.

Rebasing

When a human confirms that a stale link is still valid (the changes don't affect the relationship), the tool rebases the link by updating the stored hashes to the current values. This is an explicit human action, not automatic.

VModelWorkflow: Three Pillars

VModelWorkflow is built as three independent pillars that can be used standalone or combined. No pillar depends on the others to function.

Pillar 1: V-Model Compliance

Schemas, templates, and checklists defining what artifacts are needed and what they look like. This is the definitional layer.

Usable by: humans, agents, or both.

Alone: A human team gets V-model templates and checklists.

Pillar 2: Traceability

Data model and engine for linking artifacts, validating completeness, and detecting gaps. This is the relational layer.

Usable by: humans, CI/CD, agents, anyone.

Alone: Can validate any YAML artifacts regardless of how they were created.

Pillar 3: Agentic Skills

Prompt packages that guide AI agents to produce quality V-model artifacts. Craft skills (standalone) + orchestration (workflow).

Usable by: AI agents.

Alone: Someone can use a single requirement-writing skill outside the framework.

Combination Modes

  • Pillar 1 alone — human team with V-model templates
  • Pillar 1 + 2 — human team with templates + automated completeness checking
  • Pillar 1 + 2 + 3 — full agentic V-model development
  • Pillar 3 craft skills alone — just want a good requirement-writing prompt

Independence Principle

Pillar independence is enforced by design:

  • Craft skills reference Pillar 1 schemas via schema_ref, but schemas don’t know about skills
  • The traceability engine reads artifacts defined by Pillar 1 schemas, but schemas don’t know about the engine
  • Artifacts have no trace_up/trace_down fields — traceability is entirely Pillar 2’s concern

DRTDD: Design-Requirement-Test Driven Development

DRTDD extends TDD by adding design and requirement phases before the test-code loop. Each phase produces traceable artifacts.

The Flow

REQUIRE ──→ DESIGN ──→ TEST (red) ──→ IMPLEMENT (green) ──→ REFACTOR ──→ VERIFY │ │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ ▼ SW Req Detailed Test file Source file (same files) Trace links artifact Design (failing) (passing) (cleaner) updated artifact

Phase Details

  1. REQUIRE — Write or refine a SW requirement. Gate: human approval.
  2. DESIGN — Write the detailed design for the unit. Gate: human approval.
  3. TEST (red) — Derive test cases from the design. Tests must fail (no implementation yet).
  4. IMPLEMENT (green) — Write the minimum code to make tests pass.
  5. REFACTOR — Clean up without changing behavior. Tests still pass.
  6. VERIFY — Update trace links, run coverage, confirm all evidence is fresh.

Why Not Just TDD?

TDD (red-green-refactor) is excellent for implementation but doesn’t produce the artifacts needed for V-model compliance. DRTDD wraps TDD in a specification and traceability layer:

  • You know why you’re writing this code (requirement)
  • You know what it should do precisely (design)
  • You have evidence that it does what it should (test + trace)
