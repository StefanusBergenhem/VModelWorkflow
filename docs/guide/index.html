VModelWorkflow

A framework for AI-augmented, spec-driven software development with V-model rigor. The goal: software built by AI under tight human design review, where the human leverage point is the spec layer — not code review.

The Problem It Solves

Generating code with an LLM is cheap. Producing code that is correct, maintainable, and traceable to intent is not. Left alone, agents fabricate rationale, paper over ambiguity, and write tests derived from the implementation they just generated. The framework raises the leverage point: humans author specifications under uniform high rigor; agents do the rest under review against those specifications.

Engineering Infrastructure, Not Safety Ceremony

The V-model contributes layered thinking, traceability, and design-test coupling. The framework keeps those and drops the safety-specific baggage: no rigor tiers, no DAL/ASIL mapping, no HW/SW split, no standards-vocabulary wrapper. The remaining rigor is optimised for AI authoring at any domain.

What Comes Out of It

  • A scope tree of specifications covering the whole system, authored from six artifact types.
  • A traceability graph where every test traces to a spec element and every requirement traces to an allocation.
  • A Quality Bar per artifact type that each artifact passes before it ships downstream.
  • Code and tests that a human can accept or reject because the spec above them is unambiguous.

Four Components

Four independently usable components. Adopt one, adopt all four. Full value comes from combining them.

Documentation

Single source of truth. Per artifact type: five sections — V-model context, best practices, anti-patterns, examples, Quality Bar. AI skills are derived from this, not the other way round.

Templates & Schemas

Per-artifact YAML schemas encode structural rigor. Canonical Quality Bar YAML encodes semantic rigor. Scaffolder templates seed new artifacts. Schemas are mechanically validatable.

Traceability

Forward links live in artifact front-matter. Reverse relationships (realized_by, verified_by, derives) are computed, not stored. Validation rules enforce completeness, cycle-freedom, and reference integrity.

AI Skills

Craft skills (standalone authoring/review per artifact type) plus framework skills (orchestration, traceability, retrofit). Built against the documentation. Evaluated on the cheapest viable model tier.

Why Independent

Documentation alone — teaches the craft.

Templates + docs — manual spec authoring.

Traceability + templates — automated completeness checking.

Craft skills + docs — AI-assisted spec authoring without framework-specific machinery.

All four — full AI-augmented spec-driven development.

Use Cases

Greenfield Development

Top-down through the scope tree. Start at the root with a root product artifact (Product Brief, needs.md, or Product Description — one of, depending on elicitation path). Decompose into branches (Requirements + Architecture + TestSpec at each level). Terminate at leaves with a Detailed Design + TestSpec. Each layer's specification is authored under human steering before any code is written; agents handle the mechanical decomposition work under review.

Legacy Retrofit (Primary Market Entry)

Bottom-up from existing code. Topology discovery proposes a scope tree. Per-leaf recovery drafts Detailed Design and TestSpec from code + tests. Branch and root recovery draft Architecture, Requirements, ADRs from child contracts. Human fills rationale; AI never fabricates it. The retrofit output is an honest floor of documentation — an artifact set with many recovery_status: unknown fields is the correct outcome when rationale is lost, not a partial failure.

Target Audience

Mid-to-senior software engineers and architects orchestrating AI agents on real codebases. Not a product-management tool; not a rigor-optional framework. If the rigor is not needed for the work at hand, this is not the right tool.

Greenfield Dogfooding (active)

The framework's own purpose-built tooling project, vmodel-core, is the active greenfield dogfooding pilot. It exercises the spec-flow + build-flow round trip end-to-end before retrofit work begins.

First Retrofit Pilot (planned)

Java 17, Gradle, JUnit 5. A legacy codebase of ~100k lines, ~10% test coverage, requirements in mixed Word/spreadsheet/DOORS formats. The retrofit path is the primary market entry vector. Gated on Phase 7 (topology discovery + gap report aggregator).

Project Setup — .vmodel/ & vmodel-init

Every project that uses the framework gets a .vmodel/ directory at the project root, scaffolded by vmodel-init. Skills resolve all reference paths via this central config; nothing is hardcoded to the framework repo.

Layout

.vmodel/ config.yaml # paths, build settings, tool commands (committed) references/ # shared reference docs, copied from framework (committed) authoring-discipline.md partial-parent-protocol.md authoring-self-check.md requirements-shape-checklist.md definitions/ defer-index.md # auto-gen DEFER aggregate (gitignored) .reviews/ # spec-side verdict files (committed; forensic) .build/ # build pipeline state (mostly gitignored) tasks.yaml # task DAG from plan-build pipeline-state.yaml # live state from orchestrate-build lessons.yaml # cumulative bounded lessons (committed) tasks/<task-id>/ # per-task worker handoff files escalations/ # ESC-NNN.yaml (committed; forensic) runs/<run-id>/ # retrospectives

vmodel-init

One-time setup skill. Creates .vmodel/, copies framework reference defaults into .vmodel/references/, prompts for project name, language, root-product type (PB / needs / PD), and tool commands. Migrate mode upgrades existing .vmodel/ non-destructively after framework changes.

Why central?

Skills MUST resolve reference doc paths via .vmodel/config.yaml paths.references. Hardcoding any docs/*.md path is a violation. This makes the framework usable from any project repo without checking out the framework alongside, and lets a project override individual references locally without forking the framework.

The Ten Core Principles

Load-bearing axioms. Every later decision in the framework implements one or more of these.

  1. Documentation is the foundation. Domain knowledge lives in docs first; schemas, templates, skills are derived. If we cannot explain it in docs, we cannot claim AI skills will produce compliant output.
  2. Spec Ambiguity Test. Every specification artifact — at every layer — must be unambiguous enough that a junior engineer or low-mid-tier AI could act on it without guessing. Meta-gate for all specification work.
  3. Uniform high rigor. No rigor tiers. Every artifact is authored at maximum rigor. Rigor is encoded per-artifact as a Quality Bar checklist, not as a configurable level.
  4. Workflow decoupling. Specification, Build, and Retrofit are independent workflows, loosely coupled through artifacts only. No workflow calls, subscribes to, or has runtime knowledge of another.
  5. Tool/skill split. Tools do mechanical deterministic work (no LLM). Skills do interpretive creative work (LLM-driven). Never skills for what tools can do; never tools for what needs judgment.
  6. Tools as independent products. The framework bundles no tools. Tools live in their own repos, are distributed separately, and are invoked via stable CLI contracts. Each project declares its available subset.
  7. Retrofit never fabricates. AI must never invent content that is not derivable from code, tests, or committed evidence. Human-only fields default to recovery_status: unknown unless a human supplies them.
  8. Human leverage at the spec level. The human reviews specifications — where design intent lives — not code. Upper layers have higher human involvement per change; lower layers have higher AI autonomy.
  9. Components are independent (SOLID). No forced coupling between Documentation, Templates, Traceability, and Skills. Adopt incrementally.
  10. Tests derive from the layer's artifact, not from the code below. Unit tests derive from Detailed Design. Integration tests from Architecture. System tests from root product (PB / needs / PD) + root Requirements. V-model verification restored.

The Spec Ambiguity Test

The universal meta-gate. Every specification artifact at every layer must pass it before it is allowed downstream.

The Test

Could a junior engineer or low-mid-tier AI act on this artifact and produce correct work — without guessing, without hunting external context, without pinging the author?

If no, the artifact is not ready. The layer above is not done.

Why This Is the Right Bar

  • Senior engineers silently patch ambiguity with expertise. Juniors reveal it. Low-mid-tier AI reveals it cheaply and at scale.
  • If the cheapest capable worker can produce correct work from the spec, the spec is doing its job. If only the senior author can, the spec is a stand-in for tribal knowledge.
  • The framework's value proposition — AI agents producing well-structured software under human design review — collapses the moment specifications require interpretation beyond what a low-mid-tier model can do.

Where It Bites

Every Quality Bar checklist on every artifact page ends with the Spec Ambiguity Test as its meta-gate. Passing every other checklist item does not substitute for passing this one. Failing this one blocks the artifact regardless of other passes.

Human Leverage Model

Human drives at the strategic level. AI executes at the tactical level. Human verifies at the quality gate. Not an autonomous pipeline with checkpoints.

The Six Artifacts

One anchor artifact at the root. Four at every non-leaf scope. One at every leaf. ADRs cross-cutting across scopes. Every spec element is verified by tests at its own layer.

ArtifactLives atJobVerified byPage
Root product (PB / needs / PD — one of) Root only Anchor: why, for whom, what problem Stakeholder review product-brief.html
Requirements Non-leaf scopes Testable behavior specifications Acceptance / integration tests requirements.html
Architecture Non-leaf scopes Decomposition + composition pattern Integration tests architecture.html
ADR Cross-cutting Justify individual load-bearing decisions Review adr.html
Detailed Design Leaf scopes Internal logic, contracts, algorithms Unit tests detailed-design.html
TestSpec Every scope Test specification for this scope (its own test code) testspec.html

ADRs Are Cross-Cutting

ADRs are pulled out as first-class artifacts, not inlined in Architecture. They live in a flat /specs/adrs/ directory and are referenced by governing_adrs from any spec artifact they bound. Authoring default: during Architecture authoring as the composition exposes a decision.

TestSpec Has Three Flavors

Same envelope, different weight per layer. Leaf tests are unit tests — thin TestSpec cases; test code carries most of the intent. Branch tests are integration tests — preconditions, fixtures, doubles. Root tests are system / e2e scenarios — heavy steps, preconditions, expected; scenarios map to root product (PB / needs / PD) outcomes and root Requirements.

The canonical structural reference for all six artifacts is docs/plan/TARGET_ARCHITECTURE.md §5.3. Each sidebar-linked page teaches the craft for one artifact under the five-section structure.

Scope Tree

The system is modelled as a tree of scopes, discovered per-system, of variable depth. The same decision types apply at every depth.

Abstract Vocabulary

  • Root — the whole product. Exactly one per system. Holds the root product artifact (Product Brief, needs.md, or Product Description — one of three options depending on elicitation path).
  • Branch — any non-leaf scope. Holds its own Requirements, Architecture, TestSpec, and child scopes.
  • Leaf — where subdivision stops earning its keep. Holds a Detailed Design, TestSpec, and ultimately code.

Projects may tag scopes with domain-local names (level_name: "subsystem" | "component" | "module") for display. The framework still uses root/branch/leaf internally. Layer axis = decision kind, not structural scope. Requirements / architecture / design / tests are decision types that apply at any scope depth.

Heuristics for When to Subdivide

  • Depth test — if children interact in ways the parent's Architecture cannot explain, go deeper.
  • Cognitive-load test — a scope should fit in one Requirements+Architecture pair a reviewer can hold in their head.
  • Change-blast test — if subdividing would not let parts change independently, you have gone too deep.

Deployment and Wiring

Architecture Composition is the authoritative spec for runtime pattern, wiring approach, and deployment intent. IaC artifacts (terraform / Kubernetes / docker-compose) implement that intent — no separate Detailed Design layer above them. Integration and end-to-end coverage are carried by branch- and root-scope TestSpec tracing to Architecture Composition and child interfaces.

Rigor & Quality Bar

No tiers. No T0/T1/T2, no DAL, no ASIL. Every artifact is authored at uniform highest rigor, encoded per-artifact as a Quality Bar checklist.

Why Uniform

Tier systems create escape valves that defeat themselves. The rigor-cost curve is flat for AI authoring: the marginal cost of producing a well-structured artifact versus a lightly-structured one is negligible once the authoring skill exists. Tiers add configuration complexity, invite miscalibration, and give authors a plausible-deniability lever for cutting corners.

Two Layers of Rigor

Structural (mechanical)

Lives in the artifact schema. Mandatory sections, required fields, required traceability links. Mechanically validatable: vmodel validate <artifact> returns pass/fail.

Semantic (Quality Bar)

A concrete Yes/No checklist grouped by concern, subject to the Spec Ambiguity Test as its meta-gate. Authored as the fifth section in each artifact page; extracted as canonical YAML in Phase 3.

Cascade

Quality Bar is additive per layer. Upstream failure blocks downstream approval. A leaf Detailed Design cannot pass review if its parent Architecture Quality Bar has unresolved items. Tooling surfaces the chain rather than allowing a reviewer to rubber-stamp a leaf on top of a broken parent.

No Relaxed Variants

There is no "lightweight mode" for small projects or prototypes. If the rigor is not needed, the framework is not the right tool.

Files, Directories, IDs

File Shape

Single-file Markdown per artifact. YAML front-matter for metadata and outgoing traceability links. Body: prose sections + embedded YAML blocks for repeated structured items (per-requirement, per-child, per-test-case, per-function) + Mermaid diagrams. Artifacts are self-contained — an artifact that requires a separate lookup to be understood fails the Spec Ambiguity Test.

Directory Layout

The scope tree mirrors the directory tree under a configurable /specs/ root. ADRs flat in /specs/adrs/.

/specs/ product_brief.md # root scope requirements.md # root scope architecture.md # root scope testspec.md # root scope /adrs/ # flat, cross-cutting adr-001-use-postgres.md /app/ # branch scope requirements.md architecture.md testspec.md /checkout/ # leaf scope detailed_design.md testspec.md

Root-scope artifacts live directly at /specs/. "Root" is implicit, not a directory name. Scope directories use kebab-case; artifact filenames are snake_case.

Specification Workflow — Greenfield

Top-down from root scope. Per-layer unified pattern: specify → (build elsewhere) → allocate to children → recurse.

Per-Layer Pattern

Input: parent allocation (or root product artifact at root) ↓ SPECIFY (what) — Non-leaf: Requirements + Architecture — Leaf: Detailed Design — Any: ADRs (technical picks → constraints flow down) — Any: TestSpec (authored here, before any code) ↓ (Build workflow — separately, elsewhere) — Root: deploy orchestration + e2e / system tests — Branch: wiring / glue code + integration tests — Leaf: code + unit tests ↓ Non-leaf: allocate reqs to children → recurse

Completion Criterion Per Scope

  • Every spec artifact passes its Quality Bar (structural + semantic).
  • Traceability validation passes for the scope.
  • Every TestSpec case has non-empty verifies.
  • No recovery_status: unknown on any blocking field.

Anyone can query this. The Specification workflow emits it at nobody; it is a derived property of the traceability graph.

Human-Gate Gradient

Higher layers drive more human involvement per change; lower layers have higher AI autonomy. This is a property of skills and flow, not encoded in schemas. See Human Leverage Model.

Specification Workflow — Retrofit

Not a separate workflow. The Specification workflow run bottom-up with a topology-discovery phase prepended. Same artifact set, same Quality Bar, same Spec Ambiguity Test.

Four Phases

  1. Topology discovery. Dependency / package / module analysis produces a candidate scope tree. AI proposes; human validates. Output: an agreed scope tree before any artifact authoring.
  2. Leaf recovery. Per leaf, from code + existing tests, draft a Detailed Design, TestSpec, and leaf ADRs for visible technology. Human adds intent (why this leaf exists, what invariants must hold). Spec Ambiguity Test is the gate.
  3. Branch + root recovery. Bottom-up. Draft Architecture from child designs (composition pattern, interfaces, allocations). Draft Requirements from child contracts + externally supplied QAs. Draft TestSpec (integration at branch, system at root). Draft ADRs for visible technology. Human supplies missing rationale.
  4. Product Brief recovery (optional). Requires conversations with product owners / users. Deferred when stakeholders are inaccessible; the system runs on everything below without it.

Read-Only on Code

Retrofit produces specifications + a gap report. Code modifications are a separate follow-up activity.

Recovery Status Discipline

Fields split by type.

Code-derivable fields

Evidence exists in code, tests, git, schemas, committed comments.

Allowed states: verified | reconstructed | unknown.

Human-only fields

No code-derivable source. ADR context / alternatives / rationale / anticipated consequences. Requirements rationale. Product Brief sections. Detailed Design intent.

Allowed states: verified | unknown only. No reconstructed state.

The retrofit skill must refuse to populate human-only fields with AI inference. Enforced at the skill / tool level. Fabrication of rationale is a hard rule violation, not a quality issue.

First-class deliverable, alongside artifacts.

GapMeaningFramework response
Unintentioned leafLeaf with no discoverable purpose after human reviewCandidate for removal / refactoring

Ambiguous specArtifact fails Spec Ambiguity Test even after human reviewRefactor code for clarity, or document the gap

Structural driftRetrofit-derived architecture differs from existing documented architectureFlag; reconciliation required
ADR without decision recordTechnology visible in code but context lostADR drafted with recovery_status: unknown on context / alternatives

Primary purpose of retrofit: establish an honest floor of documentation from which forward development can proceed. Secondary: surface gaps. A retrofit output with many unknown fields is not a partial failure — it is the correct outcome when rationale is lost.

Update Mode

Enter at the highest layer that must change. A change enters at layer N if N is where something genuinely new (not derivable from existing artifacts) gets introduced.

Entry-Layer Decision Table

Change sourceEntry layer
New stakeholder, outcome, scope expansionRoot product (PB / needs / PD)
New behavior under an existing outcomeRequirements (at the scope where it applies)
Re-allocation of existing behavior to a different componentArchitecture
Technology swap / pattern changeADR + Architecture Composition
Algorithm or state-machine changeDetailed Design
Bug fix where spec was correct, code was wrongImplementation (Build workflow)
Bug fix where spec was ambiguous or wrongThe layer whose spec was wrong

Candidate-Set Propagation

Tooling walks traceability forward from the entry-layer change and produces a candidate set of artifacts that may need updates. The change is incomplete until every candidate is either updated or explicitly closed as "no change needed" with a brief reason.

Implemented as scripts/walk-impact.py <ARTIFACT-ID> --specs-root specs/ --format text|json. Pure graph walker — no LLM. Reads derived_from, parent_architecture, governing_adrs, verifies (envelope + per-case), and allocates (per-child) link types and emits the BFS reverse closure ordered by hop distance.

Upward Propagation

If a gap is discovered downstream, the change's entry layer retroactively rises. The impact set recomputes from the higher entry. If a code fix implies the spec was wrong, the spec is the entry — not the code.

Build Workflow

Consumes specifications, produces code + tests. Six build-side skills compose into the pipeline; three are dispatched as worker agents under the Task tool for parallel isolated execution.

Skills

SkillRole
vmodel-skill-plan-buildWalk specs/ tree; emit task DAG to .vmodel/.build/tasks.yaml. Populates acceptance_criteria / context_to_load / out_of_scope mechanically.
vmodel-skill-orchestrate-buildPipeline state machine. Layered execution (leaf unit → branch integration → root system). Dispatches workers via Task tool. Layer-typed escalation routing.
vmodel-skill-render-testsTestSpec → executable tests (TDD red phase). Layer-aware.
vmodel-skill-implement-leafTDD green + refactor for one leaf. Greenfield / fix / resume modes. Scope-expansion HALT via build-blocked.yaml.
vmodel-skill-review-executionVerdict at any layer (APPROVED / REJECTED / ESCALATE). Confidence-tagged. APPROVED writes no file.
vmodel-skill-retrospect-buildEnd-of-run lessons extraction. Refuses to fabricate (≥2 reproductions or one high-impact).

Worker Agents

Three thin dispatch shells under .claude/agents/ wrap the worker skills (render-tests, implement-leaf, review-execution) for true within-stage parallelism via the Task tool. The agent file declares the dispatch envelope; the skill holds the procedure. retrospect-build stays a Skill invocation (single end-of-run, no parallelism need).

Per-task File Contract

Each leaf task owns .vmodel/.build/tasks/<task-id>/ with single-producer files: current-task.yaml (orchestrator), render-report.yaml (render-tests), build-progress.yaml + build-blocked.yaml + review-ready.yaml (implement-leaf), feedback.yaml + ESC-NNN.yaml (review-execution). APPROVED writes no file — orchestrator infers from absence.

Coupling to Specification

  • No runtime coupling. Artifacts are the sole contract.
  • Build-side realizes links reserved in the traceability schema for code → spec mapping.
  • Insufficient spec → escalate. review-execution emits typed ESC-NNN.yaml with target_layer (DD / TestSpec / Architecture / ADR / Requirements / product); orchestrator routes to the matching author skill in mode B.
  • Declarative IaC has no extra DD layer. Architecture Composition is the intent spec; IaC (terraform / k8s / compose) is the implementation.

Full pipeline shape, escalation routing, resume modes, and worked walkthroughs in Skills & Workflow Architecture (Build Flow + User Workflow 1).

Skill Catalog (counts)

  • Spec-side (12) — 5 author/review pairs (Requirements, Architecture, ADR, Detailed Design, TestSpec) + 2 elicitation skills (elicit-needs, elicit-pd).
  • Build-side (6) — plan-build, orchestrate-build, render-tests, implement-leaf, review-execution, retrospect-build.
  • Setup (1) — vmodel-init.

Worker Agents (3)

Thin dispatch shells under .claude/agents/: render-tests, implement-leaf, review-execution. Each loads its canonical skill via the Skill tool. Orchestrator dispatches via the Task tool for parallel isolated execution.

Skill / Agent / Tool Split

  • Skill — interpretive, judgment-driven procedure (LLM-driven). Holds procedure, references, templates, HALT conditions.
  • Agent — dispatch shell loaded via Task tool for isolated context. No procedure of its own.
  • Tool — mechanical, deterministic check (no LLM). E.g., scripts/check-*.py, scripts/walk-impact.py, scripts/index-deferred-items.py.

Design Rules

  • Naming: vmodel-skill-* for skills, dispatch agents named after their wrapped skill (e.g., implement-leaf).
  • File-based handovers for crash-tolerance and auditability — every author ↔ review and worker ↔ orchestrator boundary is a YAML or Markdown file on disk under .vmodel/.
  • Haiku-floor evaluation per skill is deferred until Phase 6 dogfooding signal exposes real failure modes; pilot signal is higher-value than synthetic eval scenarios at this stage.
